# 全量价值压榨结果：小女巫 App 内 UI 系统重构

> 日期：2026-07-03  
> 输入：6 个核心设计师 Skill + Sagmeister 侧翼 + 主题商店规范 + 云真机截图 + Max-Value Run 口径  
> 输出：完整 UI 系统方向推演，不写 ArkTS，不做保守微调  
> 结论：当前 App 内 UI 应从“默认相册染主题色”升级为“宠物照片主演的魔法花园舞台系统”。  

## 1. 总结论

第一版 baseline 的判断是对的，但力度不够。第二轮全量压榨后，结论更明确：

1. **当前问题不是主题色浅，而是主题系统没有接管 App 内结构。**  
   桌面 2x4 Widget 已经有“主题卡片”语言，App 内首页/猫档案/我的页/Tab 仍像默认信息架构上铺了淡紫和背景图。

2. **1.5 大版本不该继续点缀染。**  
   点缀染是早期防止过重的工程过渡方案，不适合承担“首个主题大版本留存钩子”。现在需要更重的暖粉紫、深棕描边、魔法书页、花园舞台和主题组件语言。

3. **宠物照片 C 位不是限制主题，而是要求主题成为舞台。**  
   主题越强，越应该像舞台灯光、布景、相框和道具一样把宠物照片推到主角位置。主题不能抢照片，但可以非常强。

4. **App 内页面可以整体换骨架。**  
   首页从“信息卡片列表”改为“照片舞台首屏”；猫档案从“统计卡 + 大照片”改为“宠物身份证 + 舞台相册”；我的页从“设置表单”改为“桌面装扮工作台”；Tab 从系统导航改为“魔法书签/花园路标”。

5. **推荐主方向：魔法书页舞台系统。**  
   不是最克制，也不是节日爆炸，而是把桌面 2x4 已认可的主题卡片感迁移进 App 内：强主题、强舞台、照片第一、动效克制。

## 2. 当前 UI 系统诊断

### 2.1 首页

问题不是照片不够大，而是照片的“角色”不对。当前照片处在一个插画框内，但页面的其他区域仍是普通内容流：标题、卡片、轮播点、底部卡片、最近回忆标题、Tab 都像不同系统拼在一起。

具体问题：

- 大照片是内容块，不是首屏舞台核心。
- 背景插画被当成 wallpaper，而不是和照片发生结构关系。
- 轮播点、卡片边框、底部 Tab 都有淡紫，但没有共同形态语言。
- 页面下半部“最近的回忆”露出一点，造成首屏焦点分裂。
- 底部 Tab 占用大量浅色空白，选中态像系统默认导航。

应该改成：

- 首屏只服务一个叙事：今天的宠物在小女巫花园里。
- 照片舞台、今日状态、桌面投放入口和 Tab 共用同一套“魔法书页/花园路标”语言。

### 2.2 猫档案

当前猫档案页的问题是“组件正确，但身份不强”。头像、三张统计卡、大照片、日记卡片都能用，但不像一个宠物的主题档案，更像默认 profile 页面换了背景。

具体问题：

- 三张统计卡太像默认 dashboard 小卡。
- 大照片的主题舞台仍然是背景图承托，缺少强相框结构。
- 宠物名和陪伴天数没有形成品牌化标题区。
- 背景插画和前景内容对比不足，导致画面浅、散。

应该改成：

- 顶部做“宠物魔法身份证”：头像、名字、陪伴天数、生日、状态徽章合成一个身份块。
- 大照片做“主照片舞台”，不是普通相册图。
- 统计信息压缩成 2-3 个强主题胶囊，放在照片舞台边缘或底座。

### 2.3 日记页

日记页是当前最接近可用的一页，因为信息密度够，卡片节奏清楚。但它仍不够像小女巫主题。

具体问题：

- 日记卡片只是淡紫卡片，缺少“魔法便签/相册页”的物性。
- 缩略图相框存在，但和文字卡片的关系弱。
- 筛选 chip 像普通 chip，不像主题路标。
- “最新”角标、日期标签、日常标签可以更有符号系统。

应该改成：

- 日记卡片改为“魔法便签 / 羊皮纸照片条”。
- 每张卡片左侧照片是小相框，右侧文字是标签纸。
- 日期、D+1、最新等状态形成统一徽章语言。

### 2.4 我的 / 桌面设置页

我的页最大问题是“主题设置页没有成为主题工作台”。它现在像设置表单：预览卡、范围选择、内容选择、多个白色控件。桌面 2x4 预览内部好看，但页面整体没有围绕它组织。

具体问题：

- 顶部操作按钮漂浮，像系统按钮，不像工作台工具。
- 桌面预览卡没有成为页面第一主角。
- 显示范围、卡片内容等控件是普通 segmented/card。
- 页面空白仍是背景图露出来，而不是工作台留白。

应该改成：

- 我的页重构为“桌面装扮工作台”。
- 顶部大预览占据视觉中心，其他控件围绕预览像工具面板。
- 显示范围是“魔法书签切换”，卡片内容是“便签选项”。

### 2.5 底部 Tab

Tab 是当前最拖后腿的地方。它把强主题页面重新拉回“系统默认 App”。选中态只是淡紫底 + icon 变紫，视觉上没有小女巫世界观。

应该改成：

- Tab 是花园路标、魔法书签或三枚小徽章。
- 图标必须路标化，不做贴纸插画。
- Tab 背景高度变紧凑，减少大块空白。
- 选中态是底座/书签凸起/柔光边，不只是改颜色。

### 2.6 桌面 Widget 与 App 内不一致

桌面 2x4 被认可，是因为它已经具备：

- 照片 + 信息双区结构。
- 明确主题卡片边界。
- 主题标签和文字区有设计。
- 视觉密度比 App 内更强。

App 内应该反向吸收桌面 2x4 的语言，而不是桌面强、App 内弱。

## 3. Skill 分层完整判断

### 3.1 原研哉：空白与物性

1. 首页空白是“残留空白”，不是“主动的空”。它没有几何形状，也没有邀请用户看照片。
2. 猫档案的浅色统计区缺少物性，应该从普通卡片变成标签纸/档案章。
3. 我的页背景露出太多，像没填满，不像工作台桌面。
4. Tab 区大块白底不是留白，是系统导航残留。
5. 小女巫主题的物性不应是玻璃感，而应是和纸、魔法书页、蜡笔标签、相纸。
6. 加重主题时不能靠堆装饰，应先给空白边界：书页、相框、底座、折角、标签。
7. 主页照片舞台应有“容器的物性”：像一本摊开的魔法相册，而不是普通圆角卡片。
8. 背景插画不能只是铺底，必须和前景相框形成层级。
9. 该降噪的是系统默认轮播点、过多浅卡片、重复标题，不是主题舞台。
10. 不能把紫色铺满当作主题，颜色必须附着在纸、布景、描边、徽章这些物上。

### 3.2 Paula Scher：记忆点与主题音量

1. 当前小女巫 UI 不被记住，因为它没有一个可说出口的视觉钩子。
2. “淡紫色相册”不是品牌记忆点；“宠物在魔法花园相册里”才是。
3. 主题零件包应至少包含：深棕描边、暖粉紫书页、花园拱门相框、魔法书签 Tab、星形徽章。
4. 当前设计卡在 beige 区：既不够克制到高级，也不够强到被记住。
5. 1.5 主题大版本的音量应到 8 左右，而不是 6-7。
6. 首页必须有一个强图形结构，例如魔法书页、花园剧场、桌面相册，而不是普通垂直流。
7. Tab 是品牌接触面，不能放弃给系统默认。
8. 我的页也必须被记住，因为它是用户配置桌面卡片的地方，商业价值高。
9. 字体不一定引入新字体，但标题块可以更像标签/书名/档案章。
10. 强主题不等于大声促销，应该是“视觉系统很清楚地说：这里是小女巫花园”。

### 3.3 Susan Kare：符号和微情绪

1. Tab icon 应像路标，不应像缩小插画。
2. 首页的“放到桌面”入口可以用小桌面/书签/星章隐喻，不用普通加号或卡片图标。
3. 日记页标签“日常/生日/投喂/第一次”可以做成统一小徽章，不是普通 chip。
4. “最新”“D+1”“陪伴 1 天”应有统一状态徽章系统。
5. 小女巫符号不应过多：帽子、星、书签、花朵、药水瓶最多选 3 个核心符号。
6. 图标友好感来自圆角、深棕线、少量不完美，不来自 emoji。
7. 底部 Tab 的 3 个图标可以分别是：小屋/相册页/宠物徽章，但必须 0.3 秒可辨认。
8. 我的页的管理/保存不应用抽象系统符号过强，保存可以是星章确认，管理可以是小工具/魔法棒，但要清晰。
9. 图标不能抢照片；所有符号应停留在边角、标签、状态和导航层。
10. 中文标签可保留，图标负责情绪和识别，文字负责确认。

### 3.4 Yugo Nakamura：motion grammar

1. 持续飘落失败的原因不是“动画不够炫”，而是它没有服务交互，也穿越了主资产区域。
2. 小女巫 motion grammar 应是“相框有生命”，不是“背景有粒子”。
3. 所有动效应有双层响应：即时确认 + 物理余韵。
4. 首页进入时照片舞台可以轻落位，但不应让背景动起来抢视线。
5. Tab 切换应像书签被按下/轻弹起来，而不是 icon 变色。
6. 日记卡片出现可以像标签纸轻贴到相册页上。
7. 桌面预览切换可以像翻页或卡片滑入。
8. 角标呼吸如果存在，周期应很长、幅度极低，且不穿越照片。
9. 凡是阅读、照片观看、文字输入相关区域必须静止。
10. 真机稳定性优先：任何不稳定的 ambient 都应降级为静态柔光。

### 3.5 Dieter Rams：减法和诚实

1. 可以删的是廉价持续动效，不是主题舞台。
2. 可以删的是重复卡片和默认系统控件，不是承托照片的相框。
3. 当前“有背景插画但组件仍默认”的状态是一种假主题。
4. 我的页如果只是表单染紫，会降低主题商店的商业可信度。
5. 强主题成立，但装饰必须诚实地作为情绪元素，不伪装成功能。
6. 底部 Tab 的大白区没有功能辩护，应减少高度或转成主题底座。
7. 页面上每个徽章、标签、chip 都要有功能或情绪理由，否则删。
8. 不应为了“干净”把主题删回默认相册，这叫过删。
9. 长期主题商店需要可复用结构，不应每个主题都硬拼一堆图。
10. 减法后的目标不是更空，而是照片更清楚、主题更可信、任务更明确。

### 3.6 Refactoring UI：落地系统

1. 当前间距系统过松，尤其首页下方和 Tab 区。
2. 需要以 4vp/8vp 基准重建页面密度，而不是局部调 padding。
3. 字体层级应减少数量，但标题块要更有主题形态。
4. 圆角系统要和物性一致：照片舞台 20-28vp，徽章 12-16vp，Tab 书签 18-24vp。
5. 描边应用深棕，不用冷灰边框。
6. 阴影应是暖棕/紫棕双层柔影，不用纯黑。
7. 色彩要有 3-5 阶紫色，而不是一个薰衣草主色到处用。
8. 组件体系应有：舞台、标签纸、徽章、书签 Tab、工作台面板、照片相框。
9. HTML 原型必须验证组件关系，不只是画漂亮背景。
10. 要明确哪些 token 是跨主题通用，哪些是小女巫专属。

### 3.7 Sagmeister：强情绪上限

1. 上限方向要问：用户看到是否会心动，而不是只觉得专业。
2. 宠物照片本身是最真实的情绪来源，节日符号不能盖过它。
3. “小女巫守着你家猫的回忆”比“花园里有很多魔法元素”更动人。
4. 纯美丽元素可以有，但必须围绕照片：星光边、花园拱门、书页折角。
5. 假嗨风险来自满屏星星、花瓣、蝴蝶和过多角色。
6. 强主题最好像一个用户每年愿意打开的传统，而不是一次性活动页。

## 4. 五个完整 UI 系统方向

### 4.1 方向一：魔法书页舞台

视觉主张：App 是一本摊开的魔法相册，宠物照片贴在主书页上，小女巫和花园元素围绕书页服务照片。

用户第一眼：这不是普通相册，是我家猫在一本小女巫魔法书里。

首页结构：

- 顶部标题像书页标题，压缩高度。
- 主区域是大魔法书页舞台，占首屏 65%-72%。
- 宠物照片在书页中央，占舞台 45%-55%。
- 今日状态、日期、D+1 作为书页底部标签。
- “放到桌面”是书签按钮，不是普通卡片。

猫档案结构：

- 顶部做“宠物档案章”：头像、名字、生日、陪伴天数。
- 大照片像贴在档案书页里的相纸。
- 统计信息是书页边缘的 3 个小标签。

日记页结构：

- 日记卡片像一页页魔法便签。
- 左侧照片是小相纸，右侧文字是标签纸。
- 筛选栏像书签索引。

我的页结构：

- 桌面预览是一张放在书桌上的卡片。
- 显示范围、卡片内容、主题选择围绕预览排列成工具区。
- 保存按钮是书页右上角星章确认。

Tab 结构：

- 三个 Tab 是书签：主页、日记、我的。
- 选中书签向上凸起，有紫色底和星点。

Token 方案：

```text
bg:              #FFF1E4
book_page:       #FFF8ED
book_shadow:     #D79DB8
lavender:        #A66AD4
lavender_deep:   #7F45B0
pink:            #E98BA8
sun:             #F0C55A
stroke:          #6B3F32
ink:             #392831
muted:           #8A6F73
```

动效方案：

- 首页书页 360ms 轻展开。
- 照片 280ms 落位。
- Tab 书签 180ms 弹起。
- 主题切换时书页边缘柔光闪一次。

资产映射：

- `bg-book.png` 作为书页舞台参考。
- `witch.png` 放在书页边角。
- `card-2x4.png` 提取右侧信息区语言。

最大风险：如果书页形态做得太重，可能像儿童绘本而非日常 App。  
为什么值得做：最能统一首页、日记、我的页和 Tab，是推荐主方向。

### 4.2 方向二：花园剧场

视觉主张：宠物照片是舞台中央的演员，花园、拱门、小女巫、星光是布景和灯光。

用户第一眼：我家猫站在小女巫春日花园的主舞台上。

首页结构：

- 首屏用花园拱门包住照片。
- 顶部标题变轻，主视觉前置。
- 今日状态在舞台底座。
- 最近回忆不露出，避免抢戏。

猫档案结构：

- 宠物身份放在舞台上方像剧场铭牌。
- 照片大且靠中，统计是舞台前的小牌子。

日记页结构：

- 每条日记像花园路牌，照片是路牌上的小画框。
- 列表背景保留花园纵深。

我的页结构：

- 桌面预览像放在花园展台上。
- 控件像路牌/木牌/花园按钮。

Tab 结构：

- 三个 Tab 是花园路标。
- 选中态路牌点亮。

Token 方案：

```text
bg:              #FFEEDC
garden:          #F3C9E4
grass:           #A8C060
lavender:        #9B5FC9
lavender_deep:   #743FA2
terracotta:      #C0785A
sun:             #F0D878
stroke:          #704534
ink:             #3A2A25
```

动效方案：

- 舞台灯光 300ms 点亮。
- 拱门边缘柔光 400ms。
- 路牌 Tab 160ms 下压。

资产映射：

- `card-2x2.png` 的花园相框语言。
- `witch.png` 作为舞台守护者。

最大风险：花园布景过强会让照片像插画一部分。  
为什么值得做：主题记忆最直接，适合留存钩子。

### 4.3 方向三：女巫工作台

视觉主张：App 是小女巫整理宠物回忆的工作台，照片、卡片、日记、桌面设置都是桌上的物件。

用户第一眼：这是一个能摆弄宠物回忆和桌面卡片的魔法工作台。

首页结构：

- 主照片像摆在桌面上的相纸。
- 今日状态是贴在相纸旁的小便签。
- 放到桌面入口是工具按钮。

猫档案结构：

- 档案像桌面上的资料卡。
- 统计信息像贴纸章，但克制。

日记页结构：

- 日记是桌面便签堆叠。
- 筛选 chip 是抽屉标签。

我的页结构：

- 这是该方向最强页面：大桌面预览 + 工具抽屉 + 显示范围面板。
- 预览卡成为绝对主角。

Tab 结构：

## 5. 组件级设计细节

### 5.1 主照片舞台

- 推荐尺寸：首页首屏中照片本体占可视高度 38%-48%，舞台整体占 62%-72%。
- 形态：魔法书页 / 花园拱门 / 相纸贴片三选一。
- 必须有深棕描边或插画边界，避免照片漂在背景上。
- 装饰距离宠物脸部至少留出安全区。
- 角落可放星章、花、书签，但不穿越照片主体。

### 5.2 宠物档案卡

- 顶部身份块：头像 + 名字 + 陪伴天数 + 本地保存状态。
- 统计不再三张等宽默认卡，可压成徽章行。
- 生日/体重/照片数使用统一图标和标签样式。

### 5.3 日记卡片

- 左图右文保留，但从“列表卡片”改为“便签卡片”。
- 日期章放照片底部或卡片角落。
- “最新”用星章，不用黄色系统贴纸。
- 缩略图相框保持清晰，不能套过重花边。

### 5.4 统计胶囊

- 胶囊高度 28-34vp。
- 描边 1.5-2vp 深棕或深紫。
- 背景用 `paper` 或 `lavender_soft`。
- 图标 14-16vp，保持路标化。

### 5.5 桌面预览卡

- 我的页第一主角。
- 2x4 预览按真实桌面方向展示，左照片右信息。
- 下方控件围绕预览，不让预览变成普通列表项。

### 5.6 底部 Tab

- 高度建议 58-68vp，不再大面积空白。
- 三个 tab 是书签/路标/徽章。
- 选中态有形态变化：凸起、底座、星章，而不只是颜色。
- 图标必须一眼可辨，文字保留。

### 5.7 主按钮 / 次按钮

- 主按钮：深紫或暖粉紫实心 + 白字 + 深棕描边。
- 次按钮：纸底 + 深棕字 + 紫色描边。
- 按压：scale 0.96-0.98 + opacity 0.75-0.9。

### 5.8 标签 / 徽章 / Chip

- 标签像贴纸或书签，但不能 emoji 化。
- 选中 chip 用深紫描边和浅紫底。
- 非选中 chip 用纸底和低对比深棕字。

## 8. 冲突仲裁

| 冲突 | 裁决 |
|---|---|
| 照片 C 位 vs 强主题 | 强主题可以赢视觉面积，但不能赢第一落点；主题做舞台，照片做主角 |
| 原研哉留白 vs Scher 强记忆 | Scher 赢主题音量，原研哉保留组织空白和反廉价底线 |
| Rams 减法 vs Sagmeister 美丽 | 日常页 Rams 赢；庆典上限 Sagmeister 可赢 20%-30% 纯美丽元素 |
| Yugo ambient vs 真机稳定 | 真机稳定赢；ambient 降级为短时/边缘/低频 |
| Refactoring UI 系统化 vs 主题个性 | 主题个性赢；Refactoring UI 负责让它可落地，不负责变通用 |
| 图标清晰 vs 插画可爱 | 清晰赢；可爱只能在路标级清晰后添加 |
| 我的页效率 vs 主题工作台 | 工作台赢视觉骨架，但效率控件必须更清楚 |
| 日记页密度 vs 主题装饰 | 密度和可读赢；装饰只在卡片边界和徽章层 |
| 底部 Tab 默认习惯 vs 主题 Tab | 主题 Tab 赢，但文字标签保留，避免学习成本 |
| 色彩加重 vs 可读性 | 可读性赢；深紫用于强调，不大面积压文字 |
| 资产复用 vs 新结构 | 新结构赢方向；资产不足时用 CSS/HTML 验证，再决定是否补素材 |
| 商业留存 vs 产品 DNA | 产品 DNA 赢；主题商店卖的是照片舞台，不是遮住照片的皮肤 |

## 9. 自我批判

### 9.1 可能失败的方向

- 女巫花园庆典可能过重，日常使用会累。
- 花园剧场如果舞台布景太强，宠物照片可能变成插画一部分。
- 宠物魔法身份证可能偏档案产品，降低童话感。

### 9.2 可能太贵的方向

- 魔法书页舞台若需要高质量书页/拱门切图，素材成本会上升。
- 花园剧场需要更多前景/中景/背景素材分层。
- 庆典方向需要最多素材和动效验证。

### 9.3 可能偏离产品 DNA 的地方

- 过多女巫角色会让“宠物陪伴相册”变成“女巫主题插画 App”。
- 过多动效会干扰照片回忆。
- 过多商业化装扮感会让免费主题显得像广告。

### 9.4 最能体现 Skill 价值的方向

魔法书页舞台最能体现整套 Skill：

- 原研哉：书页物性和有组织留白。
- Scher：明确记忆钩子。
- Kare：书签/徽章/路标符号。
- Yugo：书页展开、相框柔光、Tab 弹起。
- Rams：删持续飘落，保留功能性舞台。
- Refactoring UI：完整 token 和组件系统。
- Sagmeister：可在庆典版上沿用。
