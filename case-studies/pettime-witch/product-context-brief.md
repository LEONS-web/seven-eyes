## 一、产品语境（来自 product-context brief）

- **产品**：萌宠时光册 — 桌面萌宠陪伴相册（HarmonyOS）
- **当前版本**：1.5.0（1000050）
- **当前主题**：小女巫春日花园（首个免费主题，1.3.0 上线）
- **核心问题**：主题显性度不足，留存钩子不够。r5 照片 C 位正确但主题太淡，r6/r7 往强主题方向调整中。App 内首页、日记、我的页空白区域无组织，照片放大后主题舞台弱。
- **硬边界**：本地优先、零权限、不接账号/支付/网络/云同步。不替换 `formImages(fd) + memory://widgetPhoto` 主链路。
- **桌面形态**：2x2 卡片、2x4 卡片、App 内全屏
- **hero asset**：宠物方形照片（用户本地选取）— 不可被遮挡/压缩/替代
- **主题显性度目标**：强主题（7/10）— 主题必须成为留存记忆点，但照片仍是 C 位

## 四、当前问题（需要修复的具体项）

1. **App 内首页大块浅紫白区域无组织**：空白是被动残留区域，不是主动的"舞台"。没有边界、没有指向、没有在组织照片和相框的关系。

2. **照片放大后主题舞台弱**：照片变大了（正确方向），但主题元素（小女巫、魔法书、花园氛围）没有形成"围绕照片的舞台"，照片和主题各自孤立。

3. **底部 Tab 占位感**：首页/日记/我的三个 Tab 切换区域在视觉上"只是系统默认导航栏"，没有融入主题。

4. **主题过淡**：r5 版主题约等于"暖白底+淡紫点缀"，用户感知不到"这是小女巫春日花园"，只觉得"这是一个淡紫色的 App"。

5. **空白不是"空"而是"空洞"**（原研哉 H1 诊断结论）：当前留白是被动的——没有形状、没有邀请、没有在组织视线。好的留白应该有几何边界，让人感到"照片即将出现在这里"而不是"这里空着不知道放什么"。

---

## 五、成功标准

> 用户打开 App 的 0-5 秒内：

1. **先看到宠物**：宠物照片是视觉第一落点。面积最大、位置最中心、没有任何东西遮挡它。

2. **再感到小女巫主题**：看完照片后，视线自然移动到主题舞台——魔法相框、花园氛围、小女巫角色的存在感。用户不需要"找主题"，主题围绕在照片周围。

3. **空白被组织**：首页没有"不知道为什么空着"的区域。每一块留白都有几何形状，都在指向或承载某些内容。

4. **底部 Tab 有主题归属**：不是系统默认导航栏，而是"小女巫花园里的一部分"。

## 六、设计参数卡（来自 design-taste-frontend）

| 参数 | 值 | 对 huashu-design 意味着什么 |
|------|-----|--------------------------|
| DESIGN_VARIANCE | 7 | 布局可以不对称、有特色，但不到实验艺术级别 |
| MOTION_INTENSITY | 4 | HTML 原型中：入场动画一次性 300ms 淡入，按压 scale 0.97，不做持续飘落/粒子 |
| VISUAL_DENSITY | 5 | 首屏 4-6 个信息块：照片舞台 + 标题/日期 + 今日状态 + 操作入口 + 底部 Tab |
| THEME_PROMINENCE | 7 | 主题必须是视觉记忆点。不能做成"暖白底+淡紫点缀" |
| MATERIAL_METAPHOR | 和纸主卡 + 毛玻璃底部 | 照片区域：暖、有纸感。底部 Tab：轻、半透明、分层 |
| COLOR | 暖白底 + 淡藤紫主调 + 蜡笔绿点缀 | 饱和度 ≤15%。紫不能是"AI 紫"，要是"春日花园紫" |

## 七、要求 huashu-design 输出 3 个变体

### 变体 A · 克制纸感版

- **方向**：温暖、清楚、照片最大。主题显性度中等（5-6）。
- **照片**：占首屏 50-60% 面积，居中或略微偏上。和纸相框环绕。
- **主题**：小女巫角色在相框边角（不遮挡照片），魔法书背景在照片后面做微妙的纹理/底纹。不出现大面积主题色块。
- **底部 Tab**：简约毛玻璃条，淡紫微透。
- **动效**：仅入场淡入。
- **风险**：可能仍然像"淡紫色 App"而非"小女巫主题"——如果主题元素太克制。

### 变体 B · 推荐强主题版

- **方向**：糖果蜡笔紫、魔法相框舞台、较强记忆点。主题显性度 7。
- **照片**：占首屏 40-50% 面积。被一个有辨识度的魔法相框环绕（不只是圆角矩形——有"魔法书页翻开"或"花园拱门"的形态）。照片仍然完整可见。
- **主题**：小女巫角色在相框上方或侧边（作为"守护者"而非"遮挡者"）。魔法书/花园元素形成照片的背景舞台。花瓣/星光点缀在照片周围的留白区——让空白变成"花园的呼吸空间"。
- **底部 Tab**：融入主题——Tab 背景有花园纹理或魔法书页形态，选中态有柔光。
- **动效**：入场淡入（300ms）+ 选中态柔光过渡（200ms）。不做持续飘落。
- **风险**：主题元素可能开始挤压照片空间——需要严格检查照片面积和完整性。

### 变体 C · 节日装扮版（上限探索）

- **方向**：更强插画与色块。主题显性度 8-9。只作为上限探索，不默认落地。
- **照片**：占首屏 35-45% 面积。仍 C 位但主题舞台更宏大。
- **主题**：大面积花园色块形成舞台。小女巫角色更大、更显著。多层插画（前景花草+中景相框+背景花园）。
- **底部 Tab**：强主题装饰。
- **动效**：可有一次性的"花园绽放"入场序列（1-2 秒，自动结束）。不做循环。
- **风险**：可能挤压照片到"不是 C 位"的程度。上线前必须过 H5 门禁（hero asset 保护）。这个版本主要用来定位"最多能做到什么程度"，实际落地大概率选 B 或 A+B 混合。

## 八、每个变体必须说明

在 HTML/CSS 原型旁标注：

1. **照片面积和位置**：照片占首屏可视区域的百分比、在画面中的坐标区域。
2. **主题插画位置**：小女巫角色、魔法书、花园元素各自的位置，以及它们和照片的距离。
3. **首屏信息层级**：用户的视觉动线——先看哪里、再看哪里、第三看哪里。
4. **底部 Tab 处理**：Tab 的高度、颜色、选中态、是否融入主题还是保持系统风格。
5. **动效策略**：入场动画的类型和时长、交互反馈的类型和时长。
6. **可能风险**：这版最可能在哪里出问题？（照片被挤？主题太淡？动效太多？ArkTS 做不出来？）

## 九、huashu-design 输入契约

必须遵守：
- ✅ 用 HTML/CSS 做可视化对比稿，**不写 ArkTS**
- ✅ 不做营销落地页，不做说明书式页面——这是 App 首页，不是官网
- ✅ 宠物照片用灰色方块 + "宠物照片" 文字标签占位
- ✅ 三个变体并排对比（单页三个 iframe 或三个截图）
- ✅ 标注每版的设计取舍和推荐落地方向
- ✅ 图片素材：可以用 CSS 几何/SVG/纯色块模拟构图，也可以引用现有主题 PNG 作为参考图层；无论哪种方式，都要标注对应的真实资产映射

禁止：
- ❌ 使用真实网络图片作为必须依赖
- ❌ 假装这是 ArkTS 像素终稿
- ❌ 要求新权限、网络素材、账号系统
- ❌ 把三个变体做成三个独立长页面——必须可以并排对比

## 附：第一层原研哉 skill 对本页面的诊断摘要

> 来自 `kenya-hara-design/SKILL.md` 输出契约字段 3（留白诊断）和字段 9（风险标注），供 huashu-design 参考。

- **H1 诊断结论**：当前留白是被动残留，不是主动舞台。需要给空白几何边界（相框/色块/舞台）。
- **材质建议**：和纸隐喻——暖白底 + 微噪点纹理，照片区域有"相纸"触感。
- **风险标注**：变体 B 和 C 可能过淡（如果紫色太浅）或过浓（如果装饰压过照片）。主题显性度 7 的"甜点区"很窄。
- **降噪建议**：如果当前有装饰性分割线、多余的 icon、非必要的文字标签 → 删掉。空白靠结构组织，不靠装饰线分隔。
