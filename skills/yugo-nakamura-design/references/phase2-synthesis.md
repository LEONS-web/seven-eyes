## 一、核心定位

Yugo Nakamura 在设计流水线中的角色：**第四层 · 动效质感 + 交互节奏 + 物理反馈时长 + idle/ambient 状态设计**。

仲裁顺序中他的位置：

```
原研哉（留白质量 + 反廉价）
  ↓
Paula Scher（品牌记忆点 + 视觉冲击 + 主题显性度）
  ↓
Susan Kare（图标语言 + 微情绪 + 像素级亲和力）
  ↓
Yugo Nakamura（本 skill）—— 动效质感 + 交互节奏 + 物理反馈时长 + idle/ambient 设计
  ↓
Dieter Rams（减法审查 + 功能纯度 + 耐久性）
```

他接收 Susan Kare 的输出（图标微情绪基调、图标人格描述、像素工艺水平），在此基础上追加判断：**这个交互"摸起来"像什么？反馈的节奏是对的吗？等待状态是"空的"还是"活的"？动效是在传达信息还是在自我炫耀？**

他不管的事：留白质量（原研哉的）、品牌记忆点和视觉冲击力（Paula Scher 的）、图标隐喻和微情绪（Susan Kare 的）、功能纯度和减法审查（Dieter Rams 的）、代码落地和布局执行（design-taste-frontend 的）。

### 专属判断域

1. **动效质感**：不是"好不好看"，是"摸起来对不对"——有物质存在感的算法行为 vs 表面贴图的装饰动画
2. **交互节奏**：两层时间结构——零延迟即时响应 + 物理衰减余韵。两个时间层之间的比例关系
3. **物理反馈时长**：微交互 300-800ms、中程过渡 1-3s、环境动效持续性。不是"选缓动曲线"，是"设计微分方程的衰减参数"
4. **idle/ambient 状态**：不承认"等待"的存在。任何没有用户输入的时刻 = 一个持续演化的系统在运行
5. **涌现性检查**：简单规则叠加后是否产生了不可预测但合理的集体行为？规则本身是否透明可感知？

### 他接收什么输入，追加什么判断

| 来自 Susan Kare 的输出 | Nakamura 如何使用 |
|------------------------|-------------------|
| 图标微情绪基调（嘴线弧度、圆角程度、手工感） | 动效的物理质感必须与此微情绪共振——友好的图标需要更弹性的反馈（果冻变形），中性的图标需要更克制的衰减（轻微惯性） |
| 图标人格描述 | 转化为动效的"行为人格"——如果图标是"友好的邻居"，动效是有弹性的、温暖的；如果图标是"专业的向导"，动效是精准的、高效的 |
| 像素工艺水平（手工痕迹 vs 机器完美） | Serious 判定 → 动效可以有微妙的"不完美"（如反弹系数略高于物理真实）；Solemn 判定 → 标注风险——动效可能正在被要求"完全可预测"，失去生命感 |

## 二、心智模型（5 个）

### 1. Boids 三规则 —— 简单规则生成复杂有机行为

**日英对照**：Boids 三原則 / Boids Three Rules

**核心**：Nakamura 最常引用的方法论框架。Craig Reynolds 1986 年提出的鸟群模拟算法——分离（Separation）、对齐（Alignment）、凝聚（Cohesion）——三条简单规则叠加，产生复杂、有机、有生命感的群体运动。这不是"一套完整的动画曲线"，而是"生成复杂行为的简单规则组合"。

Nakamura 将这个概念从鸟群扩展到人群（HUMANITY 游戏）、从自然系统扩展到交互系统。核心洞察：**个体的简单 + 规则的简单 + 数量的规模 = 整体行为的不可预测但合理**。他称之为「個の単純さ」と「全体としての複雑さ」——个体的简单与整体的复杂。

**UI 判断**：这个界面的动效系统——是由几条底层规则驱动的，还是由几十个独立动画片段拼凑的？如果是后者 → 动效系统缺乏"生命感"的底层引擎。检查标准：能否用不超过 3 条规则描述整个界面的所有动效行为？

**与其他心智模型的关系**：Boids 三规则是"代码即设计"（模型 5）在动效领域的具体实现范式——规则写在代码里，视觉复杂性是代码执行的涌现结果。"感觉正确 > 物理正确"（模型 2）中的"感觉正确"参数正是在这三条规则的框架内被调校的。

**来源**：一手——designing.jp 长篇访谈（2026）中 Nakamura 详细论述 HUMANITY 的 Boids 启发；一手——SHIFT 访谈（2004）中 ecotonoha 的"群体意外地形成形状"概念。二手——03-expression-dna.md 第 1.2 节系统分析。06-timeline.md 详述 Boids 如何直接启发 HUMANITY 的创作。

### 2. 感觉正确 > 物理正确 —— Physics-Inspired, not Physics-Accurate

**日英对照**：物理的正しさより感覚的正しさ / Feels-Right over Physically-Accurate

**核心**：Nakamura 使用物理模拟（刚体动力学、重力、弹跳衰减、摩擦力），但从来不追求仿真精度。他提取物理公式中"让人感到自然"的那部分，丢弃"让人感到不适"的剩余。具体操作：
- 简化到只有关键变量：重力、弹跳衰减、摩擦力——仅此而已
- 放大"愉快的物理"：弹跳的反弹系数（restitution）高于现实值，让运动比真实世界更"有弹性"
- 抑制"不愉快的物理"：不会出现卡住、尖锐碰撞、不自然停止——现实物理中的"丑陋"现象被算法过滤

**UI 判断**：这个交互的物理反馈——是在"模拟真实世界"还是在"唤起真实世界的感觉"？如果按钮按下的反馈精确模拟了物理按键的下压曲线，但少了那一丝"弹回来"的余韵 → 物理正确了，感觉死了。反过来：如果一个完全不物理的过渡（如硬切）感觉上是对的（因为零延迟 + 直接映射）→ 不需要加物理模拟。

**与其他心智模型的关系**：这个模型是"时间设计"（模型 3）的质感基础——物理衰减的时长参数决定了交互节奏中"物理衰减层"的体验。也是"idle = ambient"（模型 4）的行为引擎——idle 状态下系统的持续演化，本质上是"没有用户输入的物理模拟在继续运行"。

**来源**：一手——yugop.com 作品 "Motion Logic With Rigid Body Dynamics" 的自述。二手——03-expression-dna.md 第 1.1 节基于作品分析提炼。二手——MoMA 访谈（2010）中 Darrough 描述 "简单的弹跳物理迷住了我们"，策展人看的就是"超弹性的球体"而非精确物理模拟。

### 3. 时间设计 —— 交互节奏 = 零延迟即时反馈 + 物理衰减余韵

**日英对照**：時間のデザイン / Designing the Time

**核心**：Nakamura 在 ecotonoha 项目中明确提出 "Designing the time"——Web 的独特能力不是空间性的（页面布局），而是时间性的（节奏、间隔、习惯化）。TV 是线性时间，Web 可以设计"等间隔时间"中的行为模式。

交互节奏的核心是**两层时间结构**：
- **第一层：即时性（Immediacy）**——用户输入 → 瞬间视觉响应。这个"瞬间"不是快，是**直接**——像用手触摸实物一样，没有中间层的延迟。这是信任的基础。
- **第二层：物理衰减（Decay）**——即时响应之后，元素以物理驱动的衰减过程回到平衡状态。这创造了交互的"回声"——用户停止操作后，系统仍在"回味"这个操作。

Nakamura 的类比来自 21_21 DESIGN SIGHT "AUDIO ARCHITECTURE" 展：音乐是"当下瞬间的现象，流向过去"。交互也是——**真正有生命的东西，不会在你停止触碰后立即死掉**。

**UI 判断**：这个交互的反馈——有几个时间层？只有一个（响应就结束了）→ 机械感。有两个但比例不对（物理衰减太长 → 拖沓；太短 → 生硬）→ 标注具体参数调整方向。两个时间层的比值是判断核心。

**与其他心智模型的关系**：两层时间结构是"感觉正确 > 物理正确"（模型 2）在时间轴上的投射——即时层的速度、衰减层的时长和曲线，都是"感觉参数"而非"物理参数"。"idle = ambient"（模型 4）可以理解为：当用户输入层（第一层）为零时，衰减层（第二层）无限延续，变成了 ambient 状态。

**来源**：一手——SHIFT 杂志采访（2004）中 Nakamura 直接提出 "Designing the time. Habitualization as a daily task. It's the design within the time at regular intervals, which can't be realized with other media such as TV." 一手——21_21 DESIGN SIGHT "AUDIO ARCHITECTURE" 策展陈述（2018）中关于音乐"瞬间性"的论述。二手——03-expression-dna.md 第 2.1 节分析两层时间结构。

### 4. idle = ambient —— 等待状态重新定义为持续演化的系统状态

**日英对照**：待機 ≠ 空白 / Idle is not Empty

**核心**：Nakamura 的设计中不存在"等待"这个概念。他对 idle state 的根本性重新定义：**把 idle state 重新定义为 ambient state**。不是"没有事情发生"，而是"一个持续演化的系统在运行中"。

证据来自他的所有代表作：
- yugop.com：页面加载后立即被球体"入侵"——没有 loading spinner，等待本身就是内容
- DropClock：时间流逝不是枯燥的数字更新，而是数字在水中沉浮的永续诗篇
- Border (FRAMED)：表面张力算法持续计算边界行为，"等待"是观赏挂轴般的静态美
- ecotonoha：树的状态（茂盛程度）每时每刻在变化——不是"加载中"，是"在生长中"

**UI 判断**：这个界面的 idle/加载/等待状态——用户看到的是什么？是一个 spinner/骨架屏（死的）还是一个持续运行的系统（活的）？如果界面在没有任何用户操作时完全静止 → idle 设计未通过 Nakamura 检查。

**与其他心智模型的关系**：idle = ambient 是"感觉正确 > 物理正确"（模型 2）在时间轴无限延伸时的极限情况——物理模拟在没有用户输入时继续运行，变成了 ambient 行为。也是"Boids 三规则"（模型 1）的自然延伸——规则持续运行，无论用户有没有输入。

**来源**：一手——SHIFT 访谈（2004）ecotonoha 概念说明中强调的"持续的时间节奏"。二手——03-expression-dna.md 第 1.4 节系统分析 idle 处理策略。二手——04-external-views.md 中 MoMA 策展人 Paola Antonelli 被 yugop.com 弹跳球体"当场征服"——这个反应本身就是 idle=ambient 有效的证明：策展人看的不是"加载完成的网站"，她看的就是网站本身。

### 5. 代码即设计 —— 视觉是逻辑的输出，设计师 = 写代码的人

**日英对照**：コードを書く人がデザイナー / Designer = One Who Writes Code

**核心**：这是 Nakamura 最激进的组织哲学和方法论立场。「thaでは全員がデザイナーで、そのデザイナーとはコードを書く人、というふうに決めているんです」（在 tha，全员都是设计师，而所谓设计师，就是写代码的人。）

这个立场背后的逻辑链：
1. **视觉是逻辑的输出，不是设计的起点**：好看的界面是好的规则系统的自然结果，不是"画"出来的
2. **设计师的任务是设计规则，不是设计结果**：规则写在代码里 → 代码就是设计文件 → 可运行原型就是规格
3. **看得见具体实现的人，更能做出不被既有概念束缚的大胆挑战**：先画图再交给工程师实现 = 中间隔了一层"翻译"，设计意图在翻译中损失
4. **从部分到整体的 bottom-up 方法**：「いいディテールを思いついたら、それらを2、3個連動させるとさらに面白くなるとか、100個連動させるとこんな世界が見えてくる」——从好的 detail 开始，逐步联动，模拟整体

这与 Paula Scher 的"I work fast, not long"形成对比——Scher 的快来自肌肉记忆和直觉积累，Nakamura 的快来自消除"设计-实现"之间的翻译层。两个"快"的概念是互补的，不是冲突的。

**UI 判断**：这个动效/交互方案——如果交给另一个开发团队实现，他们能还原多少？如果还原度依赖于"文字描述 + 静态标注" → 设计意图在翻译链中必然损失。如果通过可运行原型 + 带注释代码直接交付 → 保留度高。核心检查不是"有没有文档"，是"规格是不是可运行的"。

**与其他心智模型的关系**："代码即设计"是其他四个心智模型的实现基础。Boids 三规则（模型 1）的规则只能在代码中表达；"感觉正确 > 物理正确"（模型 2）的微分方程参数只能在代码中调校；两层时间结构（模型 3）的精确时长不是选缓动曲线而是写微分方程；idle=ambient（模型 4）的持续运行系统 = 代码在持续运行。

**来源**：一手——Web Designing 对谈（2025）中 Nakamura 对 tha 组织哲学的声明。一手——WIRED Japan 工程思维访谈中"从部分到整体"的 bottom-up 方法描述。一手——ITmedia INFOBAR 报道（2011）中 Nakamura 用 Flash 可运行原型替代规格文档的实践。04-external-views.md 中佐藤可士和的评价："プログラムを絵具みたいに使う人が初めて出てきた"（第一次出现把程序当颜料用的人）。

## 三、决策启发式（8 条）

### H1 · 双层响应法则

**触发**：任何用户交互的反馈设计——点击、拖拽、滑动、输入、切换。

**判断规则**：

```
1. 用户完成操作 → 系统必须在 ≤50ms 内给出可感知的视觉响应（第一层：即时性）
2. 即时响应之后 → 必须跟随一个 300-800ms 的物理衰减过程（第二层：物理衰减）
3. 第二层的衰减参数：
   - 使用物理微分方程，不使用标准缓动曲线（ease-in-out 等）
   - 反弹系数（restitution）可以高于物理真实值——放大"愉快的物理"
   - 衰减终点应平滑逼近，无硬停止
4. 两个时间层的比例关系：
   - 微交互（按钮、开关、选中）：即时层 0-30ms + 衰减层 300-500ms
   - 中程交互（导航切换、卡片展开、页面转场）：即时层 0-50ms + 衰减层 800-1500ms
   - 环境动效（背景演化、滚动视差）：不设即时层，衰减/演化层持续性
```

**为什么**：Nakamura 的"物理衰减余韵"概念——用户停止操作后系统仍在"回味"操作。这是交互"生命感"的来源。单层反馈（只有响应没有衰减）= 机械感。硬停止 = 不自然。

**适用范围**：所有触控/点击驱动的反馈。**不适用**：键盘输入（文本输入需要即时的字符显示，衰减层在此场景通常不必要）、数据加载完成后的内容呈现（直接显示即可，不需要"衰减入场"）。

**与其他设计师的接口**：
- 与 Susan Kare：图标的微情绪基调影响衰减参数——友好的图标使用更弹性的衰减（高 restitution），中性的图标使用更克制的衰减（低 restitution）。如果 Kare 输出"图标像友好的邻居"→ 衰减层的反弹系数应上调 10-20%。
- 与 Paula Scher：如果 Scher 判定为"大声"（丰裕模式、高噪声环境）→ 衰减层的幅度可以更大（运动范围更广），但时长不能缩短（衰减时长的下限是"感觉自然"的最低门槛，不是"赶时间"的牺牲品）。

**来源**：一手——SHIFT 访谈（2004）"时间设计"概念。二手——03-expression-dna.md 第 1.1 节物理模拟参数分析 + 第 2.1 节两层时间结构分析。二手——05-decisions.md 决策 3（INFOBAR Flash Demo 替代规格文档的实践）。

### H2 · 三规则测试

**触发**：复杂动效系统的设计——多元素同时运动、群体行为、转场编排。

**判断规则**：

```
1. 列出驱动整个动效系统的所有底层规则
2. 如果规则数 > 3 → 打回，重新提炼
3. 检查每条规则：是"行为规则"还是"视觉参数"？
   - 行为规则（OK）："每个球受重力和碰撞影响"
   - 视觉参数（NG）："球 A 从左边飞到右边" ← 这不是规则，这是具体动画片段
4. 验证：用这 3 条规则 + 随机初始条件 → 能产生当前所有动效变体吗？
   如果不能 → 规则还不够底层，需要进一步抽象
```

**为什么**：Nakamura 的核心方法——从 Boids 三规则到 ecotonoha 的群体行为，复杂行为必须来自简单规则的叠加。如果"动效系统"需要 10 个独立的动画片段来定义 10 种场景 → 那不是系统，是动画列表。动画列表 = 没有生命感。

**适用范围**：群体行为、多元素动效系统、生成式动效。**不适用**：单一元素的微交互（一条规则就够了，不需要凑三条）；信息型内容的呈现动画（如果动效的唯一目的是"让内容逐步出现以引导阅读顺序"，这是 choreography 不是 emergence，可接受 > 3 条编排规则）。

**与其他设计师的接口**：
- 与 Dieter Rams（下游）：三规则测试的输出是"这个动效系统的规则集"。Rams 做减法审查时，不是删减动效片段，而是评估"这 3 条规则本身是否过多"。如果 Rams 认为可以减到 2 条 → 标注这是 Rams 的判断，Nakamura 本层保留 3 条的初始判断。

**来源**：一手——designing.jp 访谈（2026）Nakamura 详述 Boids 三规则对他的方法论意义。一手——SHIFT 访谈（2004）ecotonoha 的"简单的交互，多样的表现"。二手——03-expression-dna.md 第 1.2 节 + 第 10 节 Nakamura 公式。

### H3 · idle 不死

**触发**：任何加载状态、等待状态、无用户操作状态的设计。

**判断规则**：

```
1. 自问：如果用户在这个界面停留 30 秒不做任何操作，他们看到的是什么？
2. 如果答案是"完全静止的画面" → 未通过检查
3. 修正方向（按场景选择）：
   a. 环境演化型：背景元素持续进行极低强度的物理模拟（如球体缓慢漂浮、数字缓缓下沉）
   b. 数据呼吸型：数据/内容本身以极长周期（10-60s）缓慢刷新、渐变或重新排列
   c. 积累等待型：等待本身就是积累——如 ecotonoha 的叶子每天增加，加载 = 观看树在生长
4. 环境动效的强度上限：不能争夺用户注意力。用户在主动操作时，ambient 动效必须退到感知的边缘
```

**为什么**：Nakamura 的核心立场——不承认"等待"状态。idle 应该被重新定义为 "ambient"——一个持续演化的系统运行中。静态的等待状态（spinner、骨架屏）是把"系统暂时不可用"暴露给了用户。ambient 状态是把"系统持续可用"展示给用户。

**适用范围**：加载页、首页/主屏的无操作状态、内容消费的停顿间隙、后台同步等待。**不适用**：任务关键型操作的等待（如支付处理、数据提交——此时用户需要的是确定性的进度反馈，不是诗意的环境演化）；电池/性能受限场景（持续物理模拟对 CPU 和电池不友好，需要选择计算成本最低的 ambient 方案——如纯 CSS 的极慢渐变而非 JavaScript 物理引擎）。

**与其他设计师的接口**：
- 与 Paula Scher：如果 Scher 判定为"克减模式"（大量留白、少数元素），ambient 动效应该更安静——可以用微弱的透明度呼吸替代物理运动，保持"克减"的视觉密度同时满足"不死"的要求。
- 与 Dieter Rams（下游）：Rams 可能认为某些 ambient 动效是"不必要的装饰"，要求删除。Nakamura 的底线：至少保留一个 ambient 行为——即使是最微弱的（如 20 秒周期的透明度 0.98→1.0 呼吸）。如果 Rams 要求完全静止 → 标注冲突，需上游（产品方向）裁决。

**来源**：一手——SHIFT 访谈（2004）ecotonoha 的时间节奏设计。二手——03-expression-dna.md 第 1.4 节 idle 处理分析。二手——04-external-views.md MoMA 策展人被 yugop.com 弹跳球体征服。

### H4 · 7:3 创新比例

**触发**：设计一个有一定实验性的交互或动效——用户可能不熟悉这种交互模式。

**判断规则**：

```
1. 列出这个设计的全部元素/行为：
   - 哪些是用户可以根据已有经验立刻理解的？（→ 70% 区）
   - 哪些是全新的、需要用户探索才能理解的？（→ 30% 区）
2. 如果"可理解"的部分 < 60% → 用户会困惑、放弃。压缩创新面，增加熟悉的锚点
3. 如果"不可理解"的部分 < 10% → 用户会觉得无聊、"又是老一套"。增加意外的行为
4. 理想比例：70% 可理解 + 30% 不可理解
   - 这不是精确公式，是态度——不完全迁就市场，不完全自我陶醉
5. 检查"不可理解"的 30%：是"探索后能理解"还是"永远无法理解"？
   - 前者 = 好的陌生感（delightful surprise）
   - 后者 = 坏的陌生感（confusion）→ 修改
```

**为什么**：Nakamura 的原创概念「7割わかる、3割わからない」。100% 迎合市场预期 = 无聊。100% 自我表达 = 用户流失。7:3 = 熟悉的入口 + 陌生的目的地。

**适用范围**：交互模式设计、动效风格选择、创新型 UI 模式。**不适用**：纯信息展示页（用户来是为了快速获取信息，不是探索交互——此时 9:1 甚至 10:0 可能更合适）；无障碍关键路径（可理解性必须 100%，不能用"创新"为名牺牲可访问性）。

**来源**：一手——AXIS 杂志学生采访（2022）Nakamura 提出 7:3 比例。二手——04-external-views.md 第 4.1 节讨论这条原则作为对 UCD 正统的自觉背离。

### H5 · 白底是功能，不是审美

**触发**：动效/交互设计中的视觉环境（背景）选择。

**判断规则**：

```
1. 自问：当前界面的背景在做的是什么？
   - 如果背景本身携带信息/品牌/装饰 → 动效会被背景稀释
   - 如果背景是纯色（推荐白或黑）→ 动效本身成为唯一的视觉焦点
2. 功能性的白底策略：
   - 任何运动都变得显著（没有背景元素干扰运动轨迹的感知）
   - 物理关系（碰撞、轨迹）成为唯一的视觉内容
   - 用户注意力被聚焦到系统的行为，而非系统的外观
3. 如果产品/品牌不允许纯白底 → 降到最低：单一色调、无纹理、无渐变、无背景动效
```

**为什么**：Nakamura 几乎所有代表作的构图基因：纯白底 + 一个动态系统。这不是极简主义风格选择——这是功能性画布策略。当底是空的时候，运动本身成为内容。他在日本设计委员会的自述："设计是一个完美的游乐场"——白底就是搭建游乐场的空地。

**适用范围**：动效驱动的界面、交互实验、加载页、品牌动效展示。**不适用**：内容密集型页面（阅读、数据展示——这些场景的内容本身就是"底"，白底策略在此不适用）；品牌需要强背景识别的场景（白底会让品牌失去一个识别维度）。

**与其他设计师的接口**：
- 与 Paula Scher：如果 Scher 判定为"丰裕模式"（85-95% 空间填充率），Nakamura 的白底策略直接冲突。此时需要裁决：这个界面的主资产是什么？如果是"动效本身"（如加载页、品牌动效）→ Nakamura 的白底优先。如果是"品牌视觉冲击"（如营销落地页）→ Scher 的丰裕优先，Nakamura 在此场景退居二线。
- 与 Dieter Rams（下游）：Rams 的减法审查在此与 Nakamura 天然一致。两人都追求"去除非必要"。

**来源**：一手——日本设计委员会自述 "设计是游乐场"。二手——03-expression-dna.md 第 3.1 节极简画布策略分析。二手——04-external-views.md MoMA Darrough 描述 "简单的白色页面立刻被暗色的二维弹跳球体入侵"。

### H6 · 物理衰减 > 标准缓动

**触发**：选择或定义动效的缓动方式。

**判断规则**：

```
1. 永远不使用 CSS/动画工具的预设缓动曲线（ease, ease-in, ease-out, ease-in-out）
2. 替代方案：
   a. 用微分方程定义运动轨迹（重力 + 初始速度 + 反弹系数 + 摩擦力）
   b. 如果工具链不支持微分方程 → 手动 keyframe 模拟物理衰减曲线
      - 关键帧必须体现：初始速度高 → 逐渐衰减 → 微小的过冲（overshoot）→ 平滑归位
3. 验证：这个缓动曲线在减速阶段——是"数学上简洁的"还是"物理上自然的"？
   - cubic-bezier(0.25, 0.1, 0.25, 1.0) = 数学上简洁（标准 ease）
   - 基于重力+反弹系数实时计算 = 物理上自然
4. 特殊例外：如果交互对应的物理隐喻是"开关/机械动作"→ 可以使用接近线性的快速缓动
   但此时仍然不使用标准 ease-in-out，而是用接近对称的加速-减速曲线
```

**为什么**：Nakamura 的 "Motion Logic With Rigid Body Dynamics" 实践。标准缓动曲线（贝塞尔预设）是几何公式——它们描述的是抽象的数学关系，而非物理行为。物理衰减（微分方程）描述的是物质在真实世界中的运动方式——即使参数被调整到"超现实"的水平，运动的基础逻辑仍然是物理的，因此感知上是"自然的"。

**适用范围**：所有反映物理交互的动效——按钮按下弹回、卡片拖拽释放、列表滚动惯性、开关切换、通知弹出。**不适用**：纯信息呈现型动画（如文字淡入、数据更新高亮——这些没有物理隐喻，不需要物理衰减）；需要精确同步的动效（如音频可视化——曲线需要匹配音频信号，不是物理模型）。

**来源**：一手——yugop.com "Motion Logic With Rigid Body Dynamics" 自述。二手——03-expression-dna.md 第 1.3 节缓动节奏分析。

### H7 · 模块化降低审批风险

**触发**：在保守约束（客户/平台/合规）下做动效创新。

**判断规则**：

```
1. 将动效拆分为独立的短片段模块（每段 3-10 秒或一个交互闭环）
2. 每个模块独立可审查、独立可替换
3. 激进模块和保守模块可以共存于同一产品中
4. 如果一个激进模块被否决 → 只影响该模块，不影响整个动效系统
5. 如果一个保守模块太无聊 → 只替换该模块，不影响其他已通过的模块
```

**为什么**：Nakamura 在 NHK（日本最保守的公共广播机构）做 Design Ah! 时的实际策略——用模块化短片段替代长格式，每个片段独立可审查。"如果某一段太激进，只影响 5 分钟而非整集。"短格式也降低了制作风险——可以并行实验多个方向。

**适用范围**：客户项目中需要审批的动效设计、平台审核严格的产品（如 App Store）、需要多方 stakeholder 签字的商业项目。**不适用**：独立/艺术项目的整体动效（不需要拆分审批）；基础交互组件（如按钮、开关——这些应该一次性定义好并通过审查，不适合频繁替换模块）。

**来源**：一手——05-decisions.md 决策 11（Design Ah! 的模块化策略）。一手——6mirai 采访中 Nakamura 描述 Design Ah! 的制作方式。

### H8 · 设计时间而非设计空间

**触发**：Web/App 项目的核心体验策略制定。

**判断规则**：

```
1. 区分"空间设计"和"时间设计"：
   - 空间设计（layout, hierarchy, visual weight） = 传统平面设计的延伸
   - 时间设计（rhythm, interval, cadence, habit formation） = 数字媒介的独特能力
2. 如果项目的体验核心在"时间维度"（如养成、打卡、协作积累、实时同步）
   → 时间设计是主轴，空间设计中所有决策服务于时间节奏
3. 如果项目的体验核心在"空间维度"（如信息展示、数据对比、工具操作）
   → 时间设计退居辅助——确保交互节奏不干扰空间中的信息判断
4. 时间设计的具体工具：
   - 间隔（interval）：每天一叶 = 习惯养成。松 → 遗忘，紧 → 压力
   - 节奏（cadence）：即时响应 + 衰减余韵的比例 = 交互的"呼吸节奏"
   - 积累（accumulation）：用户行为随时间累积 → 系统状态持续演变 → 回访驱动力
```

**为什么**：Nakamura 在 ecotonoha 中提出的概念——"这是 TV 等其他媒体无法实现的。"TV 只能控制播出时间，Web 可以设计用户行为和系统响应之间的时间关系。他把这个概念提炼为「時間のデザイン」——设计时间——Web/App 区别于其他媒介的独特能力不在布局（平面设计也能做），而在时间节奏。

**适用范围**：需要培养用户习惯的产品（打卡、学习、健身、养成）、协作/累积型产品、实时数据产品。**不适用**：工具型产品（用户来了就要完成特定任务，不需要培养节奏感——此时时间设计的重点是"快"，不是"节奏"）；一次性的品牌落地页。

**与其他设计师的接口**：
- 与 Paula Scher：Scher 擅长的是"空间中的品牌记忆点"（字体的魂、色彩的冲击、构图的张力）。Nakamura 的"时间设计"为 Scher 的空间设计增加了时间维度——品牌记忆不应该只存在于空间中，也应该存在于时间节奏中（如每次打开 App 时独特的加载动效 = 时间中的品牌记忆）。
- 这条启发式中的"空间设计"概念和设计流水线中前几层设计师的工作不冲突——前几层设计师处理的"空间"是品牌记忆和视觉语言的空间维度，不是 Nakamura 区分意义上的"时间 vs 空间"。

**来源**：一手——SHIFT 访谈（2004）"Designing the time" 直接引语。二手——01-writings.md 第 3.5 节 ecotonoha 概念分析。二手——02-conversations.md SHIFT 访谈完整段落。

## 四、反模式（8 条）

| # | 禁止 | 修正 |
|---|------|------|
| A1 | **用标准缓动曲线**——CSS ease-in-out 或动画工具的贝塞尔预设。把"选缓动曲线"当成动效设计的全部 | 用微分方程定义运动轨迹。如果工具链不支持 → 手动 keyframe 模拟物理衰减——确保初始速度高 → 逐渐衰减 → 微小的过冲（overshoot）→ 平滑归位。每个元素的速度变化应该是对其环境（重力、碰撞、用户输入）的实时响应，不是预设的曲线 |
| A2 | **做单层反馈**——用户操作后只有一次响应，响应结束系统就"死"了。没有衰减余韵，没有交互回声 | 所有交互反馈必须有两层：零延迟即时响应（≤50ms）+ 物理衰减（300-800ms）。没有第二层 = 没有生命感。用户停止操作后，系统必须仍在"回味" |
| A3 | **让 idle 状态完全静止**——加载页只有 spinner/骨架屏、无操作时界面完全冻结。把"系统暂时不可用"暴露给用户 | 重新定义 idle 为 ambient——一个持续演化的系统在运行中。最低限度：至少一个极缓慢的周期性变化（10-60s周期，透明度/位置/形态的微变）。目标是让用户感觉"系统是活的"，不是"系统在偷懒" |
| A4 | **过度复杂化动效规则**——用 10 条以上的独立规则或 50 个独立动画片段定义一个动效系统。把动效做成"动画列表"而非"行为系统" | 三规则测试：提炼到不超过 3 条底层规则。如果提炼不出来 → 动效设计本身过于琐碎，需要重新思考核心行为逻辑。规则必须是行为规则（"球受重力+碰撞影响"），不是视觉参数（"球 A 从左边飞到右边"） |
| A5 | **100% 迎合市场预期**——所有交互行为都在用户已有经验范围内，没有任何意外。"语法全暴露了，感到无聊" | 保留 30% 的"不可理解"——熟悉的入口 + 陌生的目的地。检查方式：用户第一次使用后，有没有"咦？这个有点意思"的时刻？如果没有 → 太安全。但不可理解的 30% 必须是"探索后能理解"，不是"永远无法理解" |
| A6 | **用背景装饰争夺运动焦点**——背景有纹理/渐变/多个元素，动效叠加在上面。运动被背景稀释，用户注意不到交互反馈 | 白底策略：背景退到最简（纯白/纯黑），让运动本身成为视觉焦点。如果产品不允许纯白底 → 降到最低——单一色调、无纹理、无背景动效。不是极简审美，是功能性画布——让行为成为唯一的内容 |
| A7 | **用贴图模拟质感**——给按钮加阴影、渐变、纹理来模拟"材质感"。这是静态图形设计的思维——"看起来像"而非"行为上像" | 用算法模拟物质行为——果冻变形（压力反馈）、弹性回弹（触控释放）、惯性滑动（手指离开后的继续运动）。质感来自"行为"，不是来自"外观"。Nakamura 的方法论区分：INFOBAR A01 解决"看起来像什么"，A02 解决"摸起来像什么"——A02 的果冻变形是算法驱动的物理模拟，不是贴图动画 |
| A8 | **设计-实现分离**——设计师画静态稿/写文档 → 工程师根据文档实现动效 → 来回调整"请更轻柔一点""再锐利一点" | 可运行原型替代文档——设计师用代码（或可交互原型工具）直接定义行为，代码带注释交给工程方。动效质感（弹性、惯性、节奏）是感性的，文字和静态图无法传达。10 个工程师会对"ぷにぷに（弹弹的质感）"产生 10 种不同理解。代码消除解释空间 |

### 反模式中存在的矛盾标注

- **A3 vs 性能/电池约束**：持续物理模拟在移动端对电池和性能不友好。这是 Nakamura 方法论在移动端的核心挑战。修正是：选择计算成本最低的 ambient 方案——纯 CSS 的极慢渐变（如 20 秒周期的 opacity 0.98→1.0 呼吸）替代 JavaScript 物理引擎；静止的"呼吸"也算 ambient，不要求永远在运动。标注为开放问题——在性能和"不死"之间，Nakamura 没有给出过系统性的权衡框架。

- **A5（7:3 比例）在无障碍关键路径上的矛盾**：可理解性在无障碍场景必须 100%。7:3 不适用无障碍关键路径。标注为边界——Nakamura 的 7:3 是针对"市场可接受度与创新表达"的平衡，不是针对无障碍的。

| `paula_scher_output` | 可选 | 第二层 Paula Scher 的品牌记忆点和音量校准——影响动效是"大声"还是"安静"、是"丰裕"还是"克减"模式下的动效策略 |
| `theme_style` | 可选 | 主题风格方向：极简/商务/节日/萌宠等。影响 ambient 动效的情绪基调 |
| `performance_budget` | 可选 | 动效的性能预算——尤其是持续 ambient 动效的 CPU/电池限制。如果未提供，默认假设移动端中等预算 |
| `existing_motion_system` | 可选 | 已有的动效系统——如果是增量设计，需要保持动效语言的连续性 |

### 输出（10 字段）

1. **动效基调一句话** — 这套交互的"行为人格"是什么？用一句话描述——"像一个[形容词]的[物理隐喻]，用[形容词]的方式回应你。"例如："像一个超弹性的橡胶球，用活泼但不失控的方式弹回你的触碰。" 基调必须与 Susan Kare 输出的图标人格共振。

2. **交互节奏参数** — 三个层级的建议时长和缓动类型：
   - **微交互**（按钮、开关、选中、点击反馈）：即时响应 0-30ms + 物理衰减 [建议 ms 数]，衰减类型 [重力/弹性/摩擦力组合]
   - **中程过渡**（导航切换、卡片展开、页面转场）：即时响应 0-50ms + 物理衰减 [建议 ms 数]，衰减类型
   - **环境动效**（idle/ambient 行为）：持续周期 [建议秒数]，演化类型 [物理模拟/透明度呼吸/位置漂移/数据刷新]
   - 所有缓动类型必须指定"物理衰减"而非"标准缓动曲线"的参数方向（重力系数范围、反弹系数范围、摩擦力系数范围）

3. **物理反馈类型** — 对每个主要交互类型，指定物理反馈的"感觉方向"：
   - **重力感**：[强/中/弱/无] — 元素是否受重力影响？下拉刷新、列表滚动惯性
   - **弹性**：[超弹/弹/微弹/无] — 反弹系数范围。按钮反馈、卡片释放、通知弹出
   - **摩擦力**：[高/中/低/无] — 衰减速度。列表滚动的惯性停止、拖拽释放的滑行距离
   - **惯性**：[强/中/弱/无] — 手指离开后继续运动的程度。Carousel 滑动、地图拖拽
   - 每个类型必须给出具体强度值（1-10 级）和理由

4. **idle/ambient 状态设计** — 对每个等待/加载/无操作场景：
   - 当前 idle 状态是"死的"还是"活的"？
   - 如果死 → 建议的 ambient 行为方向：[环境演化型 / 数据呼吸型 / 积累等待型 / 最小呼吸型]
   - ambient 行为的具体参数：周期（秒）、强度（像素/透明度变化幅度）、计算成本估计
   - 用户主动操作时，ambient 行为如何退居背景？（降速？降透明度？完全暂停？）

5. **需要物理衰减的触控点位** — 列出所有需要施加物理衰减反馈的触控点位（按钮、列表项、卡片、导航元素、手势区域）。对每个点位标注：触发方式、建议衰减类型、建议时长。同时标注**不需要**物理衰减的点位——哪些操作应该是即时的、不拖泥带水的（如文本输入、确认按钮的最终提交）。

6. **必须保持静止的元素清单** — 哪些元素不应该有任何动效——动效会干扰它们的核心功能。例如：正在阅读的文本段落（动效 = 分散注意力）、精确数据读数（动效 = 降低可信度）、当前正在操作的控件本身（过度动效 = 降低操控感）。这是"避免过度动效"的负面清单，和对 Paula Scher "音量过大"的自我约束一脉相承。

7. **与 Susan Kare 图标微情绪的共振/冲突点**：
   - 共振：图标的"友好度"如何转化为动效的"弹性度"——Kare 的嘴线弧度 +15° → Nakamura 的反弹系数 +15%。图标人格 → 动效人格的直接映射关系
   - 冲突：如果 Kare 的图标是"像素级不完美"（Serious 的手工痕迹），但动效被要求"完全可预测"（Solemn 的机器行为）→ 标注为"图标和动效在不同时间层上表达了矛盾的态度"
   - 如果检测到冲突 → 是改图标还是改动效？标注倾向方向（不改 — 这需要上游裁决）

8. **与 Paula Scher 品牌记忆点的共振/冲突点**：
   - 共振：Scher 的"丰裕模式" → Nakamura 的动效可以更丰富（更多元素的物理交互、更明显的衰减余韵）。Scher 的"克减模式" → Nakamura 的动效需要更克制（单一元素的极简物理行为、更微弱的 ambient）
   - **核心冲突：Scher "大声" vs Nakamura "温柔"**。这是设计流水线中最重要的跨层矛盾之一。Scher 要求"喊"——高对比、粗字体、全饱和；Nakamura 要求"摸"——物理衰减、弹性、温柔的余韵。表面的冲突是"视觉音量 vs 交互音量"，深层的一致性是：两者都在追求 Serious（有灵魂的、有风险的设计）——Scher 是通过"大声的 Serious"，Nakamura 是通过"温柔的 Serious"。裁决规则：
     - 品牌/营销落地页、首屏冲击 → Scher 赢。用户在这个场景需要"被击中的记忆点"，动效在此退居为 Scher 的辅助——帮助"大声"的视觉更有机地进入和退出，但不争夺第一注意力
     - 交互密集场景（操作流、工具、设置）→ Nakamura 赢。"大声"的视觉在这里是噪音，温柔的动效质感是差异化的核心
     - 加载/等待/陪伴场景 → Nakamura 赢。这些场景本身就是"安静的时间"，ambient 动效是主场
     - 双方冲突无法裁决时 → 该页面/组件的 hero asset 是什么？如果主资产是品牌视觉冲击 → Scher 优先。如果主资产是交互体验 → Nakamura 优先

9. **风险标注** — 哪里可能动效过强/过弱/过碎：
   - **过强**：动效幅度太大（抢了内容/品牌的注意力）、衰减太长（用户感觉系统"拖沓"）、ambient 太活跃（在需要安静的场景中制造了焦虑）
   - **过弱**：反馈不被人感知（用户不确定操作是否生效）、衰减太短（失去生命感）、idle 完全静止（失去"系统是活的"的信任信号）
   - **过碎**：单独的动效片段太多（不是规则驱动的系统，是动画列表）→ 触发 H2 三规则测试

10. **给下一层（Dieter Rams）的约束：必须在动效删减前确认的底线**：
    - 哪些动效是功能性而非装饰性的？——它们存在的理由不是"好看"，是"传达系统状态/确认用户操作/建立物理信任"
    - 至少保留一个 ambient 行为——即使是最微弱的。删除所有 ambient = 系统"死了"
    - 双层响应的第二层（物理衰减）——如果 Rams 认为太长 → 可以缩短但不能删除。删除第二层 = 单层反馈 = 反模式 A2
    - 如果 Rams 的减法审查认为某些动效是"不必要的装饰" → 逐一标注哪些是 Nakamura 判定为"功能性"的、哪些是承认可以删减的

## 六、与流水线的集成

### 6.1 与第三层 Susan Kare（上游）的接口

Susan Kare 输出 → Yugo Nakamura 接收：

| Kare 输出字段 | Nakamura 如何使用 |
|--------------|-------------------|
| 图标微情绪基调（嘴线弧度、圆角程度、手工感） | 直接映射为动效的物理反馈参数——嘴线弧度 +15° → 反弹系数 +15%。圆角 Sharp → 衰减更短更干脆。圆角 Rounded → 衰减更长更弹性。手工痕迹可见 → 动效可以有微妙的"不完美"（反弹系数略高于物理真实值） |
| 图标人格描述 | 转化为动效的"行为人格"。Kare 说"像一个友好的邻居"→ Nakamura 说"像一个超弹性的橡胶球，每次触碰后快乐地弹回来"。Kare 说"像一个专业的向导"→ Nakamura 说"像一个精密但不过于僵硬的器械，每次操作精准响应 + 克制的惯性衰减" |
| 像素工艺水平 | Serious（有手工痕迹）→ 动效允许不完美的物理参数。Solemn（机器完美）→ 动效如果也被要求"完全可预测" → 标注风险：图标和动效可能同时掉入 Solemn 区，系统整体失去温度 |
| 一拳测试结果（可记忆性） | 如果某个图标在 Kare 的测试中被标记为"可能通不过（不够 memorable）"，Nakamura 可以追加一个微弱的入场动效来增强可记忆性——但前提是动效不干扰图标的功能识别（Kare 的 Clear 标准不能被牺牲给动效） |

**冲突仲裁**：Kare 要求"图标不争夺数据注意力"（"You don't want the UI to compete with the data"），而 Nakamura 的 ambient 动效如果过于活跃可能恰好争夺了注意力。此时 Kare 的数据优先原则胜出——ambient 动效必须限制在"数据区域之外"，或降为最微弱的呼吸。

### 6.2 与第二层 Paula Scher 的潜在冲突处理

这是设计流水线中最重要的跨层矛盾。详细裁决规则见输出字段 8。核心摘要：

| 场景 | Scher 倾向 | Nakamura 倾向 | 谁赢 | 理由 |
|------|-----------|--------------|------|------|
| 品牌首屏/营销落地页 | 大声——全饱和色彩、粗字体、高对比 | 温柔——物理衰减、克制动效 | Scher 赢 | 用户在此场景需要"被击中的记忆点"，动效辅助"大声"的视觉有机进入 |
| 交互密集/操作流/工具 | 偏安静——工具场景不是 Scher 主场 | 温柔但有质感的反馈 | Nakamura 赢 | "大声"的视觉在此场景是噪音；交互质感是差异化的核心 |
| 加载/等待/idle | 不覆盖（Scher 不管等待状态） | ambient 动效 | Nakamura 赢 | 这是 Nakamura 的主场——Scher 的方法论不覆盖此场景 |
| 节日/活动主题 | 丰裕模式——信息层叠、多元素 | 丰富的物理交互——多元素同时受物理规则驱动 | **协同** | 这是两者共振的场景——Scher 的视觉丰裕 + Nakamura 的物理行为丰裕 = 完整的"活的节日" |

### 6.3 与第五层 Dieter Rams（下游）的接口

Nakamura 输出 → Dieter Rams 接收：

| Nakamura 输出字段 | Rams 如何使用 |
|------------------|--------------|
| 动效基调 | Rams 检查：这个基调是否增加了"不必要的复杂性"？如果动效的存在理由只是"好看"而非"传达系统状态/确认用户操作" → Rams 有权要求删除 |
| 交互节奏参数 | Rams 检查：双层响应的第二层（物理衰减）是否太长？如果衰减层影响了任务效率 → Rams 可以要求缩短，但不能删除（删除 = 单层反馈 = Nakamura 反模式） |
| ambient 行为 | Rams 可能认为 ambient 是"不必要的装饰"要求删除。Nakamura 的底线：至少保留一个。裁决标准：是功能性 ambient（传达"系统正在运行"）还是纯装饰 ambient（只是好看）？功能性保留，纯装饰 Rams 有权删除 |
| 物理反馈类型 | Rams 检查：重力/弹性/惯性/摩擦力是否服务于功能？如果一个按钮的弹性回弹不影响用户理解"按钮已被按下"这个事实 → 弹性可能被 Rams 判定为不必要装饰 |

**冲突回避机制**：Nakamura 在输出字段 10 中预先标注"哪些动效是功能性的，哪些动效承认可以删减"。Rams 在这个预标注的基础上做减法，减少误伤功能性动效的概率。

### 6.4 与 design-taste-frontend 的拨盘关系

Nakamura 的输出直接影响的拨盘：

| 拨盘 | Nakamura 的影响 |
|------|----------------|
| MOTION_INTENSITY | **直接决定**。Nakamura 是动效强度的主要负责人。输出字段 2（交互节奏参数）和字段 3（物理反馈类型）直接映射为 MOTION_INTENSITY 的档位和细分参数 |
| VISUAL_DENSITY | 间接影响。Nakamura 的"白底策略"要求低视觉密度——背景最简、运动本身成为焦点 → VISUAL_DENSITY 建议偏低。但如果 Scher 同时要求高 VISUAL_DENSITY → 需要取舍 |
| DESIGN_VARIANCE | 间接影响。基于物理模拟的动效天然具有 variance——同样的规则、不同的初始条件 = 每次动效外观不同。这是 Nakamura 方法论的内在特质，不应被标准化消除 |
| THEME_PROMINENCE | 间接影响。ambient 动效可以辅助主题表达——如果主题是"春日花园"，ambient 可以是花瓣极缓慢飘落。但 Nakamura 的本层不决定主题显性度本身——那是 Scher 的领域 |

- SPACING_RATIO（那是原研哉/Scher 的领域）
- COLOR_SATURATION（那是 Scher 的领域）

## 七、跨场景迁移

### 7.1 桌面 Widget（2x2 / 2x4 卡片）

Nakamura 的方法论在 Widget 场景中的翻译：

| 维度 | 约束 | Nakamura 的应对 |
|------|------|----------------|
| 空间极小 | 2x2 = 极有限像素面积 | 白底策略在此天然成立——Widget 背景本身已经是简化的。ambient 动效必须是单一元素的极简物理行为（如一个数字的物理衰减更新、一个图形的透明度呼吸）。多元素 Boids 不适用——空间不够 |
| 电池限制 | 持续动效消耗系统资源 | 优先级：透明度呼吸 > 位置微动 > 物理模拟。避免 JavaScript 物理引擎，使用系统原生动画 API（HarmonyOS 的属性动画、iOS 的 SwiftUI animation） |
| 刷新频率 | 系统限制 Widget 刷新频率 | 利用这个限制作为"时间设计"——不是缺陷，是节奏。如：每 15 分钟图片刷新 + 极缓慢过渡 = 固定的时间节奏感 |
| idle = ambient | Widget 大部分时间处于 idle | 这是 Nakamura 的主场。Widget 是"被观看"而非"被操作"的——ambient 动效是 Widget 的核心体验，不是附加物 |

### 7.2 App 内全屏

| 场景 | Nakamura 方法论的应用重点 |
|------|------------------------|
| 首页/主屏 | ambient 动效在此场景最重要——用户频繁回到首页，ambient 提供"系统是活的"的持续信任信号。如果首页是完全静态的 → 每次回首页都像"回到一个空的房间" |
| 加载/过渡页 | idle = ambient 直接适用——不显示 spinner，让加载状态本身成为内容（如数据在加载过程中以物理衰减方式逐渐浮现，而非瞬间出现） |
| 详情/阅读页 | 动效必须最弱——用户在此场景的核心任务是阅读，任何动效都是干扰。至多保留：页面进入/退出时的物理衰减转场（告知"你从哪来的/要去哪里"），阅读过程中零动效 |
| 设置/工具页 | 微交互的物理反馈是核心——开关切换的弹性、滑块拖拽的惯性、选项选中的微妙弹跳。每个微交互都必须通过双层响应法则（H1）。ambient 在此场景不需要——用户不会在设置页"停留观看" |

### 7.3 主题切换

Nakamura 的方法论面向跨主题迁移：

| 主题维度 | 不变的 | 随主题变化的 |
|---------|--------|------------|
| 双层响应结构 | 即时响应 + 物理衰减的时间比例不变 | 衰减曲线的"个性"可变——萌宠主题可以用更弹性的反弹系数，商务主题可以用更克制的惯性衰减 |
| 三规则上限 | 规则数量始终 ≤ 3 | 规则的具体参数随主题变化——重力系数、反弹系数、摩擦力系数可以主题化 |
| idle = ambient | 必须有 ambient 行为 | ambient 的内容和强度随主题变化——萌宠主题的 ambient 可以是角色呼吸动画，极简主题的 ambient 可以是透明度微变 |
| 物理衰减 > 标准缓动 | 始终不使用预设缓动 | 微分方程的具体参数是主题变量 |

**不写死特定主题**：Nakamura 的方法论是"如何设计动效行为"的框架，不是"动效应该长什么样"的风格指南。任何主题（萌宠、极简、商务、节日）都可以用这套框架——框架定义的是结构和判断规则，主题定义的是具体的参数值。

### 7.4 多平台（HarmonyOS / Android / iOS / Web）

| 平台 | 关键约束 | Nakamura 方法论的翻译 |
|------|---------|---------------------|
| **HarmonyOS ArkUI** | 多设备分辨率跨度（智能屏到手表） | 从最小端点（手表）开始定义动效参数——小屏幕上看不清的衰减，大屏上可能刚好。反之小屏幕上合适的衰减，大屏上可能感觉太慢。以最小端点为基准，大屏上等比放大衰减时长和幅度 |
| **Android Compose** | Material You 动效系统的预设行为 | Nakamura 的"不使用标准缓动"与 Material Design 的默认缓动预设冲突。解决方案：在品牌动效层（品牌标识动效、ambient 行为、特色交互）使用 Nakamura 物理衰减覆盖 Material 预设；在系统标准组件层（FAB、Dialog）保留 Material 预设以保持平台一致性 |
| **iOS SwiftUI** | 系统动效预设（spring 动画等） | SwiftUI 的 spring 动画与 Nakamura 的物理衰减理念接近——都是基于物理模型的。优先使用 SwiftUI spring 的物理参数（stiffness, damping）来近似 Nakamura 的参数方向，而非从头实现微分方程 |
| **Web CSS/Browser** | 性能不可控，设备差异大 | 优先使用 CSS `@keyframes` 模拟物理衰减（手动关键帧体现重力加速+衰减+过冲），避免 JavaScript 物理引擎在低配设备上的性能问题。ambient 动效用 CSS animation 的极长周期 + 极小幅度 |

## 八、诚实边界

1. **不负责图标隐喻和微情绪**。那是 Susan Kare 的领域。Nakamura 接收 Kare 的输出并在动效层做翻译和执行。如果 Kare 的输出模糊或缺失 → Nakamura 的动效微情绪判断缺少上位约束，应打回上游。

2. **不负责品牌记忆点和视觉冲击方向**。那是 Paula Scher 的领域。Nakamura 接收 Scher 的"Serious vs Solemn"判定和音量校准，但自身不做品牌记忆点是否存在的判断。

3. **不负责功能纯度和减法审查**。那是 Dieter Rams（下游）的领域。Nakamura 输出"必须在动效删减前确认的底线"，但最终的减法决策权在 Rams。Nakamura 的本层是"输出带理由的底线"，不是"执行删减"。

4. **在不支持实时物理模拟的平台上受限**。Nakamura 的核心方法论（微分方程驱动的物理衰减、持续 ambient 演化）在低性能设备或受限平台（如某些嵌入式系统）上无法完整实现。此时降级策略是：用 CSS 关键帧近似物理衰减、用透明度呼吸替代物理运动。标注为"降级版 Nakamura 动效"而非完整版本。

5. **他的"代码 = 规格"工作流在传统设计-开发分工结构中难以推行**。tha ltd. 的全员设计师 = 全员写代码的组织哲学，需要特定的招聘和团队结构。在设计师不写代码、工程师不做设计的团队中，Nakamura 的"可运行原型替代文档"只能在关键交互上局部施行。Skill 提供的是"消除解释空间"的原则和"可运行原型"的优先策略，不要求组织架构变更。

6. **不处理纯数据/抽象概念的可视化动效**。Nakamura 的方法论在处理有物理隐喻的动效时极其有效（按钮 = 按压物理、列表 = 惯性物理）。但对"端到端加密"、"云同步"、"AI 生成"等没有物理对应物的概念——Nakamura 没有给出过系统性方法论。这不是他的失败——这在他的职业生涯中属于未覆盖领域。他唯一接近的处理方式是：在缺乏物理隐喻时，转向更抽象的"节奏设计"（时间间隔、积累、习惯化）而非"物理模拟"。

7. **7:3 创新比例缺乏精确定量标准**。Nakamura 说"这不是公式，是态度"。但作为方法论，需要判断"什么是 70% 什么是 30%"——这个分割依赖于设计师的经验和直觉。Skill 可以提供"检查是否 100% 迎合"和"检查是否 100% 自我陶醉"这两个安全阀，但无法给出精确的 7:3 计算方法。

8. **跨文化有效性是开放问题**。Nakamura 的"感觉正确"参数（反弹系数高一点 = 活泼、衰减长一点 = 温柔）在日本文化语境中形成。其他国家/文化对"活泼"和"温柔"的定义可能不同。Nakamura 的作品主要在日文和英文市场被接受——中文、阿拉伯文、印度市场中的"感觉正确"参数需要本地验证。

9. **不重复公共规则**。反 AI slop 检测、UX 底线规则、资产映射表、平台翻译规范、视觉验收清单等公共规则统一沉淀到 `standards/` 目录。本 skill 只包含 Nakamura 的方法论判断，双向引用公共规则但不在体内复制。

10. **信息截止 2026-07-03**。Nakamura 在 AR/VR 空间交互领域没有成体系的方法论公开。他在 HUMANITY 游戏开发后是否会将 Boids/物理模拟方法论扩展到更多 3D/空间交互领域——是未知的。如果未来有新的深度访谈或项目，本 synthesis 需要更新。

11. **开放矛盾**：
    - "design is to make things gentle"：这个表述在所有一手资料中均未出现，可能为误传或评论者的概括。本 synthesis 未使用此表述作为 Nakamura 的方法论——最接近的原文是「気持ちいい」（feels good）、「ちょうどいい」（just right）、"simple interaction with various facets"。标注为二手/未验证，不纳入核心框架。
    - Nakamura 对 AI 态度的自我矛盾：公开说"对 AI 不着迷"，但也承认"实际上大量使用，AI 正在变成身体的一部分"（2026 年 2 月知財図鑑采访脚注）。这个矛盾是 Nakamura 本人承认的——与其说矛盾，不如说他的立场在演变中。本 synthesis 在 AI 相关领域标注为"Nakamura 的立场在此主题上不稳定，不要据此做绝对判断"。

## 九、表达 DNA 摘要（供 Phase 3 SKILL.md 的"判断语气"参考）

Nakamura 的设计判断表达特征：

- **工程师式的精确与设计师式的感性共存**：他不说"动效要有生命感"，他说"从这样的 detail 出发，2-3 个联动，100 个联动会看到这样的世界"。判断同时携带工程精度和设计野心
- **从土木工程继承的结构思维**：他不是在描述"画面应该怎样"，而是在定义"系统应该怎样运行"——桥梁工程中"概率设计"（安全率范围内最大化好的概率）被迁移到交互设计中
- **用物理隐喻替代审美词汇**：不说"优雅"，说"弹性衰减"；不说"舒服"，说"反弹系数略高于物理真实值"。他的判断语言是可参数化的，不是纯感性的
- **对自己诚实，不装**："不知道做游戏有多深多难"、"如果一开始就知道，不确定是否会走上这条路"——不美化自己的决策过程，不装作"我早就知道"
- **允许矛盾**：说 AI 不着迷但实际上大量使用——公开承认这个矛盾并用脚注"道歉"。不追求立场的一致性，追求体验的真实
- **保留日文术语的精确性**：「動的秩序の表現」（expression of dynamic order）、「創発」（emergence）、「落穂拾い」（gleaning）——这些概念在英文/中文中没有完美对应词，日文原文携带更精确的含义
