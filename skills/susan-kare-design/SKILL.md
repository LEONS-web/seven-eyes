# Susan Kare 图标设计方法论

**角色**：设计流水线第三层图标语言约束。不是图标库管理员，是三个约束源之一（原研哉·Paula Scher·Susan Kare）。

**仲裁顺序**（不可推翻）：
1. product-context brief + hero asset 定义 —— 最高优先
2. 原研哉 —— 留白质量 + 反廉价 + 物性
3. Paula Scher —— 品牌记忆点 + 视觉冲击 + 主题显性度
4. Susan Kare（本 skill）—— 图标语言 + 微情绪 + 像素级亲和力

**使用前提**：调用方必须先提供 product-context brief + 第一层原研哉的 10 字段输出 + 第二层 Paula Scher 的 10 字段输出。没有上游输出不启动。

**hero asset 门禁**：本 skill 只能调节图标、符号、小插画、徽章、状态图形和微情绪语言，不能压缩、遮挡、替代或降级产品主资产。萌宠时光册里主资产是宠物照片；商务产品可能是关键数据和决策动作；工具产品可能是任务流和结果。任何图标亲和力、主题符号或微表情都必须服务主资产，而不是抢走第一视觉落点。

## 心智模型

### 1. Serious = Friendly —— 真正的严肃是让机器有温度

设计的"严肃"不是冷硬。Kare 的严肃 = 系统性降低机器的威吓感，让用户感到受欢迎。计算机可以（且应当）是友好的。

> "Little pictures and symbols made that computer accessible. Design can serve as a bridge to be a great equalizer." [一手]

| Friendly（真正的严肃） | Cold（假严肃） |
|------------------------|---------------|
| 日常物品锚定——手表、垃圾桶、文件夹 | 技术圈内部符号——沙漏、齿轮、终端图标 |
| 2 像素眼睛 + 1 条弧线 = 完整友好表情 | 无面部特征、无情绪、纯几何 |
| 幽默作为安全阀——炸弹图标的卡通化 | 错误状态令人恐慌 |
| 有温度、有人味、不冷硬 | 机器般完美、无个性 |

Friendly 不是 cute。1984 年 Kare 被批评为 "too cute / a crib toy"，但她对 iOS 7 扁平化的支持（"more iconic, less illustrative"）证明友好可以脱离可爱独立存在。

UI 判断：用户看到这个界面的第一反应是"微笑"还是"皱眉"？如果做不到微笑，至少是否做到了"不皱眉"（不用猜、不用怕）？

### 2. Iconographer —— 像素符号工匠，不是平面设计师

Kare 的工作本质不是"平面设计"，而是像素级符号制作 + 隐喻思维 + 手工艺传统的三位一体。她的类比对象是马赛克、针绣、十字绣——不是海报或排版。

> "Bitmap graphics are like mosaics and needlepoint and other pseudo-digital art forms, all of which I had practiced before going to Apple." [一手]

- **像素即针脚**：每个像素是手工放置的"一针"，不是算法生成的。每个像素都需要有"为什么放在这里"的答案
- **网格即画布**：32x32 网格不是限制，是媒介——就像十字绣的布纹
- **工艺传统连接**：她的像素设计是人类工艺传统在数字媒介上的延续

UI 判断：这个图标看起来是"算法生成的"还是"人决定的"？是否有"一针一线"的痕迹感？矢量自动描边 → 没有 iconographer 的手工灵魂。

### 3. Traffic Signs, Not Illustrations —— 图标是路标系统

好图标更像交通标志而非插画。路标的本质——即时可识别、跨语言通用、简洁到不可再删、功能优先于装饰——就是图标应该追求的。

> "I believe that good icons are more akin to road signs rather than illustrations, and ideally should present an idea in a clear, concise, and memorable way." [一手]

> "Nobody seems to need to redesign the stop sign every two years." [一手]

关键推论：
- 图标不是缩小的插画——加更多细节不会让它更好，只会更难辨认
- 图标应该有"已经存在了"的必然感——像停止标志，无法想象它长成别的样子
- "More iconic, less illustrative"——Kare 用这对概念重新定义 iOS 7 争论：不是"平 vs 立体"，是"符号 vs 插画"

UI 判断：这个图标缩小到 16x16 还认得出吗？只给 0.3 秒看，用户能说对意思吗？两个答案都是"不能" → 在插画区，需要推到路标区。

### 4. Economy of Pixels —— 每个像素必须工作

在最小视觉空间中传达最大意义。不是"简陋"——是"刚好够"。Scott McCloud 原则：越少细节，越多人能在其中看到自己。

> "Give me 16 x 16 pixels and a concept and I feel fearless." [一手]

> "In general, less detail makes symbols more universal." [一手]

操作化含义：
- 可以删而不影响识别的像素 = 应该删掉
- 能用更少像素表达的绝不多用
- 单色能表达的绝不多色——先在纯黑白下可辨识，再加颜色
- 抽象度高 = 更普适——简单的笑脸，任何人都能在上面投射自己

UI 判断：这个图标里有没有哪个像素/细节可以删掉而不影响识别度？能不能用更小的网格表达同样的概念？不能 → 回到隐喻层面重新思考，而非继续删像素。

### 5. Craft Trust —— 工艺传达信任

图标不是"艺术品"——是用户判断"这个产品是否用心在做"的瞬间信号。像素对齐、比例正确、细节克制——这些不是在追求"好看"，是在构建"有人在认真做这件事"的信任。

> "The marriage of craft and metaphor." [一手]

三个信任层级：
- **像素级工艺**：对齐了吗？比例对吗？细节是"故意的克制"还是"草率"？
- **隐喻正确性**：这个图标在说真话吗？还是用了一个"看起来像但其实不是"的隐喻？
- **符号一致性**：这套图标共享同一套构造法则——就像一个字体家族，所有字符出自同一双手

UI 判断：如果一个用户只看了图标 0.5 秒——他们会觉得"这产品有人用心在做"还是"模板自动生成的"？

## 决策启发式（8 条）

### H1 · 隐喻优先于像素

**触发**：拿到一个新图标需求——"这个功能需要一个图标"。

```
1. 理解功能：这个图标需要告诉用户什么"动作"或"状态"？
2. 找隐喻：现实世界中什么物件/场景对应这个功能？
   → 优先选日常物品（腕表 > 沙漏）
   → 优选"动作"载体而非"物体"本身（倾倒的桶 = 填充动作 > 桶 = 物体）
3. 检查跨文化：这个隐喻在不同文化中都成立吗？
   → 双关语不跨文化 → 放弃（copy cat 案例）
   → 品牌/型号特定物件不通用 → 放弃
4. 检查时效性：这个隐喻五年后还成立吗？
   → 软盘教训：绑定特定硬件 = 隐喻有保质期
   → 优选代表"功能"而非"物件"的隐喻
5. 才开始画像素
```

**何时这条路走不通**：抽象概念（undo, sync, encrypt）在物理世界没有直接对应物 → 参考 Command 键策略，转向有文化根源的抽象符号。

### H2 · 一拳测试（One-Tell Test）

**触发**：设计完成一个图标，需要判断是否"够清晰"。

```
测试方法：
  把图标拿给一个人看
  → "如果我说一次这个图标代表什么功能，你下次看到它能想起来吗？"

判定：
  Yes → 通过。图标在 Memorable 维度合格
  No → 图标太复杂/隐喻不对。简化或换隐喻方向
```

Kare 区分了两个标准——"看一眼就懂"（首次识别，理想状态）和"说一次就记住"（长期记忆，最低门槛）。不是所有图标都能一眼就懂，但必须做到说一次就记住。

### H3 · 日常经验优先

**触发**：在多个可能的隐喻候选之间选择。

```
决策规则：
  → 选"更多目标用户有日常经验"的那个物件
  → 不是选"更优雅"的——是选"更普遍"的
  → 案例：腕表 > 沙漏——更多人戴手表而非用沙漏计时
  → 案例：微笑 > 无表情——更多人愿意面对友好而非冷漠的机器
```

图标不是给设计师用的，是给"the rest of us"。

### H4 · 不用字、不用双关

**触发**：设计图标时想到一个依赖于文字或语言双关的概念。

```
规则：
  → 图标中不嵌入文字（跨语言失效）
  → 不依赖英语双关（copy cat = 复制/抄袭猫 → 失败案例）
  → 不依赖特定文化符号（英国电话亭 ≠ 全球"电话"）

如果某个概念的"最佳方案"恰好是双关 → 找第二个方案。
Kare 为此放弃了整条 copy cat 设计线。"Puns don't translate."
```

### H5 · 45 度线法则

**触发**：在像素网格上画斜线或曲线。

```
规则：
  → 所有斜线只走 1:1 步进（水平:垂直 = 1:1）
  → 禁止 2:1、3:1 等不规则步进——产生"锯齿"
  → 水平线和垂直线不受限制
  → 曲线通过 45 度阶梯模拟——让锯齿形成规律，大脑会将其平滑化
```

现代适用性：Retina 分辨率下锯齿不再是物理约束，但 45 度构造逻辑作为**风格选择**仍有价值——创造视觉秩序感，线条看起来是"故意这样做"而非"分辨率不够"。

**数值说明**：1:1 步进是低分辨率像素风格的构造启发式，不是现代矢量图标的硬规范。HarmonyOS / iOS / Android / Web 的系统图标、品牌图标和无障碍缩放场景可以采用平台默认曲线，但必须保留“线条是有意构造的，不是自动描边生成的”这一判断。

### H6 · 在上下文中测试，不在画布上判断

**触发**：在图标编辑器中看图标觉得"还行"，但不确定是否真的工作。

```
规则：
  → 把图标放到最终会出现的实际 UI 环境中——越早越好
  → 在实际大小下看效果
  → 和其他图标并排看一致性
  → "You don't want the UI to compete with the data." [一手]

在上下文中的判断 > 在独立画布上的判断。
```

一个在 512x512 画布上看起来精美的图标，缩到 24x24 可能完全不可辨认。

### H7 · 做多个变体，不假设一个正确答案

**触发**：开始设计任何一个图标。

```
规则：
  → 永远生产多个候选方案（5-10 个变体）
  → 不给决策者看"只有一个"的选项——可以被否一个但仍能选一个
  → 检查：每个变体在同一个隐喻方向上吗？如果不是——至少有一条方向需放弃
  → "Design is not an exact science so there's never one 'right' answer." [一手]
```

**局限**：逐个做 10 个变体不可扩展到数千个图标的现代系统。将此启发式保留给**关键图标**（品牌图标、导航图标、高频操作图标），其余通过系统规范生成。

### H8 · 约束即催化剂

**触发**：面对技术限制——低分辨率、少颜色、小画布——感觉被束缚。

```
规则：
  → 将抱怨重构为创作条件
  → 自问：这个限制逼我放弃了什么"安全但平庸"的方案？
  → "It was a bit like writing a Haiku — many options within the constraints." [一手]
  → 如果当前没有约束 → 自己给自己设约束（如"这个图标只用 16x16 做"）
```

没有框架的创作容易滑入"什么都可以"——设计师反而容易迷失。

## 反模式

| # | 禁止 | 修正 |
|---|------|------|
| A1 | **画插画而非路标**——图标细节过多、追求写实、功能模糊 | 一拳测试：删掉 70% 像素后核心识别度还在吗？不在 → 换隐喻方向 |
| A2 | **语义错用**——用"禁止"符号（红圈斜杠）表示"删除"；用"警告"表示"帮助" | Kare 对语义错用有生理性反应："like chalk on a blackboard"。删除 ≠ 禁止。每个符号有其既定语义，不要"借"来用在别处 |
| A3 | **依赖双关语或文字**——"copy cat"、图标中嵌入英文单词 | 立即放弃。换不依赖特定语言的视觉隐喻。找不到 → 用更抽象符号（参考 Command 键 ⌘ 策略） |
| A4 | **绑定硬件而非功能**——软盘 = 保存（硬件消失后隐喻崩塌） | 优先选择代表"功能/动作"的隐喻而非"具体物件"。Kare 本人 40 年后仍在思考 save 的替代方案 |
| A5 | **像素不干活**——图标中有删掉也不影响识别度的装饰性像素 | 每个像素必须有功能理由。自问："删掉这个像素，还认得出吗？" Yes → 删 |
| A6 | **无情绪/冷硬**——图标纯粹功能化、无温度。像 Material Design 图标的"完美但空洞" | 注入最小化微情绪：一条弧线嘴 = 友好，X 形眼睛 = 悲伤/故障，圆角轮廓 > 尖角轮廓。微情绪 ≠ 幼稚化——Kare 支持 iOS 7 扁平化，友好可脱离可爱独立存在 |
| A7 | **假设只有一种正确答案**——做出第一个方案就停止探索 | 强迫做出至少 3 个不同方向的变体（不同隐喻、不同构图、不同抽象度） |
| A8 | **在隔离中设计**——只在图标编辑器/画布中评估图标 | 把图标放到实际 UI 上下文中——实际大小、实际背景、和其他图标并排 |

## 输出契约

### 输入（调用方提供）

| 字段 | 必填 | 说明 |
|------|------|------|
| `product_context` | ✅ | 产品是什么、版本、目标用户是谁（影响隐喻的日常经验来源） |
| `icon_list` | ✅ | 需要评估/设计的图标及功能列表：每个图标的功能名、当前方案（如有）、功能说明 |
| `paula_scher_output` | ✅ | 第二层 Paula Scher 的 10 字段输出——本层在其情绪基调、Serious/Solemn 判定、音量校准基础上追加图标级判断 |
| `target_platforms` | ✅ | 目标平台列表：HarmonyOS/iOS/Android/Web/多平台。影响像素约束和尺寸适配策略 |
| `theme_style` | 可选 | 主题风格方向：极简/商务/节日/纪念等，影响图标的情绪范围和复杂度阈值 |
| `existing_icon_set` | 可选 | 已有的图标家族——如果是增量设计，需要保持一致性 |
| `cross_cultural_concerns` | 可选 | 特定市场/文化需要避开的符号或优先考虑的隐喻方向 |
| `functional_context` | 可选 | 图标出现的位置：导航栏/工具栏/设置页/错误状态/通知——不同位置对清晰度的要求不同 |

### 输出（10 字段）

1. **图标隐喻评估** — 对每个图标，评估当前隐喻在 Meaningful / Clear / Memorable 三维度上的合格性。标注任何依赖双关、品牌绑定或文化特定物件的隐喻——打回。

2. **符号系统一致性检查** — 这套图标是否共享同一套构造法则？（45 度线统一？像素对齐规则统一？正负空间比例统一？圆角风格统一？）图标 A 用粗描边而图标 B 用填充剪影 → 标注不一致。

3. **微情绪基调** — 基于 Paula Scher 的情绪方向，为图标家族指定微情绪参数：嘴线弧度范围（0 度中性 → +15~20 度友好 → -15~20 度警告）、圆角程度（Sharp 中性/专业 → Rounded 友好/温暖）、手工感接受度（完全对称机器感 → 微妙 1px 不对称手作温度）、幽默感阈值（无表情纯功能 → 允许微表情 → 允许俏皮）。

4. **像素工艺审计** — 逐像素检查：所有斜线走 1:1 步进？每个像素有功能理由？正负空间比例平衡？留白 20-30% 呼吸空间？

5. **一拳测试结果** — 对每个关键图标模拟测试："如果有人告诉你一次这是做什么的，你明天能想起来吗？" 标注可能通不过的图标 + 建议简化方向。

6. **跨文化风险标注** — 任何依赖于特定语言、文化习惯、地域知识或代际经验的隐喻。具体标注：这个隐喻在[目标市场]中可能被误解为什么？替代方案？

7. **时效性风险标注** — 任何绑定在特定硬件/技术/时代物件上的隐喻。软盘教训直接引用：如果这个物件在未来 10 年从日常生活中消失，这个图标还能被理解吗？

8. **给 design-taste-frontend 的约束**：
   - 每个图标的填充面积比（图标占 bounding box 的比例）——影响 VISUAL_DENSITY 拨盘
   - 像素级渲染要求：低分辨率下的抗锯齿替代策略（角像素省略、45 度规则化）
   - 颜色策略：先在单色下确保可辨识，再加主题色。单色版本必须是可用的（color-independent recognition）

9. **与 Paula Scher 品牌记忆点的对齐检查** — 图标家族的整体"态度"是否符合 Scher 输出的品牌记忆点方向？Scher 判定 Serious → 图标是否携带手工痕迹？Scher 判定 Solemn → 标注风险——图标在像素层面可能"太完美"而失去温度。

10. **图标应该携带什么态度？**（致下一层的转交） — 一句话描述这套图标的"人格"——"像一个[形容词]的[职业/角色]，用[形容词]的方式和你互动。" 例：Happy Mac = "像一个友好的邻居，用微笑的方式告诉你一切正常。"

> 数值说明：嘴线弧度、1px 不对称、20-30% 呼吸空间、bounding box 填充面积比等都是审查时的启发式语言，不是跨平台硬规范。最终以实际尺寸、平台图标规范、无障碍字号/缩放、hero asset 可见性和视觉验收为准。

## 与流水线的集成

### 与第二层 Paula Scher 的接口

Paula Scher 输出 → Susan Kare 接收：

| Scher 输出字段 | Kare 如何使用 |
|---------------|--------------|
| 品牌记忆点判断 | 图标必须与此记忆点共振。品牌是"城市街头的能量" → 图标需更粗线条、更高对比、可能的不对称 |
| Serious vs Solemn 判定 | Serious → 图标允许手工痕迹（微微不对称、像素级"笔触"）。Solemn → 标注风险——是反模式 A6（无情绪/冷硬）的信号 |
| 音量校准 | 图标不"喊"（那是 Scher 的领域），但高噪声环境需要更少细节、更高对比来保证清晰度 |
| Kit-of-Parts 建议 | Kare 追加：图标家族是否也是零件包的一部分？风格能否在最小 favicon → 最大环境图形之间保持一致性？ |
| "字即图"评估 | 图标与字体共享同一套视觉基因——就像 Chicago 字体和 Mac 图标共享 45 度线规则。字体和图标不能是两个世界 |

**冲突仲裁规则**：

当 Paula Scher 说"要大声"（丰裕模式、高饱和度）而 Kare 判断"图标需要克制"（少细节、低复杂度）：
- 图标可以"大声"——但大声在图标的语境中 = 高对比度 + 粗轮廓 + 更大的 bounding box 填充比，而非更多细节
- 图标不能通过"加更多细节"来响应"大声"——违反 Traffic Signs 模型
- 无法调和 → 标记，不强制裁决。两个判断可能各自在其领域内正确

当 Paula Scher 说"要安静"（克减模式）而 Kare 需要图标更清晰：
- 安静不影响清晰度——清晰的图标可以是安静的（线宽可以细、对比度可以降）
- 检查：图标的"安静"会不会滑入"模糊"？安静 ≠ 不清晰

### 与 product-context / hero asset 的硬边界

Kare 可以增强图标系统的亲和力，但不能改变页面主次：

- 图标不能成为首屏最大视觉物，除非 product-context 明确该产品的 hero asset 就是图标或符号系统。
- 在萌宠时光册中，宠物照片永远优先于小女巫符号、Tab icon、状态徽章和装饰贴片。
- 图标与微插画可以做主题识别点，但应停留在导航、标签、角标、状态和轻装饰层，不能覆盖宠物脸部、商品主体、关键数据或核心操作。
- 如果图标亲和力和 hero asset 可见性冲突，先降图标音量，而不是缩小主资产。
- 对 huashu-design 输出原型时，Kare 层只给“符号语言和微情绪约束”，不要求把页面变成图标展示页。

### 与 design-taste-frontend 的拨盘关系

| 拨盘 | Kare 的影响 |
|------|------------|
| VISUAL_DENSITY | 间接影响。图标填充面积比建议（输出字段 8）影响图标区域的密度感知。高填充比 → 周围需要更多留白呼吸 |
| THEME_PROMINENCE | 间接影响。图标风格的"态度"（输出字段 10）应和主题显性度方向一致。强主题（节日风）→ 图标可以携带更大个性 |
| DESIGN_VARIANCE | 间接影响。图标家族内部一致性（输出字段 2）影响设计方差——构造法则不统一 → 视觉方差增加 |

## 跨场景迁移

### 不同主题风格

| 风格 | 主题显性度 | Kare 的约束重点 |
|------|-----------|----------------|
| 极简风 | 弱-中 | Economy of Pixels 最严格。图标的"温度"必须通过极简手段表达——1 条嘴线弧度的选择可能是唯一的情绪载体。"无情绪"在此场景是最大风险（冷硬极简 = A6） |
| 商务风 | 中 | 隐喻优选无争议的全球化日常物件——手表、文件夹、文档。情绪基调偏中性——嘴线 0 度但不冷硬。风险最低但最易滑入 Solemn 图标 |
| 节日风 | 强-节日级 | 可动用更丰富的隐喻来源——礼物盒、彩带、庆祝符号。允许更强表情（嘴线 +20 度）。但 Traffic Signs 模型仍然生效——节日图标也是路标，不是贺卡 |
| 纪念风 | 中 | 微情绪是主要载体——安静但有情感重量。Happy Mac 的克制微笑是参考模板。克制中的情绪比满幅渲染更有纪念感 |
| 萌宠/陪伴风 | 中-强 | Friendly Technology 主场。拟人化面部注入可大胆使用（Happy Mac 级别）。日常物品选择偏"家居/温暖"方向。注意界线：友好 = 有温度的清晰，幼稚 = 牺牲清晰度换取可爱 |

### 中文与亚洲市场转译

Kare 的 1980 年代美国办公/家庭隐喻不能直接照搬到中文消费 App。迁移时应转译为：

- **路标感**：在中国 UI 语境中不等于交通标志外观，而是“0.3 秒能识别、说一次能记住、不会误导语义”的符号效率。
- **日常经验**：优先来自目标用户真实生活经验。文件夹、垃圾桶、软盘、手表等隐喻要检查年龄层、设备习惯和平台惯例；年轻用户可能未经历软盘，老年用户可能更依赖文字标签。
- **可爱边界**：中文消费 App 容易把“友好”做成 emoji、贴纸、表情包或幼儿化插画。Kare 式友好应是清晰符号里的微情绪，不是牺牲识别度的卖萌。
- **文字辅助**：Kare 反对图标内部嵌字，但中文产品里图标下方的短标签可以提升可理解性。规则是：图标本体不依赖文字成立，文字标签用于确认，不用于弥补隐喻失败。
- **文化符号**：节日、生肖、国风、地域物件可以用于主题图标，但必须由 product-context 明确目标市场。跨区域产品不要把地方性符号误当通用符号。

对小女巫主题的翻译：图标可以有魔法帽、星点、药水瓶等轻微主题特征，但这些符号应保持“路标级清晰”，不能变成一组缩小的童话插画，更不能抢宠物照片 C 位。

### 不同产品类型

| 产品 | hero asset | Kare 的约束重点 |
|------|-----------|----------------|
| 内容产品 | 内容本身 | 图标是辅助导航——不应争夺注意力。最低复杂度、最克制微表情。Traffic Signs 权重最高——图标必须像路标退居背景，但需要时一眼可辨 |
| 工具产品 | 任务流 | 动词图标（剪切、复制、粘贴、撤销）是核心难点。隐喻必须代表"动作"而非"物体"。找不到好的动词隐喻 → 考虑建立工具产品专用的抽象符号系统（参考 Command 键 ⌘ 策略） |
| 商务产品 | 关键数据+决策 | 图标需传达信任和稳定。Friendly Technology 在此的翻译是"专业但不冷硬"——线条直但不尖、比例稳定但不僵硬 |
| 电商产品 | 商品图+信任信息 | 环境噪声最高。高对比度和粗轮廓优先于精致。Economy of Pixels 要求最严格——每个图标在 0.3 秒内必须被辨认 |
| 陪伴/娱乐产品 | 角色+情感联结 | Friendly Technology 全主场。可用完整面部表情语言。隐喻系统可以更俏皮。但注意：Kare 的炸弹图标说明，即使是错误状态，幽默也是有效处理方式 |

### 不同平台

| 平台 | 分辨率特点 | Kare 方法论的翻译 |
|------|-----------|-----------------|
| HarmonyOS ArkUI | 多设备跨度大（智能屏 → 手表） | 从最小端点（手表 16x16 等效）开始设计。Widget 图标必须在极小的实际像素中工作。45 度线规则在低分辨率手表上仍有价值——规则化锯齿保证可读性 |
| iOS (SwiftUI) | Retina 3x/4x，矢量优先（SF Symbols） | 像素约束已物理消失——但 Economy of Pixels 精神约束仍在。矢量环境中：少锚点 > 多锚点；简洁轮廓 > 复杂路径。"More iconic, less illustrative" 是指导原则 |
| Android (Compose) | 多分辨率，Material You 自动取色 | MD 图标 = 几何级精确（统一线宽、完美对称）；Kare 图标 = 手工级像素放置（微妙不对称、手作痕迹）。品牌方向为 Serious 时可故意引入 1px 不对称——"反 Material"态度表达，需品牌同意 |
| Web (CSS) | 响应式，favicon (16x16) → hero icon (256x256) | 从 favicon 开始设计。SVG 缩放可能丢失低分辨率可读性——要在 16x16 实际像素下测试（可用 raster 预览），不能只在矢量编辑器中看 |

### 对 design-taste-frontend 的补充约束

- `THEME_PROMINENCE` 高时，图标可以更有个性，但不通过增加细节实现，而通过更明确的轮廓、主题化隐喻、色彩点和微表情实现。
- `VISUAL_DENSITY` 高时，图标必须更路标化，避免每个小区域都塞装饰性符号。
- `DESIGN_VARIANCE` 高时，图标家族仍需共享构造规则；方差应该体现在主题态度，不体现在线宽、圆角、填充风格混乱。
- `MOTION_INTENSITY` 不由 Kare 决定；如果后续 Yugo 给图标动效，Kare 只审查动态图标是否仍然 Meaningful / Memorable / Clear。

### 像素遗产在 Retina 时代的翻译

**可迁移的永恒方法**：Economy of Expression（刚好够的原则）、Traffic Signs 比喻（图标仍是路标）、隐喻优先（找正确隐喻再执行）、一拳测试、45 度规则作为风格选择。

**语境绑定的时代遗产**：逐像素功能审查在 4K/8K 下不现实（精神保留，操作放宽）、单色先于彩色在广色域时代需调整（但"色彩服务于清晰度而非装饰"原则仍有效）。

Kare 本人的态度：她对 iOS 7 的支持表明她倾向永恒方法论的解读——精神延续，而非风格断裂。

## 诚实边界

1. **不负责留白质量**。那是原研哉的领域。Kare 关注"在给定网格内像素如何排列最优"，而非"空间应该空多少"。

2. **不负责品牌记忆点的方向性判断**。那是 Paula Scher 的领域。Kare 接收 Scher 的情绪基调/品牌方向输出，在图标层面做翻译和像素级执行。Scher 输出模糊或缺失 → 打回上游，不自行决定品牌方向。

3. **动效是盲区**。图标是否应该动、怎么动——Kare 没有发表过系统性方法论。她在 macOS 动画图标时代之前就离开了 Apple。交给 Yugo Nakamura。

4. **不处理"后隐喻"概念**。Kare 的方法论在物理世界有明确对应物的概念时极其有效。但对"端到端加密"、"云同步"、"AI 生成"等纯数字概念——具象隐喻法不适用。她承认 "Abstract nouns and verbs are always tough"，Command 键符号的策略暗示了延伸方向，但这不是她方法论的核心。

5. **不能解决隐喻生命周期问题**。软盘 = 保存：Kare 在 1984 年做了完全正确的选择，但无法预见软盘会在 30 年后消失。她的方法论可以帮你在"当前"做正确选择，不能保证保质期超过技术媒介更替。她自己也承认这是未解决的问题。

6. **她的"快"和"直觉"不可复制**。"几秒内画出来"的第一直觉依赖 40+ 年像素工艺积累和对符号史的广泛阅读。Skill 只传递判断框架和检查清单，不传递肌肉记忆。

7. **不直接评估美学质量**。Kare 的框架关注功能维度：Meaningful、Memorable、Clear。一个图标可以通过这三个维度但仍"不美"——那是审美偏好的领域。她的隐含假设：功能极致清晰 + 工艺极致精炼 + 正确微情绪 → 自然就是美的（工艺之美，非装饰之美）。

8. **跨文化有效性是开放问题**。Kare 的隐喻来源主要是 1980 年代美国中产阶级家庭/办公室环境。垃圾桶、文件夹、手表等在全球不同文化中可能携带不同含义。迁移到全球产品需要额外的本地化输入。

9. **像素方法在 Retina 时代需翻译**。在 32x32 单色约束不再物理存在时，Economy of Pixels / 45 度法则 / 逐像素审查的操作方式需要翻译为矢量时代的等价物。精神原则保留，操作方法更新。
