## 1. 经典 Before/After 案例

### 1.1 导航栏（Navbar）：假想的航班预订应用

**来源**：Steve Schoger, CSS Day 2019 主题演讲 & Gist 笔记（ynotdraw）

**原始问题**：
导航栏被放在一个深色的独立横条（bar）里，形成了一个厚重的视觉区域，与下方的内容割裂。导航元素挤在这个窄条中，缺乏呼吸空间，文字和背景的对比度不一致。

**改动决策**：
- **关键认知**："Navbars don't need to be in a bar."（导航栏不一定非得是个"条"）
- 移除独立的导航条背景色，将导航元素直接放在页面背景上
- 使用更轻量的文字样式替代深色背景上的浅色文字
- 导航链接之间用间距区分，而非边框

**为什么有效**：
移除"条"之后，导航变成了页面自然的一部分而非一个"插件"。页面顶部的视觉重量大幅减轻，用户注意力不再被厚重的顶部区域分散。这种处理方式同时让页面看起来更现代、更"设计过"。

**可提取规则**：
- 不要默认给导航加背景色——先考虑不加背景是否可行
- 导航可以用间距和字重来建立结构，不需要物理"容器"
- 考虑使用页面本身的背景作为导航的背景

### 1.2 卡片（Cards）：度假套餐卡片改造

**来源**：CSS Day 2019 演讲 & 1c7.me 中文笔记

**原始问题**：
- 所有卡片高度不一致（因为图片纵横比不同）
- 每张卡片内有"Price:"和"Website:"标签——冗余信息
- 卡片使用大量边框来分隔内部区域
- 价格信息（最重要的决策依据）和次要信息视觉权重相同

**改动决策**：
1. **统一图片容器高度**：使用 `background-size: cover` + 固定高度容器，裁切超出部分，使所有卡片在同一行高度一致
2. **删除冗余标签**："Price" 和 "Website" 标签本身是多余的——用户看到数字知道是价格，看到网址知道是网站
3. **去边框**：移除卡片内部的分隔线，改用浅色背景区分不同区域
4. **建立信息层级**：价格用更大、更粗的字体突出显示；次要信息用更小的字号和更柔和的颜色
5. **提取关键信息**：将最重要的差异化信息（如评分、特色标签）从正文中提取为 badge

**为什么有效**：
裁剪图片后卡片获得了一致的视觉节奏，用户可以平行比较不同套餐。删除标签减少了视觉噪音，让眼睛直接聚焦在数据本身。去掉边框后，卡片内部通过间距和背景色自然分组，看起来更干净。

**可提取规则**：
- 数据格式本身就能传达含义时，不需要加标签（如价格、网址、日期）
- 统一图片容器比保持原始纵横比更重要（在卡片网格场景中）
- 用 `background-size: cover` + `center` 裁切图片是最简单的统一方案
- 重要信息抽成 badge，让用户扫一眼就能比较

### 1.3 表格（Table）：航班搜索结果表

**来源**：CSS Day 2019 & Steve Schoger Twitter："Designing nice tables can be tough"

**原始问题**：
- 表格布满竖线和横线，看起来"非常忙碌"（very busy）
- 所有数据视觉权重一致，用户无法快速区分主次信息
- 表头样式生硬，像 Excel 表格而非现代 UI
- 数字对齐不一致（使用了 proportional numbers 而非 tabular numbers）

**改动决策**：
1. **去除所有竖线**，横线仅保留在行之间，且使用极浅的灰色
2. **斑马纹替代边框**：使用交替的浅色背景行（zebra striping），让眼睛在扫描长行时不会"串行"
3. **表头改造**：表头文字用小字、大写、加粗、柔和颜色，使其更像"标签"而非"标题"
4. **应用 tabular numbers**：`font-feature-settings: "tnum"` 使数字等宽对齐，便于用户纵向比较数值
5. **次要信息降级**：将非核心列（如航班号）用更小的字号和更淡的颜色显示
6. **按钮降级**：表格内的操作按钮从蓝色主色改为灰色，避免与主 CTA 竞争注意力

**为什么有效**：
去除边框后表格从"数据牢笼"变成"干净的信息列表"。斑马纹在不引入新 UI 元素的情况下解决了长行扫描问题。tabular numbers 是 Steve 特别强调的一个"小细节大效果"技巧——不需要等宽字体就能让数字完美对齐。

**可提取规则**：
- 表格首选去边框：斑马纹 > 横线 > 竖线
- `font-feature-settings: "tnum"` 是表格设计的秘密武器
- 表头应该是"辅助性标签"而非"强调性标题"
- 表格内的操作按钮应降级为次要样式

### 1.4 表单（Form）：WP Pusher 结账页 & 航班搜索表单

**来源**：Refactoring UI YouTube（WP Pusher Checkout）& CSS Day 2019

**原始问题（WP Pusher 结账页）**：
- 表单字段挤在一起，缺乏呼吸空间
- 所有字段使用相同的视觉处理，用户难以区分必填与选填
- 标签和输入框之间的间距关系模糊

**改动决策（WP Pusher）**：
1. 将表单整体放入白色面板（panel）中，从页面背景中"抬起"
2. 大幅增加输入框的内边距和字段间距
3. 输入框高度统一为 40-48px（字体 + padding）
4. 给输入框加上 subtle 的浅灰背景色（而非仅靠边框区分）
5. 在输入框内添加图标（调低颜色透明度以免抢眼）

**原始问题（CSS Day 航班搜索）**：
- 顶部搜索表单的 Depart/Return 文字难以看清
- 单选按钮和复选框使用浏览器默认样式
- "Add a hotel" 行与上面的搜索区域难以区分

**改动决策（CSS Day）**：
1. 在有颜色的背景上，先加一个暗色/白色 overlay 来保证文字可读
2. 输入框从两行合并为一行，减少垂直空间占用
3. 替换浏览器默认的 radio/checkbox 为自定义样式
4. "Add a hotel" 行给一个独立的浅色背景以区分
5. 按钮从蓝色改为更协调的颜色，添加图标增强可识别性

**为什么有效**：
表单是用户完成任务的核心路径，任何视觉摩擦力都会降低转化率。增加空间后用户不会感觉"被催促填完"，反而更愿意仔细填写。输入框加浅色背景（而非仅边框）让它们在与白色背景区分时更明显。

**可提取规则**：
- 表单输入框总高度至少 40-48px
- 输入框除了边框外，可以用 subtle 的背景色增加可辨识度
- 表单内的图标颜色要降低透明度，不要抢文字的注意力
- 相关字段组可以用浅色背景"团"在一起

### 1.5 仪表盘/数据展示：Resolute 属性管理页

**来源**：Refactoring UI YouTube（Redesigning Resolute's Properties Page）

**原始问题**：
- 属性列表页信息密度高但无层次
- 大量"字段: 值"格式的数据平铺，缺乏扫描路径
- 搜索/筛选区域与结果区域边界不清

**改动决策**：
1. **Think outside the database**（跳出数据库思维）：不把数据按 "Label: Value" 格式直接映射到 UI
2. 将标签转化为值的修饰——例如 "In stock: 12" → "12 left in stock"
3. 如果用户会寻找标签，就强调标签；如果用户会寻找数据，就强调数据
4. 使用 accent border（彩色顶部边框）为属性卡片增加视觉趣味
5. 搜索筛选区与结果区通过不同的背景色区分

**为什么有效**：
"Think outside the database" 是 Refactoring UI 最核心的哲学之一。数据库存储的是 "label → value"，但 UI 不需要照搬这个结构。当标签被内联到值的表述中时，用户读取更快、理解更自然。

**可提取规则**：
- 不要直接把数据库字段映射到 UI："Label: Value" 模式通常可以优化
- 问自己：用户是来找标签还是来找值的？据此决定视觉权重
- 用 accent border 为信息密集的卡片注入色彩和层次

### 1.6 设置页：Bad About 话题页改造

**来源**：Refactoring UI YouTube（Redesigning BadAbout）& LOGiCFACE 评论

**原始问题**：
- 内容区域拥挤、沉重（cramped and clunky）
- 缺乏清晰的视觉层级
- 排版单调，所有内容"喊"得一样响

**改动决策**：
1. 大幅增加留白——从 "刚好不丑" 改为 "先加太多再减"
2. 建立明确的内容层级：主标题 → 副标题 → 正文 → 辅助信息
3. 用颜色和字重而非字号来区分层级
4. 将相关的内容块用间距分组，无关的用更大的间距分开
5. 侧边栏与主内容通过不同的背景色（或浅色背景）区分

**为什么有效**：
Bad About 是一个内容型页面，用户的任务是阅读。原始设计把所有东西塞在一起，读者无从下手。改造后的设计通过间距引导读者的眼球，让他们本能地知道先看哪里、后看哪里。

**可提取规则**：
- 内容型页面最需要的就是"空间"——不要怕留白
- 间距应该引导阅读路径：相关元素靠近，不相关的远离
- 同页面内最多用 2-3 种字号的差异来建立层级

## 2. 间距改造案例

### 2.1 核心案例：从"加 margin 加到刚好"到"先加太多再减"

**来源**：Refactoring UI 书籍（"Start with too much white space"）& CSS Day 2019

**原始问题**：
开发者做 UI 时最常见的间距错误是：从紧凑开始，一点一点加 margin/padding，直到"看起来不丑了"就停手。这种方法的结果是每个元素都只有"刚好够"的空间——界面看起来拥挤、廉价、业余。

> "The common approach to introducing space is to add just a bit at a time, and if it still feels too tight, you continue adding a bit more. The problem is you're only adding the minimum amount of space until the site no longer looks bad." — Steve Schoger

**改动决策**：
反向操作——从"明显太多"的留白开始，然后逐步减少，直到觉得"刚好"。

具体做法：
1. 选择一个组件（如卡片），把所有内边距和外边距翻倍
2. 在页面上下文中审视这个组件——虽然单独看"太空了"，但在完整页面中往往正好
3. 如果确实太多，逐渐减少直到满意

**为什么有效**：
心理锚定效应。当你从"挤"开始，任何一点空间都是进步，你会在"还行"时就停下来。当你从"空"开始，任何减少都是审慎的牺牲，你会在"真的够了"时才停。最终结果的空间感差异巨大。

**可提取规则**：
- 永远从"太多"的留白开始，删除空间而非添加空间
- 不要孤立判断一个组件的间距——放在完整页面上下文中判断
- 大元素（如容器面板）需要比小元素（如按钮）更多的周围空间
- 输入框至少 40-48px 总高度（字号 + 上下 padding）

### 2.2 间距系统：从"凭感觉选值"到"预定义 scale"

**来源**：Refactoring UI 书籍 & mohitkhare.me 笔记 & ajnisbet.com 笔记

**原始问题**：
没有间距系统时，开发者凭感觉选 margin/padding 值——12px, 13px, 15px, 17px……每个值都是即兴决定，导致整个 UI 缺乏节奏感。两个相邻的值（如 14px 和 16px）差异太小，用户感知不到，但开发者又纠结于用哪个。

**改动决策**：
建立预定义的间距 scale，只用 scale 中的值：

```
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 20px;   // 注意：不是 20 或 24，详见规则
--space-6: 24px;
--space-8: 32px;
--space-10: 40px;
--space-12: 48px;
--space-16: 64px;
--space-20: 80px;
```

**关键规则**：
1. **相邻值差距 ≥25%**：12px 和 16px 差 33%（可感知），16px 和 20px 差 25%（刚好），但 46px 和 48px 差 4%（不可感知，不要同时存在于 scale 中）
2. 小端密集、大端稀疏——这是感知均匀而非数学均匀
3. 偶数像素值，避免亚像素渲染问题
4. 用 4 的倍数作为基础（4, 8, 12, 16, 20, 24, 32...）
5. 一旦定义，绝不在 scale 之外取值

**为什么有效**：
约束消除决策疲劳。当只有 8-10 个间距选项时，选择瞬间完成。间距 scale 在整个应用中创造了一致的"视觉节奏"——用户可能不会意识到，但会感觉到界面"很整齐"。

**可提取规则**：
- 预定义间距 scale（8-12 个值），设计时只从中选择
- 相邻值差距至少 25%
- 小值密集（4, 8, 12, 16），大值稀疏（32, 48, 64, 80）
- "如果 16px 太挤、24px 太空，也不要发明 20px（除非它在你的 scale 中）"

### 2.3 间距与分组："避免模糊间距"案例

**来源**：Refactoring UI 书籍（Avoid ambiguous spacing）

**原始问题**：
一个表单中，字段 A 的标签和字段 A 的输入框之间的间距，与字段 A 的输入框和字段 B 的标签之间的间距，几乎一样大。用户难以判断标签属于上面的输入框还是下面的输入框。

**改动决策**：
- 组内间距（标签到输入框）< 组间间距（上一个输入框到下一个标签）
- 明确的比例关系：组内:组间 ≈ 1:2 或 1:3
- 永远确保组周围的间距大于组内的间距

**为什么有效**：
利用了格式塔心理学的"邻近性原则"——距离相近的元素被感知为同一组。当组内间距和组间间距模糊时，分组关系也模糊。

**可提取规则**：
> "Whenever you're relying on spacing to connect a group of elements, always make sure there's more space around the group than there is within it."

- 标签应该更靠近自己的输入框，而不是上一个输入框
- 用间距比例来表达分组：组内 < 组间 < 区块间
- 如果不确定，加大组间间距

## 3. 色彩改造案例

### 3.1 灰度系统：从 #000→#FFF 到 8-10 档科学灰度

**来源**：Refactoring UI 书籍预览（"Building Your Color Palette"）& mohitkhare.me

**原始问题**：
开发者只用 3-4 个灰度值（纯白 #FFF、浅灰 #CCC、深灰 #666、纯黑 #000），很快就会遇到"比 #CCC 深一点但比 #666 浅一点"的需求而无值可用。于是开始即兴创造——`#DDD`, `#BBB`, `#999`——最终整个应用有 35 个略微不同的灰色。

**改动决策**：
提前定义 8-10 档灰度，从 near-black 到 near-white，感知均匀分布：

```
--gray-50:  #f9fafb;   // 微妙背景
--gray-100: #f3f4f6;   // 浅背景
--gray-200: #e5e7eb;   // 边框/分割线
--gray-300: #d1d5db;   // 禁用态
--gray-400: #9ca3af;   // 占位文字
--gray-500: #6b7280;   // 次要文字
--gray-600: #4b5563;   // 正文
--gray-700: #374151;   // 强调文字
--gray-800: #1f2937;   // 标题
--gray-900: #111827;   // 最深文字（非纯黑）
```

**关键决策原则**：
1. 从深灰（非纯黑）开始，以稳定步长递增到 white
2. 最深色用于文字而非背景；最浅色用于背景而非文字
3. 9 档是 sweet spot（容易对半分：900-700-500-300-100，再填充 800-600-400-200）

**为什么有效**：
预定义的灰度 scale 从根源上杜绝了"即兴造灰色"的冲动。8-10 档覆盖了从文字到背景到边框到禁用态的所有需求。纯黑（#000）被排除在外因为它看起来不自然——自然界中几乎不存在纯黑。

**可提取规则**：
- 灰度需要 8-10 档，不要少
- 纯黑 (#000) 不自然，用 #111827 这样的 near-black 替代
- 定义 shade 用 900-100 编号体系，便于通信（"这个用 gray-700"）
- 不要用 CSS `lighten()`/`darken()` 动态生成——会产生 35 个几乎一样的蓝色

---

### 3.2 "灰色不一定得是灰色"：从纯灰到有色灰

**来源**：CSS Day 2019 & Steve Schoger Twitter："Greys don't have to be grey"

**原始问题**：
使用 0% 饱和度的纯灰色（#999, #CCC 等）做背景和文字，界面看起来"死气沉沉"（dull and unnatural）。

**改动决策**：
在灰色中注入微量的蓝色（冷调）或棕色/黄色（暖调）：

- 冷调灰：HSL 中给灰色加 2-5% 的蓝色饱和度
- 暖调灰：HSL 中给灰色加 2-5% 的橙色/棕色饱和度
- 主色为蓝色时倾向冷调灰；主色为红色/紫色时倾向暖调灰
- 在 HSL 中，浅色和深色两端要增加饱和度以维持色温一致性

**为什么有效**：
现实世界中不存在纯灰色——每一种"灰色"都带有微妙的色温偏向。在 UI 中使用有色灰让界面感觉更温暖、更自然、更有质感。Basecamp 是使用暖调灰的经典案例。

**可提取规则**：
- 纯灰（S=0%）看起来不自然，注入微量色相
- 冷调灰（+蓝）适合科技/专业产品
- 暖调灰（+棕/黄）适合人文/生活产品
- 在 HSL 中增加浅色和深色端的饱和度，否则会褪色

---

### 3.3 "不要在有色背景上用灰色文字"

**来源**：7 Practical Tips & CSS Day 2019

**原始问题**：
在蓝色（或其他颜色）背景上，将次要文字设为灰色（如 `#999`）——文字看起来"褪色、不协调"（washed out, clashes with background）。

**改动决策**：
两种方案：
1. **降低白色文字的不透明度**（如 `rgba(255,255,255,0.7)`）——让背景色透过文字
2. **手选一个与背景同色相的颜色**——在 HSL 中保持相同 Hue，调整 Saturation 和 Lightness 直到满意

**为什么有效**：
"让文字更接近背景色才是真正的降低对比度，而不是把它变灰。"灰色在白色背景上看起来是"降低了对比度"，但在有色背景上灰色会与背景色产生色相冲突。

**可提取规则**：
- "Making the text closer to the background color is what actually helps create hierarchy, not making it light grey."
- 有色背景上不要用灰色文字：要么降白色不透明度，要么选同色相浅色
- 这个方法同样适用于图片/图案背景

---

### 3.4 "先灰度设计"策略

**来源**：Refactoring UI 书籍 & Full Stack Radio #74

**原始问题**：
开发者在设计早期就引入颜色，导致配色决策和布局决策混在一起，互相干扰。配色没定下来之前布局怎么看都不对，布局没定下来之前配色也选不好。

**改动决策**：
"Design in grayscale first"——先用灰度完成整个界面的布局、间距、层级、排版。确认一切在没有颜色的情况下都能正常工作后，再逐步引入颜色（主色、强调色）。

**为什么有效**：
灰度设计强迫你用间距、字重、大小来建立层级——如果灰度版本就已经层级清晰，加上颜色后只会更好。反过来，如果层级依赖颜色，一去掉颜色就崩塌。

**可提取规则**：
- 在所有颜色决策之前，先完成灰度版本
- 灰度版本的层级应该已经足够清晰
- 颜色是"锦上添花"而非"雪中送炭"
- 灰度设计也帮你更早发现间距和布局问题

---

## 4. 字体层级改造案例

### 4.1 核心案例："字重比字号更重要"

**来源**：7 Practical Tips & Refactoring UI 书籍（"Size isn't everything"）

**原始问题**：
开发者建立文字层级时只用一个维度——字号。"这是重要的？放大。""这是次要的？缩小。"结果：主标题巨大（如 36px），辅助信息小到看不清（如 11px）。中间没有过渡，层级生硬。

**改动决策**：
使用三个维度建立层级：字号（Size）、字重（Weight）、颜色（Color）。

实战组合：
- **主标题**：24px | 700 字重 | near-black (#111827)
- **副标题**：18px | 600 字重 | dark gray (#374151)
- **正文**：16px | 400 字重 | body gray (#4B5563)
- **辅助文字**：14px | 400 字重 | light gray (#6B7280)
- **标签/说明**：12px | 500 字重（或 uppercase）| light gray (#9CA3AF)

**为什么有效**：
当字重和颜色参与层级构建后，字号的变化幅度可以大大缩小。主标题不需要 36px——24px 但加粗加黑就足够突出。辅助信息不需要缩小到 11px——14px 但颜色变淡就自然退后。整个界面的字号跨度更紧凑，看起来更专业。

**可提取规则**：
- UI 中只需要 2 个字重（正常 400-500 + 加粗 600-700）
- 远离 400 以下的字重（细体在小字号下难以阅读）
- 2-3 种文字颜色（主色 dark + 次级 gray + 三级 lighter gray）
- 字号是层级最后的工具，先用字重和颜色
- "如果需要弱化文字但不想缩小，用更淡的颜色"

---

### 4.2 "标签是最后的手段"

**来源**：Refactoring UI 书籍 & CSS Day 2019 Gist

**原始问题**：
设计师习惯给每个数据字段加一个粗体标签——"Price: $99"、"Website: example.com"、"Date: Jan 15"。标签占据了大量空间，且通常与数据本身同等视觉权重。

**改动决策**：
1. 如果数据格式自解释，直接删除标签——用户看到 `$99` 就知道是价格
2. 如果必须用标签，让标签弱于数据——标签用小字、淡色、`uppercase`、小字重
3. 把标签转化为值的一部分——"12 left in stock" 替代 "In stock: 12"
4. 如果用户先找标签（如在设置页），标签可以重于数据
5. "Most of the time you don't need a label at all!"

**为什么有效**：
标签是元数据，不是内容。减少标签让真正有价值的数据浮到表面。用户读 "Price: $99" 时大脑需要两次映射（"这是价格标签 → 价格是99"），但读 "$99" 只需要一次。

**可提取规则**：
- 大多数时候不需要标签——数据格式本身说明一切
- 需要标签时，使其成为辅助而非主角（small, uppercase, muted color）
- "Turn the label into the value"（把标签融入值的表述）
- 例外：用户扫描页面找标签时（如设置页），标签应该突出

---

### 4.3 文字对齐与基线

**来源**：Refactoring UI 书籍 & Twitter tips

**原始问题**：
不同字号的文字在水平排列时（如导航中 logo 旁的小链接），开发者习惯垂直居中对齐（center）。但视觉上，不同字号的文字居中对齐后会感觉"不对齐"。

**改动决策**：
按基线（baseline）对齐，而非按几何中心对齐。在 CSS 中这意味着使用 `align-items: baseline` 而非 `align-items: center`。

**为什么有效**：
人眼对文字基线的偏移极其敏感。两个不同字号但基线对齐的文字，视觉上感觉"在一条线上"。但如果按中心对齐，大文字的顶部和小文字的顶部可能对齐，但基线错位——人眼会觉得"歪了"。

**可提取规则**：
- 不同字号文字并排时，基线对齐 > 中心对齐
- 导航栏中 logo + 链接是经典应用场景
- CSS: `align-items: baseline`（Flexbox）

---

### 4.4 "两档字重足够"的更深层逻辑

**来源**：Refactoring UI 书籍 & guidefari.com 笔记

**原始问题**：
开发者倾向于使用多种字重（Light 300, Regular 400, Medium 500, Semibold 600, Bold 700）来区分层级，但过多的字重差异在视觉上难以分辨。

**改动决策**：
只用两档字重：
- 正常文字：400 或 500（取决于字体）
- 强调文字：600 或 700

如果需要第三层区分，用颜色而非字重——例如 Bold (700) 用于标题，Normal (400) 用于正文，Normal (400) + lighter color 用于辅助信息。

**为什么有效**：
两个字重之间的对比是明确、可感知的。三个字重变得模糊——400 vs 500 在很多字体中差异极小。字重种类越少，层级越清晰。

**可提取规则**：
- 两档字重足够（Normal + Bold）
- 不要把 300（Light）用在正文字号——太细难以阅读
- 需要"中等强调"时用颜色而非 Medium 字重
- "Use a lighter color or smaller font size instead of a lighter weight"

## 5. 阴影改造案例

### 5.1 "从大而黑到小而彩"：偏移阴影

**来源**：7 Practical Tips（"Offset your shadows"）& Josh Comeau 深度教程

**原始问题**：
开发者使用 `box-shadow: 0 0 20px rgba(0,0,0,0.3)` ——大模糊半径、大扩散、纯黑。阴影看起来像一个模糊的灰色光环，不像是真实的光影。

**改动决策**：
1. 添加垂直偏移（y-offset）——模拟从上方照下来的光
2. 减小模糊半径——让阴影更紧致
3. 降低不透明度——让阴影更微妙
4. 使用多层阴影（3-5 层）模拟真实光影的渐变

经典"作弊"阴影：
```css
/* Before — 模糊灰色光环 */
box-shadow: 0 0 20px rgba(0,0,0,0.3);

/* After — 自然阴影 */
box-shadow:
  0 1px 1px rgba(0,0,0,0.075),
  0 2px 2px rgba(0,0,0,0.075),
  0 4px 4px rgba(0,0,0,0.075),
  0 8px 8px rgba(0,0,0,0.075),
  0 16px 16px rgba(0,0,0,0.075);
```

**为什么有效**：
自然界的光源通常在上方，阴影应该向下投射。垂直偏移模拟了这个物理规律。多层阴影从紧密到扩散模拟了真实阴影的 umbra（本影）+ penumbra（半影）结构。人眼看到后会下意识认为"这看起来像真的"。

**可提取规则**：
- y-offset 通常为 x-offset 的 2 倍（光从上方偏左）
- 多层 > 单层：每层 offset 和 blur 递增，opacity 递减
- 不要用纯黑做阴影颜色——用与背景同色相的颜色（降低饱和度和明度）

---

### 5.2 阴影颜色：不用纯黑的阴影

**来源**：Josh Comeau 深度教程 & Steve Schoger 推特

**原始问题**：
阴影使用半透明黑色 `rgba(0,0,0,0.x)`，放在任何颜色的背景上都会让该区域变得"脏灰"（desaturated, muddy）。

**改动决策**：
阴影颜色匹配背景的色相，降低明度和饱和度：
- 浅蓝背景 → 阴影用更深的蓝色 `hsl(220, 60%, 50%)`
- 浅暖背景 → 阴影用更深的暖色
- 永远匹配背景 hue，然后降低 L 和 S

**为什么有效**：
半透明黑色叠加在任何颜色上都会降低饱和度。用同色相但更深更灰的颜色做阴影，叠加后颜色仍然"干净"，不会产生脏灰感。

**可提取规则**：
- 阴影的颜色应匹配背景色相（hue），降低明度（lightness）和饱和度（saturation）
- 不要用纯黑（`hsl(0,0%,0%)`）的阴影
- "Goldilocks" 颜色：不太灰也不过于鲜艳

---

### 5.3 阴影层级系统：从"随便加"到"明确 elevation"

**来源**：Josh Comeau & CSS Day 2019 & Frame Library 设计系统

**原始问题**：
不同元素的阴影各用各的，互不相关。两个"卡片"的阴影可能完全不同。没有统一的"光源"概念，不同元素仿佛被不同方向/强度的光照射。

**改动决策**：
建立统一的 elevation 系统（以 Material Design 为参考）：

| Level | 用途 | 阴影特征 |
|-------|------|---------|
| 0 | 页面背景 | 无阴影 |
| 1 | 卡片（网格中） | 最轻微，1-2 层 |
| 2 | 卡片（悬停态）/ 导航栏 | 轻微，2-3 层 |
| 3 | 下拉菜单 / Popover | 中等，3-4 层 |
| 4 | 模态框 / Dialog | 明显，4-5 层 |
| 5 | 通知 Toast | 最明显，多层 |

**关键原则**：
- 所有阴影共享相同的光源方向（x:y 比例一致）
- Y-offset 约为 X-offset 的 2 倍（光从上方偏左）
- 随着 elevation 增加：offset ↑, blur ↑, opacity ↓
- 添加负 spread 值来模拟"物体遮挡光源"的效果

**为什么有效**：
统一的 elevation 系统让用户下意识地理解界面层级：越高的东西越重要。模态框的阴影比卡片深，用户本能地知道模态框"在前面"。

**可提取规则**：
- 所有元素共享同一光源
- 3-5 个 elevation 级别足够覆盖所有需求
- Hover 态升一级 elevation，配合 `translateY(-2px)`
- 阴影用 CSS 变量存储：`--shadow-sm`, `--shadow-md`, `--shadow-lg`, `--shadow-xl`

## 6. "少即是多"在 UI 战术中的体现

### 6.1 删掉边框后界面为什么更清晰？

**来源**：7 Practical Tips & Refactoring UI 书籍（"Use fewer borders"）

**原始问题**：
开发者习惯用边框（border）来分隔一切——卡片有边框、表格有边框、输入框有边框、分组有边框。大量 1px 线条在界面上形成视觉噪音。

**改动决策**——三种替代方案：

| 替代方案 | 适用场景 | 效果 |
|---------|---------|------|
| Box shadow | 需要"抬起"的元素（卡片、模态框） | 比边框更柔和、更现代 |
| 不同背景色 | 相邻区域（侧边栏 vs 主内容） | 自然分区，无边框的视觉噪音 |
| 额外间距 | 同背景上的元素组 | 最纯粹的分离——只靠空间 |

**具体案例**：
- 表格：去掉所有竖线和大部分横线，用斑马纹 + 间距区分行
- 卡片组：去掉卡片边框，用 subtle 阴影 + 间距区分
- 页面分区：用浅灰和白色交替背景区分，零边框

**为什么有效**：
边框是"硬"的分隔——它在被分隔的两个元素之间插入了一个新元素（1px 线条）。间距是"软"的分隔——它不引入新元素，只是拉开距离。人的视觉系统处理"软分隔"更轻松，因为它不需要额外解析一个"线条"元素。

**可提取规则**：
> "The next time you find yourself reaching for a border, try a box shadow, two different background colors, or extra spacing instead."

- 边框是最后的选择，不是第一反应
- Box shadow 最适合"需要抬起"的元素
- 背景色区分最适合"大面积分区"
- 间距最适合"同背景上的小组"

---

### 6.2 减少颜色数量后层级为什么反而更清楚？

**来源**：7 Practical Tips & Refactoring UI 书籍

**原始问题**：
开发者使用多种颜色来区分不同元素——蓝色标题、绿色按钮、红色删除、橙色警告、紫色链接。每种颜色都想"表达自己"，结果没有一种颜色真正突出。

**改动决策**：
将颜色降为三类：
- 灰度（8-10 档）：用于所有文字、背景、边框、分隔
- 一个主色（5-10 档）：用于主按钮、链接、active 状态
- 2-3 个强调色（各 5-10 档）：红（破坏性操作）、黄（警告）、绿（成功）

**为什么有效**：
颜色是稀缺的注意力资源。当每个元素都有自己的颜色时，颜色失去了"信号"价值——用户不知道看哪里。当你把 95% 的界面做成灰度，主色的一抹蓝就会像灯塔一样引导用户的注意力。

**可提取规则**：
- 灰度承担 90% 的 UI 工作
- 主色只用在最重要的交互元素上
- 色彩越少，剩余的越有力量

---

### 6.3 "少用分割线"的具体替代方案

**来源**：Steve Schoger Twitter："Less borders" & Little UI Details

**原始问题**：
设计师频繁使用 `<hr>` 或分割线来分隔内容区块，导致页面上到处都是水平线。

**替代方案矩阵**：

| 方案 | 做法 | 效果 |
|------|-----|------|
| 间距分组 | 组间间距 = 2× 组内间距 | 最纯粹 |
| 背景色交替 | 奇偶行不同浅色背景 | 适合列表/表格 |
| 彩色 accent border | 卡片顶部 3-4px 彩色线 | 有装饰性且不"脏" |
| 对比面板标题 | 标题区域用 subtle 对比色背景 | 替代标题下的分割线 |
| Keyline 连接 | 用竖线连接而非分隔 | 让分散的内容感觉更连接 |

**Steve 的经典替代**：
> "A technique I've been using lately on panels to distinguish the titles instead of a keyline is using subtle contrast." — 给面板标题区一个微妙的对比背景色，替代传统的水平分割线。

**可提取规则**：
- 分割线（`<hr>`）是最粗暴的分隔手段
- 间距、背景色、accent border 都能更优雅地完成同样的任务
- keyline 不仅可以"分隔"还可以"连接"

## 7. 失败或过度改造的案例

### 7.1 Refactoring UI 方法不适合什么场景？

**基于方法论内在边界的分析（综合多个来源）：**

#### 7.1.1 不适合：需要高可访问性的政府/医疗产品

**来源**：Medium 评论区批评 & W3C 可访问性标准

Hector Carranza 在 Medium 上指出："Lighter not smaller... but a contrast check returns AA Fail." Refactoring UI 建议用更淡的颜色代替更小的字号来表达层级，但如果对比度低于 4.5:1（AA 标准），对视力障碍用户不友好。

**教训**：Refactoring UI 的默认建议倾向于"视觉优雅"，需要与可访问性标准（WCAG AA/AAA）进行权衡。60% 灰色是 45 岁以上用户可轻松阅读的最浅灰色——比这个更淡就会出现问题。

#### 7.1.2 不适合：内容为主的文档型网站

**来源**：ajnisbet.com 笔记 & 方法论分析

Refactoring UI 针对的是应用型 UI（app-style UI），很多建议（如 "标签可能不需要"、"h1 不需要很大"）在博客、新闻、文档等内容型网站上不适用。内容型网站需要更传统的文档层级——大的 h1、清晰的 h2/h3 层级、内容中的链接需要下划线/颜色。

#### 7.1.3 不适合：已有成熟设计系统的大型产品

**来源**：book review 多处暗示

如果你在 Facebook、Google、Stripe 这样的公司工作，设计系统已经由专业设计师建立，Refactoring UI 的大部分建议已经被内化到设计系统中。这本书最适合的是"没有设计师的开发者团队"或"小团队从零开始的项目"。

#### 7.1.4 不适合：过度追求系统化导致设计僵化

**来源**：Refactoring UI 书籍本身警告 & updivision.com 书评

"Don't force it. Not every decision you make needs to be based on a system." — 作者自己也警告不要过度系统化。如果你花了 20 分钟纠结按钮的 padding 应该是 12px 还是 16px（两者都在你的 spacing scale 中），你就已经偏离了 Refactoring UI 的初衷——系统是为了加速决策，不是制造新的纠结。

#### 7.1.5 "先灰度设计"的黑暗面

**来源**：yusuf.fyi 经验分享

一位开发者在应用"先灰度设计"原则后发现自己卡住了——"you may find it hard at first to emphasize things without using colors, but that's normal and that means you're improving your design skills." 灰度设计的目的是发现层级问题，但如果灰度版本已经"够好"，开发者可能永远不引入颜色，导致最终产品"能用但缺乏个性"。

**建议**：灰度设计是"中间态"而非"终态"。完成灰度版本后必须进入色彩阶段。

#### 7.1.6 方法与 CSS 框架的冲突

**来源**：ajnisbet.com 实践笔记

"Many of the principles from Refactoring UI are at odds with the popular frameworks." 许多 CSS 框架（Bootstrap 等）带有自己的默认间距、默认字号层级、默认边框样式。应用 Refactoring UI 的方法意味着大量覆盖框架默认值。这也是为什么作者后来开发了 Tailwind CSS——一个"不提供默认设计"的 utility-first 框架，更契合 Refactoring UI 的系统化哲学。

## 总结：可提取的决策模式

Refactoring UI 方法论的核心思维模式可以提炼为以下决策链：

```
遇到设计问题
  ↓
1. 可以先不做吗？（删掉边框？删掉标签？不做背景色？）
  ↓ 如果必须做
2. 从预设系统中取值（spacing scale? color shade? type scale?）
  ↓ 如果系统不够
3. 新增一个值，但确保与相邻值差距 ≥25%
  ↓
4. 在完整页面上下文中审视（不要孤立判断单个组件）
```

**核心战术工具箱**（按频率排序）：
1. 减少而非增加（边框、颜色、字重种类、标签）
2. 系统化而非即兴（间距、颜色、字号都从预定义 scale 中取值）
3. 字重和颜色做层级的主力，字号做辅助
4. 从"太多留白"开始，删减而非添加
5. 多层阴影模拟自然光，匹配背景色相
6. 灰度设计在先，颜色在后
