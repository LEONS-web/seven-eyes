## 一、间距系统（Spacing System）

### 1.1 间距刻度的具体数值

Refactoring UI 的核心主张：**在开始构建之前，系统化一切**。间距是最容易"露馅"的地方——开发者设计的 UI 看起来不对劲，最常见的原因就是间距不一致。

**基准单位**：4px 是最推荐的基准（而非 8px），因为：
- 4px 足够细粒度处理紧凑 UI（图标内边距、密集表格）
- 4px 可以完美整除 8、16、24、32——也就是设计师自然会用到的值
- 8px 作为基准太粗糙，2px 作为基准会产生太多几乎相同的选项

**标准间距刻度**（基于 4px 基准，参考 Tailwind / GitHub Primer / Atlassian）：

| Token | 值 | REM（16px 基准） | 典型用途 |
|-------|-----|-------------------|----------|
| `--space-1` | 4px | 0.25rem | 图标与文字间距、紧凑徽章内边距 |
| `--space-2` | 8px | 0.5rem | 按钮内部 padding、表单字段间距 |
| `--space-3` | 12px | 0.75rem | 紧凑卡片内边距 |
| `--space-4` | 16px | 1rem | 默认组件内边距、段落间距 |
| `--space-5` | 20px | 1.25rem | 稍宽松的间距 |
| `--space-6` | 24px | 1.5rem | 卡片 padding、区域内元素间距 |
| `--space-8` | 32px | 2rem | 内容组之间的间距 |
| `--space-10` | 40px | 2.5rem | 大区域分隔 |
| `--space-12` | 48px | 3rem | 页面级区域分隔 |
| `--space-16` | 64px | 4rem | 主要区域断点 |
| `--space-20` | 80px | 5rem | Hero 区域间距 |
| `--space-24` | 96px | 6rem | 营销页面级分隔 |

**关键规则**：
- **相邻刻度至少相差 ~25%**——16px 和 24px（差 50%）没问题，但 14px 和 16px 太接近，产生不了有意义的视觉差异
- 超过 32px 后，间距变大的速度加快（32 → 48 → 64 → 80 → 96），因为人眼不再能区分 36px vs 40px 这种微差
- **永远不要用刻度之外的值**——如果 16px 太小、24px 太大，要么扩展刻度，要么忍痛二选一；不要发明 20px（除非 20px 已经在你的刻度里）

### 1.2 "Start with too much white space, then remove" 的操作

这是 Refactoring UI 最反直觉的规则之一：

> **留白应该是被移除的，而不是被添加的。**
> 大多数开发者从太少的空间开始，然后试图"加点空间"——这最多只能给元素凑到刚好够的间距。反过来，从慷慨的留白开始，然后逐步缩减，最终得到的布局会干净得多。

**操作步骤**：
1. 给每个元素比你觉得需要的更多 50%-100% 的空间
2. 从最大的间距开始缩减（section 之间 → 组件之间 → 组件内部）
3. 停在"刚好不觉得空旷"的那个点——这个点通常已经比你最初设想的要宽松得多
4. 记住：**更多留白几乎总是让设计看起来更好**。如果界面看起来拥挤，首先加大 section 之间的间距，然后检查组件内部间距是否一致

### 1.3 Padding vs Margin：什么时候用什么？

**核心原则**：在现代 CSS（Flexbox + Grid）中，**让父容器管理间距，子元素保持 margin-free**。

| 属性 | 作用域 | 使用规则 |
|------|--------|----------|
| **padding** | 元素内部 | 用于组件内部呼吸空间。同一类型组件 = 相同 padding |
| **margin** | 元素外部 | 仅在父容器无法设置 gap 时使用（如长文本段落）。margin 会合并（collapse），容易与兄弟元素冲突，破坏组件可复用性 |
| **gap** | 弹性/网格布局中的兄弟元素 | **首选方案**。由父容器统一管理子元素间距，子元素不需要任何 margin |

**实际应用**：
- 所有"大"按钮应该有相同的 padding（如 `py-3 px-6`）
- 卡片网格用 `gap-6`，而不是给每个卡片加 `mb-6`
- 只在段落文本（`<p>` 标签）等无法用 gap 的场景用 margin
- 绝对不要"用眼睛微调像素"——每个 spacing 值必须来自刻度

### 1.4 通过间距表达元素关系

**内部 ≤ 外部规则**（来自 Gestalt 心理学）：

> 元素内部的 spacing 应该**等于或小于**元素外部的 spacing。

- 一张有 16px padding 的卡片，它和相邻卡片之间的间距应该 ≥16px
- 如果内部间距 > 外部间距，元素会显得"拥挤"——分组信号被混淆
- 如果外部间距远大于内部间距，元素会显得"稀疏"——但这也比拥挤好

**具体层级**：
- **组件内部**：紧凑间距（4-12px）——同一组内的元素紧密相关，应该靠近
- **组件之间**：中等间距（16-32px）——网格中的卡片、表单中的字段组
- **区域之间**：大间距（48-96px）——Hero 到 Features、Features 到 Pricing

**垂直间距的 2:1 规则**：标题上方间距应该是下方间距的约 2 倍。如果标题下面用 8px，上面用 16-24px。这让标题"锚定"到下面的内容而非上面的。

## 二、字体层级（Typography Hierarchy）

### 2.1 字重优先于字号——具体怎么做？

Refactoring UI 特别强调：**不要只用字号来创造层级**。只用字号导致的结果是巨大标题 + 微小正文的极端对比，看起来非常业余。

三个层级的工具：

| 工具 | 作用 | 使用规则 |
|------|------|----------|
| **字号（Size）** | 最大的层级跳跃 | 页面主标题 vs 正文 |
| **字重（Weight）** | 中等层级对比 | 标题用 bold（600-700）、正文用 regular（400）、辅助文字用 normal + 低对比度颜色 |
| **颜色（Color）** | 最精细的层级区分 | 主文字（深色）、次要文字（中等灰）、三级文字（浅灰） |

**实战规则**：
- **只用 2-3 个字重**：Regular（400）+ Semibold（600）覆盖大部分场景。需要更强强调时加 Bold（700）
- 避免 font-weight < 400（太细的文字在屏幕上不可读）。想弱化文字用更浅的颜色，而非更细的字重
- **大型文字 → 可以用更轻的字重**：64px 标题用 400 weight 看起来很优雅。12px 小字用 400 是底线，500 更佳
- **相邻层级不共享字重**：如果 H2 是 bold（700），H3 必须是 semibold（600）或 medium（500）。同字重相邻会"塌缩"层级——不管字号差多少
- **"Save all three for one"**：最大字号 + 最粗字重 + 最深颜色，只留给页面上那一个最重要的元素

### 2.2 推荐的字体配对（Font Pairing）原则

**第一原则：一款字体足够。**

大多数成功的 SaaS 产品只用一个字体家族。Inter Bold 做标题 + Inter Regular 做正文——没有配对，就是最干净的方案。

**第二原则：如果需要配对，最多两款。**

| 配对类型 | 例子 | 适合场景 |
|----------|------|----------|
| **Serif 标题 + Sans-serif 正文** | Playfair Display + Inter | 编辑类、金融科技、高端品牌 |
| **Geometric sans 标题 + Humanist sans 正文** | Space Grotesk + Inter | 科技产品、现代 SaaS |
| **单一家族 + 字重变化** | Inter（Bold + Regular） | 大多数产品 UI |

**配对铁律**：
- x-height 要匹配——两款字体在相同 px 值下，字母物理高度差异过大，标题和正文看起来就不协调
- 忽略少于 5 个字重的字体——不够用的
- 两个 display 字体会互相打架。一个 display（标题演出）+ 一个 workhorse（正文承载）是最干净的方案

### 2.3 字号比例建议

Refactoring UI 没有强行指定某个数学比例，但推荐 modular type scale：

**三种常见比例**：

| 比例 | 倍率 | 字号序列（基准 16px） | 适用场景 |
|------|------|----------------------|----------|
| **Major Second** | 1.125 | 12, 14, 16, 18, 20, 23, 26 | 密集 UI、Dashboard、管理后台 |
| **Minor Third（推荐默认）** | 1.2 | 12, 14, 16, 19, 23, 28, 34 | 大多数产品 UI |
| **Perfect Fourth** | 1.333 | 12, 14, 16, 21, 28, 37, 49 | 需要强烈层次的内容站点、Landing Page |

**推荐的实用字号集**（来自 Refactoring UI skill 实现）：`12, 14, 16, 20, 24, 30, 36px`

**字号与内容宽度**：
- 正文最佳阅读宽度：45-75 字符/行，或 20-35em
- 更宽的文本需要更大的行高

### 2.4 行高和字间距的实用规则

**行高（line-height）**：

| 元素类型 | 行高建议 | 16px 字号的绝对行高 |
|----------|----------|-------------------|
| **正文** | 1.5 – 1.75 | 24–28px |
| **标题** | 1.0 – 1.25 | 紧凑，因为标题不会换多行 |
| **Hero 大标题** | 1.0 – 1.1 | 几乎无行间距 |

- 1.5 是正文的几乎通用默认值
- 文字越宽，行高应该越大
- 标题的上下行（ascenders/descenders）绝对不能碰到一起

**字间距（letter-spacing）**：
- 标题字体通常天生间距较紧，不需要额外调整
- 小字（如 uppercase label）可以加一点 tracking：`letter-spacing: 0.05em`
- 全大写的 label 用 `text-xs font-medium uppercase tracking-wide` 是一个 common pattern

## 三、阴影系统（Shadow & Elevation System）

### 3.1 层级体系——几个层级、各自参数？

Refactoring UI 建议 **5 个 elevation 层级**就够用了：

| 层级 | 用途 | 阴影特征 | 典型组件 |
|------|------|----------|----------|
| **Level 0** | 无阴影 | 无 | 页面背景、正文区域 |
| **Level 1** | 微微抬起 | 小偏移、小模糊，接近背景 | 按钮（默认）、输入框（inset） |
| **Level 2** | 明显分离 | 中等偏移、中等模糊 | 卡片、面板 |
| **Level 3** | 浮起 | 较大偏移、较大模糊 | 下拉菜单、Tooltip |
| **Level 4** | 最高 elevation | 大偏移、大模糊 + 负 spread | 模态框、对话框 |

**具体的阴影数值参考**（基于 Josh Comeau 的 Shadow Palette Generator + Refactoring UI 原则）：

```
/* Level 1 — 按钮、轻微抬起的元素 */
--shadow-sm: 0 1px 2px 0 rgba(0, 0, 0, 0.05);

/* Level 2 — 卡片 */
--shadow-md: 0 1px 3px 0 rgba(0, 0, 0, 0.1),
             0 1px 2px -1px rgba(0, 0, 0, 0.1);

/* Level 3 — 下拉菜单 */
--shadow-lg: 0 4px 6px -1px rgba(0, 0, 0, 0.1),
             0 2px 4px -2px rgba(0, 0, 0, 0.1);

/* Level 4 — 模态框 */
--shadow-xl: 0 10px 15px -3px rgba(0, 0, 0, 0.1),
             0 4px 6px -4px rgba(0, 0, 0, 0.1);

/* Level 5 — 最高 elevation（更大的模态/抽屉） */
--shadow-2xl: 0 25px 50px -12px rgba(0, 0, 0, 0.25);
```

**为什么不直接用 opacity 渐变**：同一个阴影颜色应使用相同的 alpha 值，便于后续"着色阴影"时保持一致。

### 3.2 阴影的两个组成部分

Refactoring UI 的核心洞察：

> 每个阴影其实由两部分组成——**环境光阴影** + **直射光阴影**。

- **环境光阴影**（Ambient）：更紧、更暗、模糊半径小。代表元素下方几乎没有光线到达的区域
- **直射光阴影**（Direct）：更大、更柔、垂直偏移更多、模糊半径更大。代表来自上方光源的光被遮挡投射出的阴影

**实际实现**——用两层 box-shadow：

```css
.card {
  box-shadow:
    /* 环境光——紧、暗 */
    0 1px 2px 0 rgba(0, 0, 0, 0.15),
    /* 直射光——大、柔 */
    0 4px 8px 0 rgba(0, 0, 0, 0.10);
}
```

**随着 elevation 升高**：环境光阴影逐渐消失（越高的元素，底面越暴露在环境光中），直射光阴影变大变柔。在最高 elevation（模态框），环境光阴影几乎完全消失，只剩大而柔的扩散阴影——这就是为什么 `--shadow-2xl` 用 `-12px` 的负 spread 值。

### 3.3 "Use shadows to create depth, not to make things pop"

> 更大的阴影（更高的 blur radius）让元素感觉更靠近用户。越靠近用户，越吸引注意力。

**关键规则**：
- **不要给一切东西加阴影**——如果一切都浮起来，就没有任何东西有深度
- 阴影是 elevation 的信号，不是装饰
- 静态内容（页面上的普通卡片）考虑是否真的需要阴影——通常留白就足够区分
- 阴影最适合：模态框（最高 elevation）、下拉菜单（浮在内容之上）、按钮交互态（hover 时微微抬起）

### 3.4 阴影 vs 边框

| 用阴影 | 用边框 |
|--------|--------|
| 有深度的卡片、容器 | 列表项之间的分割线 |
| 需要 elevation 的元素 | 表格单元格边界 |
| 浮层（下拉、模态） | 表单输入框轮廓（可访问性） |
| 不同背景上的元素（阴影用透明度自适应） | 密集 UI 中的极细分割线 |
| hover/focus 的"抬起"效果 | |

**阴影作为边框替代**：`box-shadow: 0 0 0 1px rgba(0,0,0,0.06)`——1px 的 ring，效果等同于边框但更微妙，不占盒模型空间。

### 3.5 彩色阴影（Colored Shadows）的具体做法

这是 Josh Comeau 广为人知的技术，与 Refactoring UI 的理念完全一致：

**问题**：`rgba(0,0,0,0.4)` 的黑色半透明阴影，叠在任何颜色上都会让颜色 **变灰变脏**——降低饱和度。

**方案**：匹配阴影的 hue（色相）到背景色，降低饱和度和亮度。

```
/* ❌ 灰色阴影——在彩色背景上看起来脏 */
box-shadow: 0 4px 8px rgba(0, 0, 0, 0.25);

/* ✅ 彩色阴影——匹配背景色相，降饱和、降低明度 */
box-shadow: 0 4px 8px hsl(220deg 60% 50% / 0.25);
```

**操作公式**（来自 Josh Comeau）：
- 保持与背景相同的 hue
- 降低 saturation 到 ~60%（不是 100%，否则太鲜艳）
- 降低 lightness 到 ~50%（比纯黑柔和，但保持深度）
- Alpha 通常 0.2-0.4

**在 Tailwind CSS 中使用**：
```
阴影颜色工具类支持：shadow-blue-500/25、shadow-red-500/25
```

**实用场景**：
- 蓝色品牌按钮 → `box-shadow: 0 4px 12px rgba(37, 99, 235, 0.35)`（蓝色调的阴影，不是灰的）
- Hero CTA 按钮的 glow 效果
- Feature 卡片的 hover 状态

## 四、边框与分割（Borders & Dividers）

### 4.1 "Use fewer borders"——那用什么替代？

Refactoring UI 最有名的建议之一：**抵制用边框隔离元素的冲动**。太多的边框让设计看起来很忙碌和拥挤。

三种替代方案（按优先顺序）：

**① 用更多留白替代边框**

> 还有什么比增加间距更好的方式来区分元素呢？

如果两组元素之间需要区分，直接加大它们之间的间距——不需要引入任何新 UI。

**② 用不同的背景色**

给相邻区域稍有不同的背景色（如白色卡片 vs `#f9fafb` 的页面背景），这种微妙的对比足以区分元素。

**③ 用 box-shadow 替代 border**

1px 的阴影 ring（`box-shadow: 0 0 0 1px rgba(0,0,0,0.06)`）效果等同于边框，但更微妙。在元素与背景不同色时效果最好。

### 4.2 分割线 vs 留白 vs 背景色差——各自适用场景

| 方法 | 适用场景 | 不适用场景 |
|------|----------|-----------|
| **留白**（加大间距） | 内容区域之间的区分、卡片网格 | 密集 UI（如数据表）、需要明确界限的场景 |
| **背景色差** | 侧边栏 vs 主内容区、表头 vs 表体、卡片 vs 页面背景 | 白色背景上的白色卡片（无对比度可调） |
| **1px 低透明度分割线** | 列表项之间、表格行间、菜单项间 | 间距已经很大的区域 |
| **box-shadow ring** | 元素与背景不同色时、需要深度感的卡片 | 同色背景上的元素 |

**实操流程**（来自 Refactoring UI skill 的去杂乱步骤）：
1. 从一切开始（边框、阴影、背景色都加上）
2. **移除边框** → 改用间距
3. **移除背景色** → 改用留白分组
4. **移除分割线** → 加大 section 间距
5. **移除阴影** → 只保留给 elevation（模态、下拉）
6. **只加回必要的**——用于层级或清晰度

### 4.3 用 accent border 添加色彩

如果设计感觉平淡，**在一侧添加彩色 accent border** 是最简单的"设计感"技巧：

```html
<!-- 左侧 accent border（适合 alert、重点卡片） -->
<div class="border-l-4 border-l-blue-500 bg-blue-50 p-4">...</div>

<!-- 顶部 accent border（适合卡片） -->
<div class="border-t-4 border-t-indigo-500 rounded-lg p-6">...</div>

<!-- 底部 accent border（适合 active tab） -->
<a class="border-b-2 border-b-blue-500">Active Tab</a>
```

## 五、色彩系统（Color System）

### 5.1 主色/辅色/中性色的定义方法

Refactoring UI 把色彩分为三个类别：

| 类别 | 数量 | 用途 |
|------|------|------|
| **灰度（Greys）** | 8-10 个色阶 | 文字、背景、面板、表单控件——UI 中绝大多数元素都是灰色的 |
| **主色（Primary）** | 5-10 个色阶（1-2 个颜色） | 主要操作按钮、active 导航、链接。这就是让人想到"Facebook 是蓝色的"那个颜色 |
| **强调色（Accent）** | 每个语义色 5-10 个色阶 | 成功（绿）、警告（黄）、错误/危险（红）、信息（蓝）、以及用于分类或突出新功能的额外颜色。复杂 UI 可能需要多达 10 个不同的 accent 颜色 |

### 5.2 灰度色板的构建（不是纯 #000 到 #FFF）

**纯灰色看起来没有生命力**。Refactoring UI 的关键洞察：

> 给灰色添加微妙的饱和度——用蓝色色调做冷色 UI（科技产品），用黄色/棕色色调做暖色 UI。

**标准灰度色板**（冷色调 = 带蓝色）：

```
White / near-white:  #FFFFFF, #F9FAFB
Gray 100:  #F3F4F6  — 微妙背景
Gray 200:  #E5E7EB  — 边框、分割线
Gray 300:  #D1D5DB  — 禁用态
Gray 400:  #9CA3AF  — 占位符文字
Gray 500:  #6B7280  — 次要文字
Gray 600:  #4B5563  — 正文文字
Gray 700:  #374151  — 强调文字
Gray 800:  #1F2937  — 标题
Gray 900:  #111827  — 接近黑色的文字
```

**关键规则**：
- **永远不要用纯 `#000000`**——它太过刺眼，不自然。最深的灰色（`#111827`）即可
- 暖色 UI 的灰度色板应该带黄色/棕色饱和度（如 Basecamp 的暖调背景）
- 冷色 UI 的灰度应该带蓝色饱和度（`#64748b` 类型的 slate gray 最适合蓝主色产品）
- 在 HSL 中操作灰度：保持低饱和度（5%-15%），在极端浅/深色时稍微提高饱和度以保持一致的色温

**Radix Colors 的灰度选择指南**：
- `Gray`（中性）——任何品牌色都适用
- `Slate`（蓝调）——配合 Indigo、Blue、Sky、Cyan
- `Mauve`（紫调）——配合 Tomato、Red、Crimson、Pink、Purple
- `Sand`（暖黄调）——配合 Yellow、Amber、Orange

### 5.3 主色色板的构建

**方法**（直接来自 Refactoring UI 原文）：
1. **选择基准色**（base color）——选一个适合做按钮背景的色阶
2. **选择最浅和最深的色阶**——根据实际使用场景（最浅用于 alert 背景，最深用于文字）
3. **填充中间色阶**——先填第 3 和第 7 级（约 300 和 700），让它们成为相邻色阶之间"完美的折中"
4. **实际操作中**：浅色 → 提高亮度、降低饱和度、色相向 60°（黄色方向）微调；深色 → 降低亮度、提高饱和度、色相向 0°/240°（红/蓝方向）微调
5. **绝对不要用 `lighten()`/`darken()` 函数动态生成**——你会得到 35 个看起来一样的蓝色。预定义 5-10 个固定色阶

### 5.4 "Destructive actions don't need to be red"——颜色语义的替代方案

这是 Refactoring UI 最常被误解的规则之一。它的意思不是"删除永远不用红色"，而是：

> **语义 ≠ 视觉权重**。不是所有 destructive 动作都需要一个巨大的、红色的、加粗的按钮。

**按钮层级中的 destructive 动作处理**：

| 情境 | 视觉处理 | 原因 |
|------|----------|------|
| **主要 destructive 动作** | 红色实心按钮 + 确认对话框 | 这是页面上的关键动作，需要警告用户 |
| **次要 destructive 动作** | 红色文字按钮（ghost/link 风格） | 如设置页的"删除账户"——不应当和"保存修改"争夺注意力 |
| **三级 destructive 动作** | 最小化处理，直到确认时再升级 | 如列表页的单个"删除"按钮 |

**关键原则**：
- "Delete"按钮不应当比 "Save" 按钮更突出——如果两者都是实心填充，视觉竞争会让用户困惑
- 将 `variant`（视觉权重）和 `intent`（语义含义）分离：`variant=ghost` + `intent=danger` = 红色文字链接（适合不重要的 destructive 动作）
- 确认对话框中才是 destructive 按钮应该"变大变红"的地方——此时用户已经确认意图

### 5.5 色彩对比度的实用检查规则

**文字-背景对比度**（WCAG）：
- 正文文字（<18px）：≥ 4.5:1
- 大文字（≥18px 或 ≥14px bold）：≥ 3:1
- UI 组件（按钮边框等）：≥ 3:1

**Refactoring UI 的实用检查**：
- `#374151`（gray-700）在白色背景上 = ~10.5:1，是可读正文的下限推荐
- 用 `text-gray-900`（深色）、`text-gray-600`（中等）、`text-gray-400`（浅色）三层文字色
- **在白色背景上不要用比 gray-500（`#6B7280`）更浅的灰色**——那会低于 4.5:1

## 六、圆角系统（Border Radius System）

### 6.1 圆角半径的层级体系

> 更大的 border-radius = 更 playful、更友好；无圆角 = 更严肃、更正式。

**标准圆角刻度**：

| Token | 值 | 适用元素 |
|-------|-----|----------|
| `--radius-none` | 0px | 表格、数据密集区域 |
| `--radius-sm` | 2-4px | 小徽章、标签、输入框 |
| `--radius-md` | 6-8px | 按钮、小卡片 |
| `--radius-lg` | 12-16px | 大卡片、面板、模态框 |
| `--radius-xl` | 20-24px | 超大卡片、特色区域 |
| `--radius-full` | 9999px | 药丸形按钮、头像 |

**关键规则**：
- 小元素用小圆角，大元素可以用大圆角
- 同一个组件类型在全局使用相同的圆角——所有卡片要么都 `rounded-lg` 要么都 `rounded-xl`，不能有的 8px 有的 16px
- 没有两个相邻刻度应该太接近——0、4、8、16、24、9999 提供了足够的视觉差异

### 6.2 嵌套元素圆角的关系——"内圆角 = 外圆角 - 间距"

这是最精确数学化的设计规则之一，也是视觉品质的试金石：

> **Inner Radius = Outer Radius − Gap**

其中 Gap = 内边距（padding）+ 边框宽度（border-width）。

```css
/* 示例：卡片有 16px 圆角 + 12px padding 内含图片 */
.card {
  border-radius: 16px;   /* outer radius */
  padding: 12px;          /* gap */
}

.card-image {
  border-radius: 4px;    /* 16 - 12 = 4 ✓ */
  /* 如果写成 border-radius: 16px 就会很奇怪——两个圆弧不同心 */
}
```

**用 CSS calc 动态计算**：

```css
:root {
  --outer-radius: 16px;
  --padding: 12px;
}

.card-inner {
  border-radius: calc(var(--outer-radius) - var(--padding));
}
```

**例外情况**：
- 当 gap 大于 outer radius 时，内圆角应该为 0（不能是负数）
- 当 gap > 24px 时，内外元素可以视为独立表面，各自的圆角可以独立选择
- 内部元素如果与容器在视觉上独立（如大 banner 内的小卡片），使用自己的组件圆角 token 而非公式计算

**禁忌**：
- 绝对不要在圆角容器内放直角元素（`border-radius: 0`）——这会打断空间逻辑
- 绝对不要让子元素比父元素更圆（视觉反演）

## 七、组件设计模式

### 7.1 按钮层级（Button Hierarchy）

Refactoring UI 的按钮层级——**一个页面/视图只有一个 primary 按钮**：

| 层级 | 背景 | 边框 | 文字色 | 用途 |
|------|------|------|--------|------|
| **Primary** | 品牌色（实心） | 无 | 白色/浅色 | 最主要的 CTA："保存"、"提交"、"创建" |
| **Secondary** | 灰色实心（或透明） | 品牌色（如 outline 风格） | 品牌色或灰色 | 替代动作："取消"、"返回" |
| **Tertiary** | 透明 | 无 | 品牌色或灰色 | 低优先级动作："了解更多" |
| **Destructive** | 红色 | 无 | 白色 | 删除、移除（不应当比 primary 更突出） |
| **Ghost** | 透明 | 无 | 灰色 | 最不显眼的动作 |
| **Disabled** | Gray-200 | 无 | Gray-400 | 无法执行 |

**关键规则**：
- **一个页面一个 primary**——两个 primary = 没有 primary
- 如果错误情况下有 "Retry"，它应该继承语义色而不是品牌色
- Gray secondary（灰色实心次要按钮）要用 gray-400~500 级别，**不要用 gray-200**——那看起来像 disabled
- Destructive 按钮如果和 primary 同级显示，应当降级为文字链接（text-red），不应和 primary 争夺视觉焦点
- "Cancel" 可以用灰色实心，但 outline 风格给了更清晰的区分

**按钮位置**：Primary 放在按钮组的最右侧（或最后）——因为用户从左到右扫描，最后看到的是最重要的。

### 7.2 表单设计的最佳实践

**布局原则**：
- **单列布局优先**——简单表单永远单列
- 双列**仅**用于语义相关的字段（First/Last name, City/State/ZIP）
- **标签永远在输入框上方**（不要侧边标签，手机端必然崩塌）
- 用间距分组相关字段——组内紧凑（8-12px），组间宽松（24-32px）

**输入框宽度与预期内容匹配**：
- Email/URL：全宽或 ~400px
- 电话：~200px
- 邮编：~100px
- 城市：~200px
- 州下拉：~150px

**输入框状态**：
```
Default:  border-gray-300
Focus:    border-blue-500 + ring-2 ring-blue-200
Error:    border-red-500 + ring-2 ring-red-200
Disabled: bg-gray-100 text-gray-400
```

**Placeholder 永远不是 Label**：
- Placeholder 在 focus 时消失，所以不能作为唯一的标签
- Placeholder 适合做提示/示例："john@example.com"

**按钮标签**：用**具体动词 + 对象**——"Save changes"、"Create project" 而不是 "Submit"、"OK"

### 7.3 空状态 / 加载状态 / 错误状态

Refactoring UI 特别强调：**不要忽视空状态**。

**空状态（Empty State）**：
- 空状态不应该是 "No data found" 一行字
- 做三件事：① 解释为什么是空的 ② 告诉用户下一步做什么（清晰的 CTA）③ 让用户预览填充后的样子
- 如果依赖用户生成内容，空状态应该是**优先设计**的，不是事后补充
- 隐藏空状态中没有意义的 UI（如空的表格还有筛选栏、标签栏）

**加载状态（Loading State）**：
- < 500ms：不需要 loading 指示器（太快了反而闪烁）
- 500ms – 2s：spinner
- \> 2s：**skeleton screen**（低精度内容轮廓 + 脉冲动画）
- \> 5s：显示进度条或"这比预期时间更长" + 取消/重试按钮
- Skeleton 的布局必须匹配真实内容的布局——否则内容加载后会产生 layout shift

**错误状态（Error State）**：
- 错误是局部范围的——一个卡片加载失败不应该让整个页面变空白
- 回答三个问题：① 出什么问题了（说人话）② 用户能做什么（重试/联系支持）③ 回到正常状态的路径是什么
- 不要在错误信息中展示技术细节（stack trace）

### 7.4 数据可视化的小规则

来自 Tufte 的 Data-Ink Ratio 原则（与 Refactoring UI 的"去装饰"理念一脉相承）：

- **移除一切不承载数据的视觉元素**：背景色、网格线（除非必要）、3D 效果、渐变、阴影
- 直接标注数据点而不是用图例（眼睛不需要来回移动）
- **Sparkline**（微型趋势图）在卡片中展示趋势比完整图表更有效——去掉所有轴和标签
- 最多 4 种不同色相的图表色——超过这个数量人眼无法区分
- 饼图不如条形图（人眼不擅长比较角度）
- 网格线如果有必要保留 → 用 `#d8d4ce` 或等价的最浅灰色

## 八、"强调通过弱化"（Emphasize by De-emphasizing）

### 8.1 核心理念

这是 Refactoring UI 最反直觉、也最有威力的设计战术：

> **不要放大你想强调的元素——缩小其他一切。**

当重要元素不够突出时，第一反应是让它更大/更粗/更鲜艳。但很多时候这个元素已经达到了合理的尺寸上限。这时真正的杠杆在另一边：**弱化竞争元素。**

### 8.2 具体操作方法

**弱化技巧（按激进程度排序）**：

| 技巧 | 操作 | 效果 |
|------|------|------|
| **降低字号** | 辅助文字从 16px → 14px 或 12px | primary 内容自然变大 |
| **降低字重** | 从 bold → medium → regular | 减少视觉表面积竞争 |
| **降低对比度** | 从 gray-900 → gray-600 → gray-400 | 次要内容"退后" |
| **缩小/移除图标** | 减小图标尺寸或用更低对比度的颜色 | 减少视觉重量 |
| **去掉背景色** | 移除不必要的卡片背景、色块 | 让内容在留白中呼吸 |
| **去掉阴影** | 静态元素不加阴影 | 只有真正需要 elevation 的元素保留阴影 |
| **合并标签和值** | "In stock: 12" → "12 left in stock" | 消除冗余元素 |
| **移除整个元素** | 删除不贡献信息的装饰性 UI | 最激进的弱化——但往往效果最好 |

**Label 处理的关键规则**（来自 Refactoring UI 原文 "Labels are a last resort"）：

> 在展示数据时，尽量**不用 label**。数据的格式和上下文往往足以传达含义。

- "In stock: 12" → "12 left in stock"（合并标签和值）
- "Email: john@example.com" → 直接显示 "john@example.com"（格式即含义）
- 如果必须保留 label → **把 label 做成辅助内容**：更小字号、更低对比度、更轻字重、甚至可以 uppercase + tracking
- **例外**：当用户会主动寻找 label 时（如产品规格表），强化 label 而非弱化它

**De-emphasize 检查清单**：
1. 页面上哪个元素应当最突出？
2. 哪些竞争元素可以被弱化？（降低对比度、减小字号、移除装饰）
3. 有没有可以合并的 label-value pair？
4. 有没有可以完全删除的装饰性元素？
5. 在灰度下测试——没有颜色后，层级是否依然清晰？

### 8.3 适用场景

- **导航栏**：当前 active item 保持深色，其他 item 大幅降低对比度（gray-500 级别）
- **Card 列表**：只保留核心数据的高对比度，描述文字、标签、时间戳全部弱化
- **Dashboard**：关键指标的数值应该"跳出来"，label 应该"退回去"
- **定价页面**：推荐方案用阴影/边框突出，其他方案整体降低对比度和字号
- **产品图片网格**：图片才是主角——去掉卡片阴影、缩短描述文字、缩小按钮

## 总结：核心战术规则速查表

| 系统 | 核心规则 | 关键数值 |
|------|----------|----------|
| **间距** | 4px 基准，只从刻度选值，内部 ≤ 外部 | 4, 8, 12, 16, 20, 24, 32, 48, 64, 80, 96px |
| **字体** | 字重 > 字号，2-3 个字重，1-2 个字体家族 | 字号 12/14/16/20/24/30/36；行高 1.5（正文）/ 1.1（标题） |
| **阴影** | 5 层 elevation，每层包含环境光+直射光两个部分 | sm→md→lg→xl→2xl；彩色阴影匹配背景 hue |
| **边框** | 用更多留白和不同背景色替代边框 | 1px low-alpha border、box-shadow ring、accent border |
| **色彩** | 8-10 灰 + 5-10 主色 + 每语义色 5-10 色阶；灰度加饱和 | 预定义所有色阶，灰度用 `#111827` 而非 `#000` |
| **圆角** | 内圆角 = 外圆角 - 间距 | 0, 4, 8, 16, 24, 9999px |
| **按钮** | 一页一个 primary；destructive 不抢 primary 焦点 | 三级（filled / outline+ghost / text-only） |
| **状态** | 空状态是设计机会，skeleton > spinner | 加载态 <500ms 不显示，>2s 用 skeleton |
| **强调** | 弱化竞争元素，不是放大目标元素 | 降低对比度 → 减小字号 → 移除装饰 → 合并/删除元素 |
