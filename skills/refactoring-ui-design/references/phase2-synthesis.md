# Refactoring UI 设计方法论 · Phase 2 框架提炼

> 提炼日期：2026-07-03
> 来源：6 份 Phase 1 调研文件（~190KB, ~2,800 行原始材料）
> 提炼原则：方法论而非人格模仿。Refactoring UI 是战术系统而非设计哲学，关注"怎么做"而非"为什么"。
> 公共规则（反 AI slop / UX 底线 / 资产映射 / 平台翻译 / 视觉验收）不放入本 skill，统一沉淀到 `standards/`。

---

## 一、核心定位

Refactoring UI 在设计流水线中的角色：**第六层 · 落地战术执行层**。它不是设计师，是一套可操作的视觉设计规则系统。

仲裁顺序中它的位置：

```
Dieter Rams（减法闸门 + 诚实性审查 + 耐久性判断）
  ↓
Refactoring UI（本 skill）—— 间距系统 + 字体层级 + 阴影深度 + 色彩系统 + 边框替代 + 组件模式
  ↓
（流水线终点 — 输出可直接交给代码执行层）
```

它接收 Dieter Rams 的输出（"减法后"的界面——哪些元素被删了、哪些保留了、功能理由是什么），在此基础上追加判断：**在删干净的基础上，间距应该多大？字重应该多粗？阴影应该多深？颜色应该选哪个？**

它不管的事：留白质量（原研哉的）、品牌记忆点（Paula Scher 的）、图标隐喻（Susan Kare 的）、动效质感（Yugo Nakamura 的）、功能减法（Dieter Rams 的）。

### 专属判断域

1. **间距系统**：定义间距刻度（4px 基准）、管理 padding/margin/gap 的使用规则、确保内部间距 ≤ 外部间距
2. **字体层级**：字重优先于字号、2-3 字重覆盖全场景、字号 scale 定义、行高与字宽规则
3. **阴影与深度**：5 层 elevation 体系、环境光+直射光双阴影、彩色阴影、阴影 vs 边框决策
4. **色彩系统**：灰度色板构建（8-10 阶+色温）、主色色阶（5-10 阶）、HSL 操作法则、"标签是最后手段"
5. **边框替代**：用间距/背景色/box-shadow 替代边框、accent border 注入色彩
6. **组件模式**：按钮层级（primary/secondary/tertiary）、表单布局、空/加载/错误状态
7. **"强调通过弱化"**：降低次要元素而非放大主要元素
8. **圆角系统**：嵌套公式（内圆角 = 外圆角 - 间距）、层级刻度

### 他接收什么输入，追加什么判断

| 来自 Dieter Rams 的输出 | Refactoring UI 如何使用 |
|-------------------------|------------------------|
| "减法后"的界面——删了哪些元素 | 不在落地时重新加回来。删掉的元素在间距和布局中不预留位置 |
| 保留元素的"功能理由" | 功能关键元素 → 最高的视觉权重和落地精度；理解关键元素 → 中等；情绪关键元素 → 可灵活调整 |
| 耐久性要求 | 结构性决策固化为 design token（间距 scale、字号 scale、色板） |
| 细节意志审查发现的任意性 | 修正：统一间距、统一圆角、统一过渡时间 |

## 二、心智模型（5 个）

### 模型 1：Design = System —— 设计即系统

**一句话**：好设计不是关于艺术天赋，而是关于系统和约束。预定义一切——间距、字号、字重、颜色、阴影、圆角——让每次设计决策从有限选项中挑选，而非从无限可能中即兴发挥。

**来源证据**（≥5 个独立来源）：
- Adam Wathan 核心洞见："Good design is not about artistic talent. It is about systems and constraints."
- 书的起手式："Limit your choices. Define systems in advance so you're picking from a curated set of options, not trying to choose from an infinite palette."
- Tailwind CSS 的设计哲学直接体现了这一点——utility class 的有限集合强制一致性
- Steve Schoger 的实践：每次设计都在 Figma 中使用预定义样式库，不从空白画布开始
- CSS Day 2019 演讲结尾："Make your ideas look awesome, without relying on a designer."

**应用方式**（UI 判断用）：
- 系统检查：间距是否来自预定义 scale？颜色是否来自预定义色板？字体大小是否来自预定义 type scale？
- 如果发现"随手选的数值"→ 标记为任意性（A6），要求替换为系统值
- 设计系统越完善，设计决策越快，输出越一致

**局限性**：
- 过度系统化可能压抑创意（批评者所说的"SaaS 同质化"）
- 品牌差异化场景需要突破系统——Scher 的领域

---

### 模型 2：Tactics, Not Talent —— 战术代替天赋

**一句话**：设计不需要"天赋"。90% 的好设计来自于知道一系列可学习、可复用的"作弊技巧"。传统设计教育太关注色彩理论、排版学等高阶原则，但这些从未帮人做出即时改善。

**来源证据**（≥5 个独立来源）：
- Adam："Tricks that didn't require any artistic talent, but made things look better instantly for reasons that made sense to me as a developer."
- "7 Practical Tips for Cheating at Design"——文章标题本身就是宣言
- Ladybug 播客：Steve 自称"我是那种拿一个用户提交的糟糕设计，一路重构到看起来像一个设计师做的的人"
- Full Stack Radio #74：五种开发者自学设计的方法全部是战术性的，不涉及理论
- Ben Nadel 评论："90% 的设计就是选好 margin 并确保元素对齐"

**应用方式**（UI 判断用）：
- 遇到设计问题时，先找对应的战术规则而非求助于"设计感觉"
- 每个战术规则都有明确的"触发条件 → 操作步骤"结构
- 战术之间可以独立应用——不用理解整个设计体系就能改善单个问题

**局限性**：
- 战术不能替代深层理解——复杂设计问题需要第一性原理思考
- "完全依赖战术" = 永远只会修修补补，不会系统性创新

---

### 模型 3：Emphasize by De-emphasizing —— 强调通过弱化

**一句话**：不要放大你想强调的元素——缩小其他一切。这是 Refactoring UI 最反直觉也最有效率的战术：降低次要元素的视觉重量，比放大主要元素更有效。

**来源证据**（≥4 个独立来源）：
- 原文："Instead of trying to further emphasize an element, give inactive elements a softer color so they sit more into the background."
- 导航栏案例：当前 active item 保持深色，其他 item 降到 gray-500 级别
- 卡片列表案例：只保留核心数据的对比度，标签/时间戳全部 weak
- Dashboard 案例：关键数字"跳出来"，label "退回去"

**应用方式**（UI 判断用）：
- 弱化技巧（按激进程度）：降低字号 → 降低字重 → 降低对比度 → 缩小/移除图标 → 去掉背景色 → 去掉阴影 → 合并标签和值 → 移除整个元素
- 检查：页面上哪个元素应该最突出？所有比它更"视觉上更重"的元素都需要弱化
- 灰度测试：去掉颜色后，层级是否依然清晰？

**局限性**：
- 弱化不能无限进行——某个元素已经被弱化到不可读/不可操作时，说明需要用其他策略
- 需要强调的元素也可能确实需要放大——弱化不是唯一工具

---

### 模型 4：Hierarchy is Everything —— 层级决定一切

**一句话**：视觉层级是界面最重要的属性。不是所有元素都平等——界面的工作就是让重要的事情看起来重要，让次要的事情退后。字重优先于字号，颜色优先于装饰。

**来源证据**（≥5 个独立来源）：
- 书籍第二章标题："Hierarchy is Everything"
- "Size isn't everything"——只用字号控制层级是业余的标志
- "2-3 种颜色 + 2 种字重即可覆盖大部分 UI 工作"
- Steve 的 15 条战术体系中有 5 条直接服务于层级
- 按钮设计：层级优先于语义——破坏性操作不一定是红色

**应用方式**（UI 判断用）：
- 层级三工具：字号（最大的跳跃）→ 字重（中等的对比）→ 颜色（精细的区分）
- "Save all three for one"——最大字号+最粗字重+最深颜色只留给页面上那一个最重要的元素
- 每个页面只能有一个 primary button，只能有一个"视觉第一落点"

**局限性**：
- 在信息密度极高的界面（金融终端、航空管制），层级规则需要调整
- 不同文化对"什么应该突出"的直觉不同

---

### 模型 5：Constraints are Freedom —— 约束即释放

**一句话**：预定义的约束不是限制创造力，而是保护你免受自己的糟糕决策之害。从有限的、精心挑选的选项中做选择，比每次打开色板随意挑选更快、更一致、更好看。

**来源证据**（≥4 个独立来源）：
- Adam Wathan Rails World 2023："If you're just trying to get stuff done, it actually is helpful to have more constraints to save you from shooting yourself in the foot."
- 书的核心理念：在开始构建之前定义好所有系统——颜色 shade、字号、间距、阴影、圆角
- Tailwind CSS 的 utility classes 本身就是一个约束系统——你在 50 个 spacing 值中挑选，而非无限个
- Steve 的实践：在 Figma 中定义好样式库，所有设计从此派生

**应用方式**（UI 判断用）：
- 系统覆盖检查：这个项目的间距值有多少个？> 12 个 → 系统可能过碎
- 相邻刻度检查：有没有两个间距值"看起来差不多"？→ 合并
- 系统≠不能扩展——但扩展必须有充分理由

**局限性**：
- 约束太少 = 每次决策都是任意性的。约束太多 = 系统臃肿不可维护
- 新项目应先从最小可用系统开始，逐步扩展

## 三、决策启发式（8 条）

### H1 · 间距系统检查

**触发**：评审任何界面的间距使用。

```
1. 基准检查：
   → 间距是否基于 4px（或 8px）基准？有没有"随手选的数值"？
   → 超过 3 个不同间距值在 4px 以内（如 12/14/16）→ 合并。相邻刻度至少差 25%。

2. 关系检查：
   → 组内间距是否小于组间间距？"相关元素靠近，不相关远离"。
   → 标题上方间距是否是下方间距的约 2 倍？→ 标题锚定到下方内容。

3. 容器检查：
   → 内部间距（padding）是否 ≤ 外部间距（margin/gap）？
   → 所有同类型组件（如"大按钮"）共享相同的 padding。

4. 首选方案排序：
   → gap（父容器管理子间距）> padding（元素内部呼吸）> margin（最后手段，仅用于无法用 gap 的场景）

5. 从过多留白开始：
   → 如果界面感觉拥挤 → 先加大 section 间距，再处理组件内间距。
   → "留白应该是被移除的，而不是被添加的。"
```

**为什么**：间距不一致是"开发者设计的 UI"最显眼的破绽。系统化的间距是设计专业感的基石。

---

### H2 · 字体层级检查

**触发**：评审界面中的文字层级。

```
1. 字重优先：
   → 是否只用了字号来控制层级？→ 同步使用字重和颜色。
   → 界面中用了几个不同字重？> 3 个 → 减少到 2-3 个（Regular + Semibold + Bold）。

2. "Save all three for one"：
   → 最大字号 + 最粗字重 + 最深颜色 → 只留给最重要的元素。其他元素至少"牺牲"其中一项。

3. 字号 scale：
   → 字号是否来自预定义 scale？建议：12, 14, 16, 20, 24, 30, 36px。
   → 相邻字号间隔是否 ≥ 1.2 倍？(Major Second 1.125 用于密集 UI，Minor Third 1.2 用于大多数产品)

4. 行高规则：
   → 正文：1.5-1.75（16px → 24-28px）
   → 标题：1.0-1.25
   → 正文行宽：45-75 字符/行（20-35em）

5. 标签处理：
   → "Labels are a last resort"——能用格式传达含义就不用标签。
   → 如果必须用标签 → 标签更小、更浅、更低对比度。数据是大字深色。
```

**为什么**：字体层级是用户理解"先看哪里"的主要线索。只用字号不用字重和颜色 = 层级扁平。

---

### H3 · 阴影与深度检查

**触发**：评审界面中的阴影和深度表达。

```
1. 光源一致性：
   → 所有阴影的垂直偏移方向是否一致？（模拟来自上方的光源）
   → 垂直偏移应该始终为正（向下）——如果出现向上的阴影，打破物理一致性。

2. 双阴影规则：
   → 每个有阴影的元素是否同时有"环境光阴影"（紧致、暗、小模糊）和"直射光阴影"（大、柔、垂直偏移）？
   → 高 elevation 时环境光阴影消失 → 只剩大而柔的扩散阴影。

3. 层级限制：
   → 界面中是否使用了超过 5 个不同的阴影值？→ 压缩到 5 层（sm → md → lg → xl → 2xl）。
   → 静态内容（普通卡片）→ 考虑是否需要阴影（留白通常足够）。

4. 彩色阴影：
   → 阴影颜色是否用纯黑+opacity？在彩色背景上会"变灰变脏"。
   → 替代：匹配背景色 hue、降低 saturation 到 60%、降低 lightness 到 50%。

5. 阴影 vs 边框决策：
   → 需要 elevation → 阴影。需要精确分离 → 边框。
   → 可以用 1px box-shadow ring 替代 1px border（更微妙，不占盒模型）。
```

**为什么**：阴影是 UI "摸起来像真的"的关键。单阴影（只有模糊没有结构）+ 灰色阴影（不匹配背景色温）= 业余感的最大来源之一。

---

### H4 · 色彩系统检查

**触发**：评审界面的色彩使用。

```
1. 灰度色板：
   → 是否有 8-10 个灰阶？还是只有 #000、#333、#666、#999？
   → 灰度是否注入了色温（冷灰=slate蓝调，暖灰=黄/棕调），还是纯灰？
   → 最深文字色是否避免了纯 #000？

2. 主色色阶：
   → 主色是否有 5-10 个色阶（从最浅到最深）？
   → 色阶构建规则：基准色 → 最深色 → 最浅色 → 填补中间。
   → 浅端：提升亮度 + 降低饱和度 + 色相微调向 60°（暖）；深端：降低亮度 + 提高饱和度。

3. HSL 操作：
   → 是否用 HSL 而非 HEX/RGB？HSL 更符合人类直觉。
   → "不要让明度扼杀饱和度"——明度远离 50% 时要手动提高饱和度。

4. 彩色背景上的文字：
   → 是否用了灰色文字放在彩色背景上？→ 替换为同色相、调饱和度/明度的颜色。
   → 本质是降低对比度，而非"变灰"。

5. 依赖颜色的信息：
   → 是否有只靠颜色传达的信息？→ 添加图标/对比度辅助（色盲友好）。

6. 无障碍：
   → 正文文字 ≥ 4.5:1 对比度。大文字 ≥ 3:1。
   → 白色背景上不要用比 gray-500（#6B7280）更浅的灰色作为正文。
```

**为什么**：色彩系统不完善 = 每个页面都要重新选色 = 不一致。Refactoring UI 的方法论核心就是"提前定义系统，使用时只做选择"。

---

### H5 · 边框替代检查

**触发**：评审界面中边框的使用密度。

```
1. 边框过多检测：
   → 页面中是否每一行/每一个区域都有边框？→ 逐一检查边框是否必要。
   → "Too many borders can make a design look really busy."

2. 替代方案（按优先顺序）：
   a. 增加间距——两个元素之间加大 gap 比加边框更好
   b. 不同背景色——相邻区域用略微不同的背景色区分
   c. box-shadow——比 border 更微妙，效果等同但不分散注意力
   d. accent border——在卡片/区块的单侧（顶部/左边）加 4-6px 彩色条

3. 边框保留场景：
   → 表单输入框轮廓（可访问性要求）
   → 表格中极细的分割线
   → 高对比度需求（如数据密集的仪表盘）

4. 去边框流程：
   → 删除所有边框 → 增加间距 → 添加微妙的背景色差异 → 只在必要时恢复最少量的边框
```

**为什么**：边框是最"重"的视觉分隔手段。每一条边框都在向用户喊"这里有分界"。过多的边框 = 视觉噪音 = 不专业。

---

### H6 · 按钮层级检查

**触发**：评审界面中的按钮设计。

```
1. 一页一个 primary：
   → 页面中是否有多个"看起来差不多重要"的按钮？→ 只保留一个 primary（实心高对比度品牌色）。
   → 两个 primary = 没有 primary。

2. 层级体系：
   → Primary（实心品牌色）→ Secondary（轮廓或低对比度）→ Tertiary（文字链接风格）
   → 是否有三级以上的按钮样式？→ 合并。三级足够覆盖。

3. 破坏性操作：
   → 破坏性按钮是否始终是红色实心的？→ "不是所有的 destructive 都需要红色"。
   → 如果 destructive 不是页面主操作 → 降级为 ghost/link 风格的文字红色按钮。
   → 确认对话框中才是红色实心按钮该出现的地方。

4. 位置：
   → Primary 放在按钮组的最右侧（或最后）——用户从左到右扫描，最后看到的是最重要的。

5. 标签：
   → 用具体动词+对象："Save changes" > "Submit"；"Create project" > "OK"。
```

**为什么**：按钮层级混乱 = 用户不知道该点哪个 = 犹豫 = 转化率降低。"一个页面一个 primary"是最简单也最常被违反的规则。

---

### H7 · 空/加载/错误状态检查

**触发**：评审界面的非理想状态处理。

```
1. 空状态（Empty State）：
   → 是 "No data found" 一行字，还是有解释 + CTA？
   → 空状态应该是优先设计的，不是事后补充。
   → 三个要素：① 解释为什么是空的 ② 告诉用户下一步做什么 ③ 让用户预览填充后的样子。
   → 隐藏空状态中没有意义的 UI（如空表格还有筛选栏）。

2. 加载状态（Loading）：
   → < 500ms：不需要指示器。
   → 500ms-2s：spinner。
   → > 2s：skeleton screen（内容轮廓 + 脉冲动画）。
   → > 5s：进度条 + "这比预期时间更长" + 取消/重试按钮。
   → Skeleton 布局必须匹配真实内容的布局。

3. 错误状态（Error）：
   → 错误是局部范围的——一个卡片加载失败 ≠ 整个页面变白。
   → 三个回答：① 出了什么问题（说人话）② 用户能做什么（重试/联系支持）③ 回到正常状态的路径。
   → 不要在错误信息中展示 stack trace。
```

**为什么**：非理想状态是"设计品质感"的真正试金石。大多数业余设计只覆盖了 happy path——空状态、加载态、错误态才会暴露设计是否完整。

### H8 · "强调通过弱化"审计

**触发**：感觉界面中多个元素在"竞争注意力"。

```
1. 识别竞争：
   → 页面上哪个元素应该最突出？（视觉第一落点）
   → 列出所有视觉重量 ≥ 它的元素。

2. 逐项弱化（按激进程度）：
   a. 降低对比度（深色 → gray-600/500）
   b. 降低字重（Bold → Semibold → Regular）
   c. 缩小字号
   d. 缩小/移除图标或用更低对比度颜色
   e. 去掉背景色（不必要的卡片底色）
   f. 去掉阴影
   g. 合并标签和值："In stock: 12" → "12 left in stock"
   h. 移除整个元素

3. 灰度测试：
   → 去掉所有颜色。层级是否仍然清晰？
   → 如果不清晰 → 说明当前层级依赖颜色而非结构。需加强字重和间距的组织作用。

4. "最后手段"检查：
   → 标签（Label）是最后的手段——在展示数据时，尽量不用标签。
   → 数据本身的格式和上下文通常足以传达含义。
```

**为什么**：这是 Refactoring UI 最反直觉的战术，也是最被低估的。大多数人的本能是"把重要的东西放大"——但放大有上限。弱化次要元素几乎没有上限，而且效果往往更好。

## 四、反模式（8 条）

| # | 禁止 | 修正 |
|---|------|------|
| A1 | **随意间距**——"差不多就行"的值、不用预定义 scale、同一页面上 14px 和 16px 混用 | 建立 4px 基准间距 scale，只从中选值。相邻刻度差 ≥ 25%。所有间距值必须来自系统。"If 16px is too small and 24px is too big, pick one — don't invent 20px unless 20px is already in your scale." |
| A2 | **纯灰阴影**——`rgba(0,0,0,0.25)` 叠在任何彩色背景上都会让颜色变灰变脏 | 匹配阴影 hue 到背景色——保持同色相、降饱和到 60%、降明度到 50%。彩色背景上的阴影应该是彩色的。 |
| A3 | **灰色文字放在彩色背景上**——用 `text-gray-500` 在蓝色/紫色/绿色背景上 | 选择与背景色同色相的颜色，调饱和度/明度来降低对比度。本质不是"变灰"，是降低对比度。 |
| A4 | **每个元素都在"喊"**——多个 primary button、所有文字都是深色、所有区域都同样重要 | 一页一个 primary。用字重和颜色区分三级信息：最重要的深+粗 → 次要的灰+正常 → 三级浅灰+小字。"强调通过弱化。" |
| A5 | **边框泛滥**——每行有边框、每个区域有边框、表格布满了网格线 | 删掉所有边框，用间距+背景色差异+box-shadow替代。只在必要时恢复最少量的边框（表单输入框、极细表格分割线）。 |
| A6 | **依赖颜色传达信息**——"红色=删除、绿色=成功"是唯一的区分方式 | 添加图标、文字标签、对比度差异作为辅助信息载体。色盲用户（占男性 8%）无法只依赖红绿色区分。 |
| A7 | **"Label: Value" 照搬数据库结构**——直接把数据库字段映射为 UI 标签 | "Think outside the database." "In stock: 12" → "12 left in stock"。如果格式本身能传达含义就不需要标签。必须用标签时→标签更小更浅。 |
| A8 | **忽略非理想状态**——空状态是"No data"一行字、加载态只有 spinner、错误态显示 stack trace | 空状态=优先设计（解释+CTA+预览）。加载态>2s用skeleton。错误态=说人话+给行动路径+不暴露技术细节。非理想状态才是设计品质的试金石。 |

## 五、输出契约（10 字段）

### 输入（调用方提供）

| 字段 | 必填 | 说明 |
|------|------|------|
| `product_context` | ✅ | 产品是什么、版本、目标用户 |
| `current_page_or_component` | ✅ | 哪个页面/组件/流程 |
| `dieter_rams_output` | ✅ | 第五层 Dieter Rams 的 10 字段输出——本层在"减法后"界面上做落地战术 |
| `target_platforms` | ✅ | HarmonyOS/iOS/Android/Web/多平台 |
| `existing_design_system` | 可选 | 已有的 design token（如果有，本层基于其审核而非从零定义） |
| `theme_prominence_target` | 可选 | 弱/中/强/节日级——影响色彩和圆角的激进程度 |
| `brand_color` | 可选 | 品牌主色（如有），用于定义主色色阶 |

### 输出（10 字段）

1. **落地战术一句话** — "这个界面在落地时，核心的视觉策略是什么？" 例："基于 4px 基准间距 + Minor Third 字号比例 + Slate 冷灰 + 5 层阴影的 SaaS 产品风格"。

2. **间距系统参数**：
   - 推荐间距 scale（基于 4px 或 8px 基准）
   - 关键区域的间距建议（section 间/组件间/组件内）
   - 间距关系规则：内部 ≤ 外部、标题上方 2x 下方、组内 < 组间

3. **字体层级参数**：
   - 推荐字号 scale（12/14/16/20/24/30/36 或等价）
   - 字重策略（2-3 个字重，具体用途分配）
   - 行高规则（正文 1.5-1.75，标题 1.0-1.25）
   - 字体选择建议（1-2 个字体家族，如有品牌字体则适配）

4. **阴影系统参数**：
   - 5 层 elevation 的具体数值建议（sm → 2xl）
   - 是否启用彩色阴影（如品牌色 glow 效果）
   - 阴影 vs 边框的场景决策

5. **色彩系统参数**：
   - 灰度色板建议（色温方向：冷灰/暖灰/中性灰）
   - 主色色阶建议（基准色 → 深浅端点 → 中间填充）
   - 破坏性/成功/警告语义色建议
   - 无障碍对比度阈值检查结果

6. **边框与分割策略**：
   - 用间距替代边框的具体点位
   - accent border 的位置和颜色
   - 必须保留边框的场景（表单输入框等）

7. **组件级建议**：
   - 按钮层级（primary/secondary/tertiary/destructive 的视觉参数）
   - 表单字段布局 + 输入框宽度匹配规则
   - 空状态/加载态/错误态的处理策略

8. **圆角系统参数**：
   - 圆角层级刻度（0/4/8/16/24/9999px 或等价）
   - 嵌套规则验证：内圆角 = 外圆角 - (padding + border-width)

9. **"强调通过弱化"应用**：
   - 哪些元素应该弱化？（降低对比度/字重/字号的具体方案）
   - 灰度测试结果——去掉颜色后层级是否清晰？
   - 标签优化建议——哪些 Label: Value 可以合并？

10. **给代码执行层的约束**：
    - 必须固化为 design token 的参数（间距 scale、字号 scale、色板、阴影层）
    - 平台翻译建议（ArkUI 的 vp 单位 / CSS 的 rem / Compose 的 dp / SwiftUI 的 pt）
    - 不可实现的降级方案（如双层阴影在低性能设备上的简化）

| 拨盘 | Refactoring UI 的影响 |
|------|----------------------|

## 七、跨场景迁移

### 不同主题风格

| 风格 | 主题显性度 | Refactoring UI 的约束重点 |
|------|-----------|--------------------------|
| 极简风 | 弱-中 | 间距 scale 偏大（更宽松）。灰度色板中性。阴影层 2-3 层足够。圆角偏小（2-4px） |
| 商务风 | 中 | 间距精准、灰度冷调（slate）、阴影克制。圆角 4-8px。字体中性无衬线 |
| 节日风 | 强-节日级 | 色彩系统可更丰富（更多强调色）。圆角可更大（16-24px）。阴影可更戏剧化。但系统化原则不变 |
| 纪念风 | 中 | 灰度暖调（sand/warm gray）。间距宽松。圆角柔和（8-16px） |
| 摄影风 | 弱 | 间距最大——照片需要呼吸空间。灰度深色为主。阴影极少。圆角 0-4px |

### 不同产品类型

| 产品 | hero asset | Refactoring UI 的约束重点 |
|------|-----------|--------------------------|
| 内容产品 | 内容本身 | 字体层级权重最高——阅读体验的核心。正文行宽 45-75 字符。行高 1.5-1.75 |
| 工具产品 | 任务流 | 间距系统权重最高——密集 UI（仪表盘/管理后台）用小 scale。按钮层级清晰。数据可视化去装饰 |
| 商务产品 | 关键数据+决策 | 系统稳定性权重最高——design token 固化为长期标准。灰度冷调传达专业感 |
| 电商产品 | 商品图+信任信息 | 色彩系统权重最高——品牌色饱和度可提高。按钮层级明确。价格信息最高对比度 |
| 陪伴/娱乐产品 | 角色+情感联结 | 圆角和阴影可更柔和——传达温暖。色彩系统可用更丰富的强调色 |

### 不同平台

| 平台 | Refactoring UI 翻译要点 |
|------|------------------------|
| HarmonyOS ArkUI | 间距单位用 vp（1vp ≈ 1px @160dpi）。阴影用 `shadow` + `blur` + `offset`。系统字体 HarmonyOS Sans。按钮用 `stateStyles` |
| Android Compose | 间距用 dp。阴影用 `elevation` + `shadowElevation`。Material You 动态色可能冲突 → 建议用固定色板。字体默认 Roboto/系统字体 |
| iOS SwiftUI | 间距用 pt。阴影用 `shadow(radius:y:)`——SwiftUI 的阴影比 CSS 简单，双层阴影可能需要替代方案。SF Pro 字体自带良好层级 |
| Web CSS | 全部战术规则原生适用。间距用 rem（16px 基准）。阴影支持完整双层。HSL 色板原生支持。`font-feature-settings: "tnum"` 可用 |

### 迁移步骤（每次新项目/新主题/新平台必跑）

1. 接收 Dieter Rams 的"减法后"界面 + 功能理由
2. 确认产品类型和主题风格（决定各系统的激进程度）
3. 检查/定义间距 scale + 字号 scale + 色板 + 阴影层
4. 跑 H1-H8 决策链
5. 填写 10 字段输出契约
6. 输出 design token + 平台翻译建议

## 八、诚实边界

1. **不负责设计哲学和"为什么"**。Refactoring UI 是战术手册，不是设计理论。需要"为什么这个设计好"的深层理解 → Laws of UX、Don Norman、About Face。

2. **不负责品牌差异化**。这些战术让你做出"好看"的设计——但不保证"独特"。品牌差异化是 Paula Scher 的领域。

3. **不负责交互设计和信息架构**。Refactoring UI 管的是视觉层，不管"用户流程应该怎样"或"信息应该怎么组织"。

4. **不负责游戏 UI / 品牌营销页 / 非标准界面**。方法论的默认假设是 Web 产品界面（仪表盘、表单、列表、卡片）。游戏 UI、品牌营销页需要不同的方法和工具。

5. **不负责动效质感**。那是 Yugo Nakamura 的领域。Refactoring UI 只定义过渡时间（200-300ms）和 hover 反馈的基本参数。

6. **不负责留白质量判断**。"留白有没有组织？"是原研哉的领域。Refactoring UI 只管"间距值本身是否来自系统"。

7. **过度应用会助长同质化**。批评者指出经由 Tailwind → shadcn/ui → AI 工具这条链路，Refactoring UI 间接塑造了"所有 SaaS 看起来一样"的问题。使用时需要配合 Scher 的品牌差异化判断。

8. **预设值需要根据产品和平台调整**。提供的间距 scale、字号 scale、色阶等都是推荐起点，不是不可改变的教条。

9. **信息截止 2026-07-03**。Tailwind CSS v4 / Catalyst / ui.sh 等后续发展可能引入新方法论。
